# System_alpha 迭代与防漂移规范

## 1. 单一事实源

- 系统入口只有 `System_alpha/README_系统入口.md`；
- 当前任务只有 `DEV-ROADMAP_当前开发路线.md`；
- 当前架构只有 `DEV-ARCH_系统架构与边界.md`；
- 历史总表、根 Todo、设计实施稿不得同时声明不同的“唯一下一步”；
- 每次阶段变化先更新路线图，再修改其他索引。

## 1.1 简单流程优先

- 本规范管理后台开发，不得把全部治理步骤转嫁给每次创作；
- 默认执行链为“目标 → 最小资料 → 生成 → 硬边界检查 → 用户反馈”；
- 方案、经验、方法论和测试分开存储，但普通任务不逐层读取；
- 运行实例、详细失败码和完整审计只用于高风险、测试、恢复或故障诊断；
- 新增流程节点必须通过 `DEV-ARCH_系统架构与边界.md` 的必要性门禁。

## 2. 每轮开发记录

开始一项非微小开发前，至少明确：

|字段|说明|
|---|---|
|需求|用户真正想改善什么|
|范围|涉及哪些模块和文件|
|非目标|本轮明确不解决什么|
|输入版本|使用哪些正式资料与测试输入|
|未知项|哪些地方必须问用户|
|验收|怎样判断有效|
|回退|失败后恢复到哪里|

开发结束后记录：变更、测试、用户结论、遗留问题和是否晋升。

## 3. 正文修改后的迭代流程

每篇正文经用户修改后必须复盘，但不是把修改稿直接转成更多规则。

### 3.1 先描述差异

只记录可验证差异：删了什么、补了什么、顺序如何变化、哪类句子被否定、人物反应如何调整、读者信息何时获得。

### 3.2 再做问题归因

|类别|典型问题|应修位置|
|---|---|---|
|世界事实|人物/事件资料错误或缺失|设定系统、事件数据库|
|叙事结构|场景顺序、主钩子、信息释放错误|大纲、细纲、叙事契约|
|章节承接|人物站位、动作、情绪余温断裂|前章承接状态|
|人物意图|不知道为什么说或为什么做|人物意图控制或用户确认|
|章节表达|POV、误导、轻重、留白错误|章节表达决策|
|正文执行|句面过度总结、解释、工整或失去人味|正文 Prompt / Skill 规则|

不得把上游问题全部归咎为“文风问题”，也不得用句面规则修补世界事实。

### 3.3 判断适用范围

- `local`：只适用于本场或本章；
- `character`：某人物长期稳定的语言或行为边界；
- `work`：本作品的叙事与审美规则；
- `system_candidate`：可能跨章节、跨作品复用的方法候选。

只有 `system_candidate` 才进入通用规则候选，而且至少需要用户明确确认或一次近迁移/远迁移验证。作者具体原句、完成桥段、笑点落点和标志性表达不得作为通用生成素材。

### 3.4 回归验证

规则修改后至少回答：

1. 原问题是否消失；
2. 相邻章节是否也改善；
3. 是否引入正文僵硬、解释不足、人物失声或 Token 膨胀；
4. 该规则是否应保留、缩小适用范围或撤回。

## 4. 测试到正式的晋升门禁

测试 Skill、模板或规则只有同时满足以下条件，才能晋升：

1. 有明确需求和非目标；
2. 有固定输入版本和可复现测试；
3. 通过结构/边界检查；
4. 通过至少一个真实章节测试；
5. 用户确认结果方向正确；
6. 没有读取历史归档、外部成稿或污染输入；
7. 记录已知失败与回退方法；
8. 更新正式需求、路线图和变更记录。

“文件已存在”“Skill 能运行”或“某一章效果好”都不等于正式可用。

## 5. 防漂移检查表

每次开发前：

- [ ] 是否从系统入口进入；
- [ ] 是否读取当前路线图；
- [ ] 是否确认目标文件状态；
- [ ] 是否排除了历史归档和污染测试；
- [ ] 是否发现同级冲突并向用户提问；
- [ ] 是否明确本轮不会顺手扩展的范围。

每次开发后：

- [ ] 当前路线图是否更新；
- [ ] 文件状态是否与实现一致；
- [ ] 测试产物是否仍留在测试区；
- [ ] 用户修改是否经过归因而非直接规则化；
- [ ] 旧入口是否被标记过期；
- [ ] 下一步是否仍只有一个。

## 6. Codex 停止与提问协议

遇到不清楚的地方，Codex 应输出：

```text
已确认：从哪些文件得到什么结论
不清楚：缺少或冲突的具体信息
影响：不同选择会改变什么
建议：基于当前证据更推荐哪种选择及原因
问题：请用户确认哪一项
```

在等待回答时，可以继续做不依赖该决策的只读审计和可回退整理；不得越过决策点进行目录迁移、正式晋升、世界事实修改或主链路改造。
