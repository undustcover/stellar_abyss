# System_alpha 系统架构与边界

## 1. 核心开发模式

本系统不是由模型一次设计完成，而是由“用户决策 + Codex 执行 + 产物验证 + 规则迭代”共同推进。

```text
用户提出需求或反馈
    ↓
Codex 识别影响范围、冲突和缺口
    ↓
必要时向用户提问并形成明确决策
    ↓
修改测试实现或正式系统文件
    ↓
使用真实章节/样板进行验证
    ↓
用户评价结果，Codex 归因
    ↓
局部修正 / 通用规则候选 / 架构回退
    ↓
通过门禁后晋升为正式能力
```

用户拥有创作意图、审美、世界事实和重大路线的最终决定权。Codex 负责读取、路由、实现、检查、记录和暴露不确定性。Codex 不得用“看起来合理”的自动推断替代用户的关键决策。

## 2. 系统业务分层

|层级|正式目录|职责|禁止越界|
|---|---|---|---|
|开发控制层|`00_总控台/`|权威入口、路线图、治理规则、模板和变更状态|不保存小说世界事实|
|叙事系统|`01_叙事系统/`|全书至正文的叙事组织与正式交付产物|不反向改写世界事实|
|设定系统|`02_设定系统/`|人物、世界观、组织、地点、能力等长期事实|不承担章节临时写法|
|事件数据库|`03_事件数据库/`|事件事实、因果、时间与信息分层|不承担章节演出安排|
|Skill 系统|`04_Skill系统/`|正式需求、输入输出契约、规则和能力晋升记录|不保存一次性测试正文|

## 3. 创作执行分层

当前仍有效的 V1.3 基础分层如下：

1. 世界事实层：人物、事件、设定和因果；
2. 叙事决策层：读者体验、POV、信息显隐和章节表达目标；
3. 结构组织层：大纲、细纲、场景顺序和章节总控；
4. 人物意图层：人物为什么说、为什么做及不确定时的中断；
5. 正文执行层：在已确认边界内一次生成 Draft；
6. 评估迭代层：用户修改、问题归因、规则候选和回归测试。

2026-07-21 已停止 `Draft → Public` 二次改写分支。当前正文主方向是从 Draft 源头解决故事表达和可读性问题，不再依赖下游规则化重写补救。

## 4. 文件状态模型

所有新建正式系统文档至少使用下列状态之一：

|状态|含义|可否作为默认输入|
|---|---|---|
|`placeholder`|只有位置或待设计说明|否|
|`proposal`|待用户或测试确认的方案|否|
|`draft`|已成形但未通过晋升门禁|仅在任务明确指定时|
|`active`|当前生效的规则、需求或控制文档|可以|
|`confirmed`|用户确认的创作事实或章节决策|可以|
|`deprecated`|已被替代，等待归档|否|
|`archived` / `unavailable`|只供追溯|否|

`remake`、`Remake` 等非标准状态只视为历史遗留，不得推断为 `confirmed`。

## 5. 三类信息必须隔离

### 5.1 正式系统信息

已经确认且可被后续任务复用，包括世界事实、正式模板、active 规则与 confirmed 契约。

### 5.2 测试证据

一次测试的输入、生成稿、用户修改、对比和问题记录。它们证明某条规则是否有效，但自身不能自动成为正式规则。

### 5.3 历史信息

失败分支、被替代版本、污染测试和旧方案。默认不加载；只有追因或防止重复失败时才读取。

## 6. Codex 交互约束

Codex 必须提问的情况：

- 用户需求存在两种会改变架构或产物的合理解释；
- 两份同级 active/confirmed 资料冲突；
- 人物为什么说或为什么做无法从已确认资料确定；
- 是否将测试结论晋升为通用规则无法由证据决定；
- 操作会改变世界事实、作者审美方向或主开发路线；
- 文件归属会改变“正式系统”和“测试实现”的边界。

Codex 不需要提问的情况：

- 能从权威入口和已确认资料得到唯一答案；
- 只是补链接、元数据、索引或格式一致性；
- 运行只读检查、测试或生成可丢弃的诊断结果；
- 按用户已确认的方向进行局部、可回退的实现。

## 7. 全系统奥卡姆原则

本原则适用于资料调用、世界设定、大纲、细纲、人物意图、正文生成、质量检查及以后新增的所有模块。

### 7.1 知识可以复杂，流程必须简单

作者经验、写作方法论、案例、候选方案和测试证据可以不断增长，但不能等量转化为生成步骤。复杂知识留在模块后台；每次任务只调用实现当前目标所需的最少信息。

全系统默认流程只有：

```text
理解用户目标
    ↓
按需读取最小资料
    ↓
直接生成目标产物
    ↓
检查硬边界并交给用户判断
    ↓
反馈只回流到对应知识单元
```

大纲、细纲和正文都不得为了“系统完整”强制增加中间文件、确认轮次或分析报告。

### 7.2 新增步骤的必要性门禁

新增任何流程节点前，必须证明：

1. 不增加该节点会稳定产生可识别失败；
2. 该失败无法通过改进已有输入或 Skill 内部判断解决；
3. 新节点节省的返工成本大于交互、Token 和认知成本；
4. 低风险任务可以跳过该节点；
5. 该节点可以单独撤销，不会牵连整个系统。

不能证明时，不新增节点。

### 7.3 默认内部选择，不要求用户配置

模块、方法和控制单元由 Skill 根据任务内部选择。普通生成不要求用户填写运行实例、选择规则编号或逐项确认。只有以下情况才显式生成运行状态或诊断：

- 高风险内容；
- 多轮中断与恢复；
- 可复现测试；
- 发生失败，需要定位原因；
- 用户明确要求查看控制过程。

### 7.4 创造性优先级

系统优先约束事实、视角、信息边界、目标与禁止越界；不预先规定唯一动作、句子、桥段、节奏和表达解法。方法论为 LLM 提供可能性，不把可能性变成机械清单。

如果一项规则只能让模型“更像在执行规则”，却不能稳定改善故事结果，应删除或降级为可选经验。
