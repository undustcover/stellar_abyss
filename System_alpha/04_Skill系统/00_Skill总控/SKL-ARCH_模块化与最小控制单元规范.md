# Skill 模块化与最小控制单元规范

> [!important] 生效状态
> 本规范已于 2026-07-22 经用户确认，作为全系统生效原则。适用于设定、资料调用、大纲、细纲、人物意图、正文生成、质量检查及未来新增模块。

> [!warning] 奥卡姆校准
> V0.1 对知识类型和控制单元的拆分方向正确，但把运行实例、单元诊断和五层结构写得过于接近强制流程，存在工程过量风险。V0.2 明确：这些结构主要用于后台存储与故障诊断；普通创作不逐层建文件、不要求用户配置控制单元，也不默认生成运行实例。

## 1. 要解决的问题

写作系统会随着作者经验、能力和审美提高持续变化。如果把需求、方案、经验、方法论、执行规则、实例和测试结果写进同一个模板或 Skill，会产生：

- 上下文越来越大；
- 新旧结论互相覆盖；
- 模型无法判断哪条内容当前生效；
- 生成失败时无法定位责任单元；
- 修改一处规则时影响整条链路；
- 规则过多时压缩 LLM 的临场判断与创造空间。

因此，系统采用：

```text
模块化存储
    + 最小控制单元
    + Skill 内部按需选择
    + 单元级测试与归因
    + 渐进式读取
```

## 2. 核心结论

最小控制单元不会天然扼杀 LLM 创造性。真正扼杀创造性的不是“规则被拆小”，而是：

- 所有规则每次同时激活；
- 把经验写成绝对禁令；
- 规定具体句子、桥段和唯一解法；
- 用大量负面检查代替创作目标；
- 不区分事实边界与表达自由；
- 失败后只加规则，不撤回无效规则。

正确做法是：存储时拆小，运行时少量选择；只锁定边界、目标和风险，把具体表达、动作组合、语言节奏和现场发现留给 LLM。

## 3. 后台知识结构，不是前台流程

```text
模块 Module
├─ 需求与问题 Requirement
├─ 方案 Plan
├─ 经验 Evidence / Experience
├─ 方法论 Method
├─ 最小控制单元 Control Unit
└─ 测试与评估 Test / Evaluation
```

以上只是后台分类。正常生成不按此顺序逐层读取，而是由 Skill 从模块索引中选择最少必要内容后直接生成目标产物。

### 3.1 模块

模块对应一个相对稳定的问题域，例如：

- 源头 Draft 生成；
- 章节承接；
- 视角与故事性；
- 场景因果；
- 人物意图；
- 信息显隐；
- 对话；
- 幽默；
- 可读性。

模块只负责定义边界、索引和接口，不堆放全部正文规则。

### 3.2 不同知识类型必须分开

|类型|回答的问题|能否直接控制生成|
|---|---|---|
|需求|系统要解决什么问题|否|
|方案|准备如何解决|否，确认后再转控制单元|
|经验|某次创作或测试观察到什么|否|
|方法论|在什么条件下可能有效|默认否，是控制单元候选来源|
|控制单元|本次运行具体约束什么|可以|
|运行记录|仅在测试、恢复或故障诊断时记录启用单元|条件性使用|
|测试|单元是否有效、失败在哪里|否，只提供证据|

不使用一个内容模板承载全部类型。可以共享少量基础元数据，但正文结构必须按类型分别设计。

## 4. 最小控制单元

### 4.1 定义

一个最小控制单元只控制一个可以独立触发、验证、关闭和归因的问题。

合格例子：

- `POV-LIMITED-001`：普通场景只允许一个人物限知视角；
- `POV-PANORAMIC-001`：宏大汇合场景允许全景客观视角；
- `POV-EXTERNAL-EVIDENCE-001`：全景视角必须保留外显人物证据；
- `HANDOFF-001`：新章必须继承前章未完成状态；
- `READABILITY-LOAD-001`：同一段不得同时引入过多新对象和新信息；
- `SCENE-CHANGE-001`：每场至少改变人物状态、关系、信息或问题之一。

不合格例子：

- “写得更自然、更好看”；
- “执行所有作者风格”；
- “检查视角、对白、幽默、节奏和人物”；
- 一份包含几十条禁令的“正文质量单元”。

### 4.2 基础元数据

所有文件只共享最小外壳：

```yaml
id:
type:
module:
status:
version:
updated:
```

控制单元再增加必要字段：

```yaml
trigger:
scope:
priority:
freedom: hard | guided | optional
conflicts_with: []
depends_on: []
```

正文中只保存：目标、触发条件、控制动作、禁止越界、成功证据和失败码。解释、研究过程和案例放在经验或测试文件，不复制进控制单元。

### 4.3 自由度分级

|级别|适用内容|对 LLM 的控制|
|---|---|---|
|`hard`|世界事实、信息边界、视角禁入、用户明确禁区|必须遵守，不允许创造性覆盖|
|`guided`|故事目标、场景变化、节奏倾向、人物压力|规定效果，不规定唯一实现|
|`optional`|经验性技巧、候选方法、灵感方向|可选择、组合或不用|

系统应尽量把“怎么写”放在 `guided` 或 `optional`，只把真正不能改变的内容设为 `hard`。

## 5. 运行时稀疏选择

存储库可以拥有很多单元，但每次生成只选择当前任务真正需要的少量单元。普通任务由 Skill 内部完成选择，不创建独立运行实例文件。

只有高风险章节、跨窗口恢复、正式测试或失败复现需要显式运行记录。此时可使用简化记录：

```yaml
id: RUN-P0-C03-DRAFT-001
type: draft_runtime_record
module: source_draft
active_units:
  - HANDOFF-001
  - POV-LIMITED-001
  - SCENE-CHANGE-001
conditional_units:
  - POV-PANORAMIC-001
disabled_units: []
```

显式运行记录只说明：

- 为什么启用该单元；
- 适用到章节、场景还是片段；
- 哪些单元未启用；
- 单元之间如何裁决冲突；
- 本次允许 LLM 自由决定什么。

Skill 默认只读取模块索引和被选中的控制单元，不全文读取同模块所有方案、经验、方法论和历史测试。没有显式运行记录时，不得因此阻止生成。

## 6. 创造性保护机制

### 6.1 约束效果，不指定成品

控制单元可以要求“通过外显动作让读者判断人物承受压力”，不能指定必须使用某个动作、某句台词或某个成品桥段。

### 6.2 给生成过程保留创作空白

Skill 在内部保留 `creative_space`，不要求普通任务生成独立表格：

- 具体动作与物件选择；
- 对话措辞与断裂方式；
- 段落节奏；
- 笑点或反差的现场发现；
- 场景内不改变上游事实的细节；
- 多种合理实现之间的选择。

### 6.3 稀疏激活

- 普通场景只加载基础边界和当前主要问题；
- 高风险场景才加载额外控制单元；
- 不适用单元必须标记 `not_applicable`，不能为了完整而强行执行；
- 新增单元前先判断能否替换、合并或撤销旧单元。

### 6.4 经验不得自动升级为硬规则

作者一次修改形成的是经验或候选方法。只有经过确认和迁移验证后，才能成为控制单元；即便晋升，也优先成为 `guided`，而不是 `hard`。

### 6.5 允许受控偏离

当 LLM 找到更有效的表达方式，且没有违反 `hard` 边界，可以保留。测试应评估结果是否达到目标，而不是检查是否机械复现方法步骤。

## 7. 单元级失败定位

控制单元的诊断状态只在测试或失败定位时输出，不进入正常产物，更不进入小说正文：

```text
PASS
FAIL:<unit-id>:<failure-code>
NOT_APPLICABLE
CONFLICT:<other-unit-id>
ASK_USER
```

例如：

```text
FAIL:POV-EXTERNAL-EVIDENCE-001:CHARACTER_BECAME_INVENTORY
FAIL:HANDOFF-001:EMOTION_RESIDUE_DROPPED
CONFLICT:POV-LIMITED-001:POV-PANORAMIC-001
```

这样生成失败时可以定位到控制单元、输入、触发条件或冲突，而不是笼统归因为“正文质量不好”。


目录只表示知识类型与生命周期，不要求运行时全部读取。

### 8.1 模块定位地图

每个进入样板建设或正式运行的模块必须有一张后台定位地图，至少登记：

- 模块入口和当前运行状态；
- 需求、方案、方法、控制单元、运行记录与测试证据的位置；
- 默认输入、按需工具和输出之间的接口方向；
- 常见可观察问题的首要责任位置及转向条件。

地图只保存位置、职责和引用，不复制各文件正文，也不增加作者日常步骤。一次问题先记录一个有证据的首要定位点；不能确定时进入诊断，不得同时在多个模块堆规则。

源头 Draft 样板地图见 `../03_正文生成/01_源头Draft/00_模块总控/MAP-SOURCE-DRAFT-001_生产与问题定位地图.md`。

## 9. 迁移策略

1. 不立即拆完所有旧文档；
2. 先以“源头 Draft”作为样板模块；
3. 把当前问题重定需求视为需求聚合稿，不作为运行时模板；
4. 从已经确认的视角规则拆出第一批最小控制单元；
5. 由 Skill 内部选择少量单元；只有正式测试时建立简化运行记录；
6. 用一次真实章节测试验证 Token、创造性和失败定位；
7. 样板成立后再迁移承接、故事表达和可读性规则；
8. 未证明有效的旧规则继续留在历史设计基线，不批量继承。

## 10. 成功标准

- 单个控制单元可以独立启用、关闭和测试，但不增加用户日常操作；
- Skill 不再默认加载整套方法库；
- 一次失败可以定位到具体单元或输入缺口；
- 作者经验可以版本化演进，不覆盖历史证据；
- Token 随当前任务复杂度增长，而不是随知识库总量增长；
- LLM 在硬边界内仍能产生不同但有效的表达方案；
- 规则升级后，同类错误减少，但正文没有变得更僵硬、更同质。

## 11. 已确认决策

`DEC-MOD-001` 已确认：采用“后台模块化、前台短流程”的简化结构，并以源头 Draft 作为第一个样板模块。

默认前台流程固定为：

```text
用户目标 → Skill 按需取最小资料 → 直接生成 → 硬边界检查 → 用户反馈
```
