# 单元大纲创建需求文档

## 一、定位

本文件属于 Skill 系统中的大纲创建需求文档，用于定义“单元大纲如何由人物卡、事件卡、设定卡和单元上下文包生成”。

它不保存具体单元大纲正文，也不替代单元大纲卡本体。

```text
本文件定义“怎么创建单元大纲”
单元大纲卡保存“创建后的叙事设计结果”
事件卡保存“小说世界中真实发生的故事级事实”
```

单元大纲创建 Skill 处在资料调用和章节大纲之间。

```text
人物卡 + 事件卡 + 设定卡
        ↓
资料调用 / 候选资料包
        ↓
单元上下文包
        ↓
单元大纲创建 Skill
        ↓
单元大纲卡
        ↓
章节大纲卡
        ↓
章节细纲卡
        ↓
正文生成 Skill
```

---

## 二、核心判断

故事级事件卡不是单元大纲。

但故事级事件卡可以成为单元大纲的事实骨架。

二者区别如下：

|文件类型|回答的问题|保存内容|不能保存什么|
|---|---|---|---|
|事件卡|世界里真实发生了什么|事实、因果、状态变化、信息分层、逻辑限制|章节安排、读者体验、情绪节奏、结尾钩子|
|单元大纲卡|这些事实在一个单元中怎么讲|叙事目标、单元结构、信息释放、人物弧线、章节拆分方向|改写世界事实、替代事件卡、正文细节|
|章节大纲卡|单章如何使用单元事实|本章功能、出场人物、事件使用方式、读者/主角信息差|完整事件事实库、场景级执行细节|
|章节细纲卡|单章如何落成场景|场景顺序、行动推进、对白重点、节奏安排|世界事实权威定义|

因此，单元大纲创建 Skill 的核心任务不是“编事件”，而是：

```text
把已经确认的故事级事件事实，组织成一个可写的叙事单元。
```

---

## 三、输入资料

单元大纲创建时，输入资料建议来自：

```text
1. 单元上下文包
2. 故事级事件卡
3. 核心人物卡
4. 必要设定卡
5. 作者提供的单元目标
6. 已有总纲 / 卷纲 / 整体大纲
```

V1.2 样板验证中，可先使用：

```text
马木人物卡
EVT-0001~EVT-0007 事件卡
总体大纲
必要世界观设定
作者本轮确认的单元目标
```

---

## 四、输出产物

单元大纲创建 Skill 的输出不是正文，而是单元大纲卡草案。

建议输出文件类型：

```yaml
type: unit_outline
system: narrative
```

单元大纲卡应回答：

```text
1. 本单元讲哪一个故事级事件或事件组
2. 本单元的叙事目标是什么
3. 本单元的主角认知如何变化
4. 本单元的读者认知如何变化
5. 本单元使用哪些人物
6. 本单元使用哪些事件事实
7. 本单元必须遵守哪些设定限制
8. 本单元如何拆分为若干章节
9. 每个章节只承担什么功能
10. 哪些信息必须隐藏到后续
```

---

## 五、职责边界

### 5.1 单元大纲创建 Skill 负责

```text
1. 读取已确认的单元上下文包。
2. 读取相关故事级事件卡的摘要、因果链、状态变化、信息分层和逻辑限制。
3. 判断一个事件卡是否足以支撑一个单元，或多个事件卡是否应组合为一个单元。
4. 将事件事实组织为单元级叙事推进。
5. 明确主角认知线、读者认知线、人物关系变化和信息释放计划。
6. 给出章节拆分建议，但不写章节细纲。
7. 标记每个章节的功能，而不是写具体场景。
8. 输出待作者确认的单元大纲卡草案。
```

### 5.2 单元大纲创建 Skill 不负责

```text
1. 不改写事件卡中的世界事实。
2. 不把事件卡拆成场景卡。
3. 不生成章节细纲。
4. 不生成正文。
5. 不替代资料调用 Skill。
6. 不替代正文生成 Skill。
7. 不提前释放事件卡中被标记为隐藏的信息。
8. 不把读者奖励、情绪节奏写回事件卡。
```

## 六、事件卡到单元大纲的转换规则

### 6.1 事件卡提供的内容

单元大纲创建时，事件卡主要提供：

```text
事件摘要
世界时间
事件性质
起因
时间线
结果
状态变化
因果链
后续影响
涉及人物
涉及设定
信息分层
逻辑限制
```

### 6.2 单元大纲生成的内容

单元大纲卡应在事件事实基础上生成：

```text
单元标题
单元定位
单元主事件
单元辅助事件
主角认知推进
读者认知推进
人物弧线
信息释放表
章节功能表
单元结尾状态
后续承接
写作禁忌
```

### 6.3 不允许的转换

```text
不允许把事件卡时间线节点直接当章节。
不允许把事件卡中的每个状态变化拆成一个章节。
不允许为了章节爽点改变事件事实。
不允许把世界真相提前写入主角认知。
不允许把单元大纲写成章节细纲或正文片段。
```

---

## 七、单元划分原则

一个单元可以由一个故事级事件支撑，也可以由一个主事件加若干辅助事件支撑。

判断标准：

```text
1. 是否有完整起因、推进、结果和后续影响。
2. 是否能形成主角认知变化。
3. 是否能形成读者认知变化。
4. 是否能支撑多个章节的叙事展开。
5. 是否具有明确单元结尾状态。
6. 是否能承接上一个单元并开启下一个单元。
```

当前样板中：

```text
EVT-0003_六合集市出现事件
可以作为一个完整单元的主事件。

它内部包含：
账户风控
任战异能后果
张凡与小满代价
APP出现

这些是事件内部节点，不应拆成独立事件卡。
```

后续第一卷事件：

```text
EVT-0004_豆豆事件
EVT-0005_苏芳菲事件
EVT-0006_苟弋事件
EVT-0007_未发生的事实事件
```

每个都可以先作为一个单元主事件，后续再逐一细化为单元大纲。

---

## 八、建议流程

```text
1. 作者指定要创建单元大纲的事件卡或事件组。
2. 资料调用机制读取相关人物卡、事件卡、设定卡。
3. 生成或读取单元上下文包。
4. 单元大纲创建 Skill 读取单元上下文包与核心事件卡。
5. Skill 识别单元主事件、辅助事件、人物线和设定限制。
6. Skill 生成单元大纲卡草案。
7. 作者确认单元大纲卡。
8. 确认后的单元大纲卡再用于章节大纲创建。
```

---

## 九、单元大纲卡草案输出要求

单元大纲卡草案必须满足：

```text
完整：能说明本单元从哪里开始，到哪里结束。
清晰：能区分事件事实、叙事安排和信息释放。
可追溯：每个核心事实能追溯到事件卡或人物卡。
可拆章：能自然拆出若干章节功能，但不进入场景执行。
可确认：作者能逐项检查、修改和确认。
```

输出时必须标记：

```text
待作者确认的问题
事实依据不足的位置
不能提前暴露的信息
后续需要细化的人物或事件
```

## 十、与其他文件的关系

|文件|关系|
|---|---|
|事件卡|提供世界事实骨架|
|人物卡|提供人物状态、认知边界、行为边界|
|设定卡|提供能力、组织、世界观限制|
|单元上下文包|提供压缩后的当前单元资料|
|单元大纲卡|保存本 Skill 输出的叙事设计结果|
|章节大纲卡|承接单元大纲，拆分单章功能|
|章节细纲卡|承接章节大纲，拆分场景执行|
|正文生成 Skill|承接确认后的大纲与上下文，生成正文|

## 十一、V1.2 样板验证目标

V1.2 当前不急于创建完整章节大纲和正文。

优先验证：

```text
1. 是否能由 EVT-0003_六合集市出现事件 生成一个单元大纲卡草案。
2. 单元大纲卡是否能保留事件事实，又增强叙事组织。
3. 单元大纲卡是否能明确章节拆分方向，但不越界写成章节细纲。
4. 单元大纲卡是否能作为后续章节大纲创建的输入。
```
