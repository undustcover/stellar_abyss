# 源头 Draft 问题重定需求

> [!note] 模块化边界
> 本文件只聚合当前阶段的需求与问题，不作为运行时模板，也不承载未来所有写作规则。后续按 `04_Skill系统/00_Skill总控/SKL-ARCH_模块化与最小控制单元规范.md`，将确认内容分别拆为方案、经验、方法论、最小控制单元和测试文件；普通生成由 Draft Skill 内部按需选择少量单元，不要求作者维护运行实例。

## 1. 文件定位

本文件用于重新定义测试版 `generate-chapter-prose-draft` 下一阶段要解决的问题。它不是 Skill 实现，也不直接批准任何新规则。

当前目标不是把 Draft 经过更多后处理变成 Public，而是让第一次生成的 Draft 在四个方面成立：

1. 故事表达：读者能跟上当前发生的事、人物诉求、场景变化和章节重点；
2. 章节承接：本章从上一章真实结束状态继续，而不是重新开机；
3. 可读性：读者不需要穿过说明书、规则执行痕迹和平均用力的段落才能获得故事；
4. 故事性：故事通过明确的视角人物被经历、发现和误解，让读者愿意代入并继续阅读。

四者相关但不能合并：故事表达负责“讲清当前故事”，承接负责“故事没有断”，可读性负责“读起来不费力”，故事性负责“这样讲是否有趣、是否让读者进入人物体验”。正文可以清楚且顺畅，却仍然没有故事性。

## 2. 当前 Skill 暴露的源头问题

### 2.1 输入链路过重

当前测试 Skill 默认要求协调：章节创作总控规划、章节级叙事表达契约、章节细纲、场景演出契约、章节执行包、人物意图结果和运行态。

问题不是这些产物都无价值，而是尚未证明它们每章都必须独立存在。多个产物重复描述章节目标、风险、表达和调度时，会产生三种后果：

- 模型把精力用于核对文档，而不是讲故事；
- 同一目标在不同文件里出现轻微差异，增加冲突；
- 正文容易呈现“完整执行设计”的痕迹。

### 2.2 前章承接不是必需输入

现有 `input-contract.md` 只把“前一章摘要”列为可选输入，没有要求读取前章正式结尾，也没有稳定的 `previous_chapter_handoff`。

因此新章可能在地点、站位、动作、物件、情绪余温、未完成对话和当前压力上重新启动。P0C3 测试已经出现过这一问题。

### 2.3 质量规则以禁止项为主

现有 references 对 AI 感、解释、比喻、对白、误会、POV 和笑点提供了大量限制，但对以下正向问题缺少简洁门禁：

- 读者此刻是否知道发生了什么；
- 读者是否知道当前人物想解决什么；
- 每场是否由上一动作或压力自然引发；
- 本段信息是否是当前故事必须知道的；
- 读者读完一场后为什么愿意进入下一场。

### 2.4 故事表达与句面优化混在一起

“讲清故事”“人物像活人”“句子不像 AI”“笑点有效”目前集中在正文导演层同时处理。职责过多时，系统容易优先做可枚举的句式检查，而忽略故事焦点与因果推进。

### 2.5 输出审计过重

当前正文模板的生成备注包含大量风险栏目。完整审计对开发测试有价值，但不应反向成为每次生成时的主要注意力负担。后续需区分：

- 生成前真正影响写作的少量门禁；
- 生成后的开发诊断；
- 给用户阅读的干净 Draft。

## 3. 四个目标的可观察定义

故事表达、章节承接、可读性、故事性和视角切换边界均已由用户于 2026-07-22 确认。

### 3.1 故事表达

一章表达成立，至少满足：

1. 开篇较快恢复人物、地点、当前动作或压力；
2. 读者能判断这一场“谁想做什么、什么在阻碍”；
3. 场景变化来自前一行动、信息、决定或后果；
4. 世界信息只在当前冲突需要时出现；
5. 每场结束时，人物状态、关系、信息或问题至少有一项发生变化；
6. 全章存在一个最主要的推进方向，不被支线笑点和解释抢走。

### 3.2 章节承接

本章开头必须继承上一章结尾的必要状态。用户于 2026-07-22 确认采用四字段最小接口，不把候选信息继续拆成独立表格：

|字段|说明|
|---|---|
|`source_chapter`|上一章正式版本与路径|
|`continuation_anchor`|用一句话合并时间、地点、在场人物和最后动作|
|`must_carry`|只列不能凭空消失的待回应对话、关键物件、情绪余温或现实压力；没有则为空|
|`handoff_question`|读者被带入下一章的一个主要问题；没有明确问题时允许为空|

默认在开始生成下一章时由 LLM 从上一章正式正文内部提取并直接使用，不新增用户表单或确认步骤。只有版本冲突、承接与本章目标冲突、需要改写前章事实或无法可靠判断时才向用户提问。当前不增加 Python、缓存或独立运行记录；真实测试证明有必要后再评估轻量 Harness。

### 3.3 可读性

可读不是“所有信息都解释清楚”，而是：

1. 读者始终知道当前场景在处理什么；
2. 新名词、新人物和新信息不会同时过量进入；
3. 对话能追踪说话者、话题和压力，即使人物答非所问；
4. 段落长短和信息密度随场面变化，不平均用力；
5. 解释不会中断正在发生的动作；
6. 关键冲突被写出来，过场被压缩；
7. 章末留下明确但不过度解释的继续阅读动力。

### 3.4 故事性

故事性单独验收。它首先不是增加情节数量或制造强刺激，而是选择“读者跟着谁经历这段故事”，再通过这个人的认知、视线、欲望、情绪和误判组织材料。

至少检查：

1. 默认每章有一个主视角人物，每个普通场景只有一个视角；必要切换只能发生在明确的场景边界；
2. 场景只直接呈现该人物能够看到、听到、感到、想到或合理推断的内容；
3. 叙述注意力有角色偏向：同一现场中，这个人会先注意什么、忽略什么、误解什么；
4. 场景信息按人物经历顺序出现，而不是按作者后台设计顺序说明；
5. 人物在场景中有当前欲望或压力，视角不是被动摄像机；
6. 读者通过人物的有限认知参与发现、期待、担心、误会或修正判断；
7. 其他人物的真实动机主要通过言行、停顿、物件和后果显露，不直接越过视角解释；
8. 场景结束时，视角人物的认知、情绪、选择或处境发生可感变化；
9. “有趣”来自人物体验与现实之间的距离，包括误判、反差、期待落空、信息差、关系压力或选择后果；
10. 视角限制不能妨碍基本理解：允许读者暂时不知道真相，但不能不知道当前正在发生什么。

#### 视角、故事性与可读性的关系

|情况|判断|
|---|---|
|信息清楚、语言顺畅，但像全知说明|可读，故事性弱|
|严格限制视角，但人物只看不参与|视角形式成立，代入仍弱|
|人物体验强烈，但读者不知道当前动作和目标|可能有氛围，故事表达与可读性失败|
|读者知道当前发生什么，同时只能随人物逐步理解|故事表达、可读性与故事性共同成立|

因此，视角限制不是让信息更少，而是决定信息由谁、以什么顺序、带着什么情绪和误差到达读者。

#### 上帝视角（全景客观视角）特例

本系统中的“上帝视角”专指不能进入任何人物内心的全景客观视角，不等同于可读取多个人物内心的全知叙述。

关键章节中，当多条故事线的人物汇合，或场景规模已经超出任何单个人物可以完整感知的范围时，允许把视角暂时上升为“全景客观视角”。它是受控特例，不是默认叙述模式。

适用场景包括：

- 多线人物在同一时间、地点汇合；
- 战争、灾难、撤离、庆典、仪式、群体冲突等大规模行动；
- 需要建立整体空间关系、行动规模或同时性；
- 章节转折需要让读者看见“个人尚未意识到，但整个局面已经改变”。

使用规则：

1. 必须有明确的规模性叙事目的，不能只为方便补信息而启用；
2. 进入与退出必须有可感边界，例如章节开头、场景分隔、时间节点或全局事件触发；
3. 可以呈现超出单个人物视线的空间、群体、同步动作和客观后果；
4. 不进入任何人物内心，只写可观察事实与外显人物证据；
5. 不得借全景视角提前解释隐藏机制、人物真实动机或未来结果；
6. 全景段落必须有组织中心，例如一个事件、倒计时、空间轴、声音、信号或共同后果，不能变成素材罗列；
7. 建立规模后，应在合适位置重新落回一个或少数核心人物，让宏大事件重新获得身体感、选择和代价；
8. 全景视角与人物视角不得在同一段连续滑动，必须让读者知道叙述范围已经变化；
9. 即使完全客观，叙述仍有选择与顺序，不能把“客观”理解为平均描写所有人和所有事件；
10. 若全景信息会破坏既定悬念或认知边界，必须 ASK_USER。

全景客观视角解决的是“规模与同时性”，人物限知视角解决的是“代入与体验”。宏大章节通常应先建立整体局面，再落回人物承受局面的具体瞬间；二者互补，不互相替代。

#### 外显人物证据规则

全景客观视角不进入人物内心，不等于删除人物细节。相反，人物必须通过读者能够观察的证据存在于场景中，否则宏大叙述会退化为 AI 式事件说明。

允许描写：

- 人物的站位、移动、动作、动作中断和动作失误；
- 表情、呼吸、视线落点、身体反应和衣物状态；
- 说出的语言、语气、音量、停顿、改口和沉默；
- 人物如何拿、放、藏、递、丢或保护关键物件；
- 人物与他人的距离、服从、对抗、回避、保护和协作；
- 人物做出的公开选择，以及选择产生的可见后果；
- 当前环境直接作用于人物身体与行动的结果。

禁止直接断言：

- “他心里想……”；
- “她终于明白……”；
- “他其实害怕／愤怒／后悔……”；
- 人物没有说出或做出的真实动机；
- 只有全知作者才知道的回忆、计划、判断或未来意图。

判断原则：叙述可以写“他第三次把手伸向门把，又收了回来”，但不直接写“他害怕门后的人”；可以写“她回答得很快，手里的证件却一直没有递出去”，但不替读者确定她在怀疑什么。

外显人物证据必须服务当前境况：让读者判断人物此刻受到的压力、处境、关系位置或可能选择。不得把它写成镜头清单，也不得平均分配给所有人物。

## 4. 最小输入候选

下一轮先验证下列最小输入，而不是直接继承全部六产物链：

1. 用户本轮创作目标与明确禁区；
2. `previous_chapter_handoff`；
3. 本章结构骨架：场景顺序、每场视角人物、每场变化、章末状态；
4. 当前章所需的最小世界事实包；
5. 关键人物的当前诉求、认知边界和不能替其决定的事项；
6. 少量已经验证的正文通用原则。

人物状态默认由卷级统一基础行为、单元级记录差异和重大事件前后变化，章节不重新定义。Draft 生成前只运行轻量人物意图审核并保存 `PASS / ASK_USER` 日志：审核把控人物方向、认知边界和转变位置，不规定具体动作、台词或表演路径；只有资料无法裁决且会改变人物、关系、剧情或信息边界时才询问用户。

故事表达只审核四项结果：当前故事可定位、推进关系可跟随、故事通过人物被经历、存在继续动力。四项不规定固定开场、节拍、反转、钩子或具体实现；若达到目标所需的方法尚未在对应模块 Skill 中由作者确认，必须先与作者讨论方法。

源头 Draft 采用“模块入口 → 少量模块 Skill → 少量适用方法”的三层运行结构。方法属于对应模块 Skill，不作为独立 RAG 库；当前不预建空 Skill、向量检索、Python 路由或多人物 POV 调用规则。

每个单元先形成作者确认的单元上下文包和单元总体规划：前者压缩本单元相关事实与边界，后者保存作者的单元创作安排。每个独立章节对话框默认读取 `单元上下文包 + 单元总体规划 + 章节大纲／细纲 + 章节 Draft Brief`，不重复读取全量原始资料。

默认章节级中间产物收缩为章节 Draft Brief 与一份 Draft 运行日志。大型章节契约和总控规划退出默认链路；场景演出契约与章节执行包只在方法共创、复杂调度、恢复、测试或故障复现时按需启用。

章节契约、场景演出契约、执行包、人物意图结果等现有产物暂不删除。P1 要逐项判断它们属于：

- 默认必需；
- 高风险章节按需启用；
- 可合并字段；
- 仅保留为开发诊断；
- 应停止使用。

## 5. 下一步设计顺序

1. 确认目标、视角边界、最小接口与问题定位地图；
2. 选定一章，对单元上下文包、单元总体规划、章节大纲／细纲和 Draft Brief 做就绪审计；
3. 冻结上一章正式正文与本章输入版本，补齐阻断缺口；
4. Codex 不修改旧测试 Skill，按 confirmed 新链路人工生成测试 Draft 与紧凑运行日志；
5. 作者只审阅最终 Draft，以自然语言评价或直接修改；
6. Codex 根据定位地图一次只修一个有证据的首要责任位置，并重新生成；
7. 作者确认可接受 Draft 基线后，再修改 `.codex-tmp/skills/generate-chapter-prose-draft/`；
8. 用同一冻结输入验证 Skill 复现，再使用相邻章节或保留输入回归。

## 6. 当前禁止动作

- 不迁移、安装或正式化 Draft Skill；
- 不恢复 Draft-to-Public；
- 不先增加新的后处理 Skill；
- 不把所有历史文字质量规则默认装入新链路；
- 不读取历史归档正文作为生成素材；
- 不在目标和最小输入未确认前生成新的正式正文。
- 不继续把新增经验和规则全部堆入本需求聚合稿或单一 Skill reference。

## 7. 待用户确认

`QUESTION-001`：故事表达、承接和可读性的基本定义已确认；故事性已单列并以视角人物的有限体验为核心。

`QUESTION-002`：下一轮应选哪一章做基线。这个决定在测试方案阶段确认，本轮不预设。

`QUESTION-003`：已确认默认一章一个主视角、一个场景一个视角；必要时只在明确场景边界切换。关键多线汇合或宏大场景允许全景客观视角特例。

`QUESTION-004`：已确认全景客观视角不进入任何人物内心，但必须通过言行举止、身体反应、物件处理、关系动作和可见后果保留人物细节，让读者自行判断人物的实际境况。
