# 笑话写作方法论库运行说明

## 一、目录定位

本目录保存可以在正文生成时直接调用的幽默与笑话写作方法。它解决的是：

> 当前场景需要幽默时，可以从什么现实材料出发，经过哪些步骤，写出哪一类笑点，并如何避免退化成硬梗、套话或 AI 式段子。

本库不是笑话成品库，不保存作者原句供模型仿写，也不要求每个场景都制造强笑点。

## 二、与现有三类资料的关系

|资料|回答的问题|使用优先级|
|---|---|---|
|作者审美决策库|作者喜欢什么、拒绝什么、文学价值如何裁决|决定是否应该使用幽默|
|笑话写作方法论库|当前素材怎样加工成幽默或笑点|决定具体怎样生成|
|结构化喜剧写法|喜剧的一般理论、分类和研究候选|用于扩展候选和解释机制|
|作者经验归纳|历史经验与综合写法总结|用于补充参考，不直接视为已验证方法|

当资料发生冲突时，先服从当前场景事实和作者审美决策，再使用本库条目；广泛研究不能覆盖作者已经确认的边界。

## 四、条目等级

|等级|含义|调用方式|
|---|---|---|
|L1 候选方法|模型或研究提出，作者尚未确认|只能用于实验，不能作为作者方法|
|L2 重复有效|在多个案例中有效，但适用边界仍需收窄|结合场景谨慎使用|
|L3 作者确认|作者明确确认可作为喜剧写作手法|适用条件成立时可以调用|

L3 只表示作者确认，不表示任何场景都适用。每次调用仍需检查现实锚点、人物归属、喜剧对象和场景底色。

## 五、正文生成调用流程

```text
确定场景任务
→ 找当前事实、物件、话术、关系或群体语义
→ 从方法索引选择一个主方法
→ 按条目步骤生成多个候选
→ 检查人物与现实真实性
→ 检查落点和失败边界
→ 只保留真正推进人物、关系或读感的版本
```

不要先选一个笑话模板，再强行修改剧情事实来迁就它。

## 六、每轮更新协议

1. 在案例层保留原句、修改对和作者解释。
2. 只将作者确认的方法写入 L3；未确认候选只能写 L1。
3. 同类证据优先更新旧条目的适用条件、步骤和失败边界，不重复造条目。
4. 新方法必须包含：素材入口、核心机制、生成步骤、适用条件、失败边界、自检问题和来源案例。
5. 在变更日志记录新增、升级、收窄、合并或废弃。
6. 更新当前工作状态的最小读取集。
7. 正式方法库不复制作者原句或具体成品笑话。

## 七、最低质量门槛

一个候选笑点至少通过以下检查：

- 它依附于当前真实事实，而不是外插段子吗？
- 换一个人物还能原样使用吗？如果能，是否说明人物归属太弱？
- 读者不看解释，能理解基础逻辑和意外点吗？
- 笑点之后，人物、关系、处境或读者理解是否至少推进一项？
- 被笑的对象是否准确，是否误伤无辜或破坏人物好感？
- 删除这个笑点，剧情会更干净还是更苍白？
