# 模块 Skill 与方法稀疏调用

## 核心决定

作者方法论不是独立 RAG 知识库，也不堆入一个整体写作 Skill。方法作为对应模块 Skill 的内部组成，由 Skill 按需读取。

运行时只保留三层：

```text
源头 Draft 模块入口
→ 选择少量模块 Skill
→ 每个 Skill 只读取少量适用方法
```

系统分类、测试证据和后台生命周期目录不算运行步骤。

## 模块入口职责

源头 Draft 模块入口只负责最小输入、前章承接、审核、能力选择、方法缺口判断、作者询问、生成调度和运行日志，不保存全部写作方法。

## 模块 Skill 建立门禁

一种能力只有同时具有独立问题、触发条件、方法集合和测试边界，并且不拆分会使现有 Skill 持续膨胀时，才建立独立模块 Skill。

当前只确认以下候选能力：

- 人物限知 POV；
- 上帝视角；
- 人物意图审核；
- 故事表达四项审核。

轻喜剧等写作基底保留扩展位置，等作者共创出第一批方法后再建立。普通场景、局部技巧和单次桥段不提前升级为 Skill。分类先作为注册字段，出现真实能力后才建目录。

## 视角边界

本系统统一定义：`上帝视角 = 全景客观视角`。它可以观察超出单个人物视野的空间、同步行动、外显人物证据和客观后果，但不能进入任何人物内心。

一章默认一个主 POV、一个场景一个 POV。未来若真实创作需要同章展示多个 POV，再讨论 Skill 调用和切换规则；当前只占位，不实现。

## 方法冷启动

生成目标需要某种写作方法时，模块 Skill 先检查内部是否已有作者确认的适用方法：

- 有适用方法：记录方法 ID 后调用；具体实现由 LLM 创作；
- 没有方法或适用性不清：说明目标、缺口、候选方向、影响和建议，询问作者；
- 作者确认后：抽象为方法，排除具体成品句与固定桥段，并更新该 Skill 的方法索引。

方法只使用两个状态：

- `待正文验证`：作者已确认，可用于当前和相同情境；
- `已验证`：实际产物经作者确认达到预期效果。

## 稀疏读取

每个模块 Skill 保持简短核心流程，具体方法分文件保存，并使用轻量索引定位。当前由 LLM 读取小型索引；不建立向量 RAG，也不实现 Python 检索。只有真实方法数量证明索引读取成为成本后，才评估确定性标签过滤。

运行日志必须记录实际选择的模块 Skill、方法 ID、方法缺口和询问结果，以便最终产物失败时区分选择错误、方法错误和执行错误。

## 当前边界

本方案只确定骨架，不创建正式 Skill 源码，不迁移、安装或修改任何测试 Skill，不预建空 Skill 或多层物理目录。
