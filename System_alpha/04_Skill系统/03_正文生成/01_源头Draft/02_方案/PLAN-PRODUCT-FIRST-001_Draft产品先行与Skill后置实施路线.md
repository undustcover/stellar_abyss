# Draft 产品先行与 Skill 后置实施路线

## 核心决定

2026-07-22，用户确认采用“产品先行、Skill 后置”：先由 Codex 按已确认的新链路人工执行一次 Draft 生产，让作者只通过最终 Draft 判断问题；通过运行日志和定位地图在后台反推责任位置。只有获得作者认可的 Draft 基线后，才修改测试 Skill，并验证 Skill 能否稳定复现该生产路径。

Draft 产品测试不依赖先修改 Skill；Skill 行为测试必须在实现修改之后进行。两者不得混为一轮。

## 作者前台

作者默认只需要：

1. 说明本章目标、明确不要什么；
2. 阅读 Draft；
3. 指出不满意、看不懂、不相信、不像人物或不想继续读的位置，也可以直接修改；
4. 判断修正方向及最终 Draft 是否达到可接受基线。

作者不需要理解模块、控制单元、方法 ID、运行日志字段或失败码。只有问题会改变人物、世界事实、审美方向或剧情路线时，Codex 才把必要选择翻译成自然语言询问作者。

## Codex 后台

Codex 负责：

- 冻结并记录本次实际输入版本；
- 按 confirmed 接口人工执行源头 Draft 链路；
- 保存紧凑运行日志；
- 把作者对最终 Draft 的反馈转成可观察问题；
- 使用 `MAP-SOURCE-DRAFT-001` 找到一个有证据的首要定位点；
- 只修改该责任位置并重新生成；
- 保留具体表达的创作自由，不把作者修改稿机械转成通用规则。

## 实施阶段与门禁

### 阶段 A：人工试产准备

1. 选择一章作为基线；
2. 冻结上一章正式正文和本章相关输入版本；
3. 准备并确认单元上下文包、单元总体规划、章节大纲／细纲；
4. 生成并确认章节 Draft Brief；
5. 检查卷级人物基础、单元人物差异、关键事实和方法缺口；
6. 存在会改变创作方向的缺口时先询问作者。

门禁：四类默认章节输入可追溯且无阻断问题；测试 Skill 保持不变。

### 阶段 B：人工 Draft 产品验证

1. Codex 不调用或修改旧测试 Skill，按新链路人工生成一份测试 Draft；
2. 同时保存一份紧凑运行日志；
3. 作者只审阅 Draft，以自然语言评价或直接修改；
4. Codex 将反馈定位到事实、单元规划、章节结构、Brief、handoff、人物意图、控制单元、方法或执行层；
5. 一次只修首要责任位置，重新生成并让作者复核；
6. 直到作者明确确认该章达到可接受 Draft 基线。

门禁：作者认可的是故事与阅读体验，不要求复现固定句子、桥段或修改稿表达。

### 阶段 C：Skill 实现与复现验证

只有阶段 B 通过后才允许：

1. 将已验证的最小输入、路由、审核和日志行为写入测试 Skill；
2. 使用阶段 B 的同一组冻结输入运行 Skill；
3. 比较作者体验、事实边界、人物方向、承接和故事四项结果，不比较固定句子；
4. 定位 Skill 选择、接口读取或执行差异并迭代；
5. 使用相邻章节或保留输入做回归，防止只适配基线章。

门禁：Skill 能稳定执行已验证路径，且没有因规则增加导致正文僵硬或前台流程膨胀。

### 阶段 D：晋升判断

只有真实章节、作者确认、近迁移或回归验证及治理门禁都通过后，才讨论测试 Skill 的正式化、迁移和安装。本路线本身不授权这些操作。

## 产品基线的含义

“先完成 Draft 质量保证”不是在修改 Skill 前保证所有未来章节永远合格，而是先得到一份作者认可、问题可定位、修正可复现的真实章节基线。

作者至少应能确认：

- 当前故事能看懂并愿意跟随；
- 人物方向和关键选择可信；
- 前章承接、事实和信息边界没有明显错误；
- 故事通过人物被经历，而不是后台说明；
- 章末仍有符合本章目标的继续动力；
- 整体已经达到作者愿意让 Skill 尝试复现的 Draft 水平。

## 禁止倒置

- 不先改 Skill 再让作者替系统发现产品方向；
- 不要求作者审阅技术日志来证明 Draft 好坏；
- 不因一次作者修改同时改多个模块；
- 不把人工基线的具体句子当成 Skill 必须复刻的答案；
- 不把手动执行成功误认为 Skill 已经实现或正式可用。
