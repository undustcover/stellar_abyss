# 正文生成链路改造方案 V2.4

## 一、核心结论

当前正文链路改造采用：

```text
前三个 skill 前置分场景导演，自下而上测试，最后再自上而下全链路验证。
```

原因：

1. 当前最不稳定的是正文生成效果；
2. 如果先从单元上下文包、单元大纲开始测，会把问题扩散到全链路，难以定位；
3. 先把正文生成、章节执行包、细纲模板逐层打磨稳定，再反推上游模板与资料调用规则，风险更低；
4. 最终仍然要回到完整正向链路，从资料调用到正文生成完整执行一次。
5. 去 AI 化、文风一致性和对白幽默化不再作为默认后置 skill，而是并入细纲、执行包和正文生成的分场景导演逻辑。

当前默认主链路：

```text
章节细纲：锁读者体验曲线
    ↓
章节执行包：分场景导演 + A/B/C 场景等级 + 字数预算
    ↓
正文生成：按导演表一次生成，内置去 AI 感、文风和对白幽默
```

失败后的返修保险：

```text
草稿诊断 -> 定位失败场景 -> 分场景再导演 -> 正文局部替换
```

返修保险只处理失败场景，不作为默认整章后处理流程。

---

## 二、两条链路必须分开

### 2.1 生成链路

生成链路不得读取合格正文参照稿。

```text
系统资料 / 已批准规则
    ↓
章节级测试资料包
    ↓
章节执行包
    ↓
正文生成 skill
    ↓
测试正文
```

生成链路允许使用：

1. 设定系统；
2. 事件数据库；
3. 叙事系统；
4. 人物卡、事件卡、设定卡；
5. 章节文风模式；
6. 角色语言与幽默字段；
7. 结构化喜剧规则；
8. 分场景导演规则；
9. 当轮测试明确提供的系统资料摘要。

生成链路禁止使用：

1. 合格正文参照稿；
2. 外部成稿；
3. 对比报告；
4. 成品样文；
5. 成品梗；
6. 成品对白；
7. 成品落点；
8. 非系统写作材料中的段落顺序或标志性表达。

### 2.2 测试评估链路

测试评估链路必须读取合格正文参照稿。

```text
测试正文
    +
合格正文参照稿
    ↓
正文测试评估报告
    ↓
归因到正文 skill / 执行包模板 / 执行包 skill / 细纲模板 / 人物卡 / 事件卡 / 设定卡
    ↓
只回写通用规则、模板字段或系统资料缺口
```

合格正文参照稿当前为：

```text
System_alpha/01_叙事系统/06_正文/P001/Public-C0001-第1章 我那一无是处的堂哥.md
```

它的用途是评估生成稿是否达成同等章节内核和读者体验，不是生成输入，也不是结构模板。

---

## 三、测试边界

### 3.1 追求内核一致

测试正文需要尽量靠近合格正文参照稿的章节内核。

|维度|要求|
|---|---|
|读者体验|开篇吸引力、阅读顺滑度、轻喜入口、后续期待应达到同等级|
|剧情推进|推进到同一关键状态，不偏离本章承担的故事任务|
|章节功能|完成第一章的开局、人物建立、误会滤镜、钩子建立|
|人物形象|人物给读者留下的基本印象一致|
|人物关系|关系压力、误判、亲疏感和冲突位置一致|
|信息边界|隐藏信息不得提前泄露|
|情绪落点|结尾情绪方向一致，可以表现方式不同|
|类型气质|轻喜外壳、现实压力、后续不安的组合比例接近|

### 3.2 保持表现独立

测试正文不需要、也不应该追求形似。

|维度|要求|
|---|---|
|场景数量|可以不同|
|场景顺序|可以不同|
|场景物件|可以不同|
|具体桥段|可以不同|
|具体梗|必须不同或自主生成|
|具体对白|必须不同或自主生成|
|段落结构|不得逐段复刻|
|结尾句|必须自主生成|

评估问题不是：

```text
生成稿像不像参照稿？
```

而是：

```text
生成稿是否在不复刻参照稿表现形式的情况下，达成同等章节功能和读者体验？
```

## 四、自下而上测试路线

### 阶段 1：正文生成测试

目标：

```text
验证正文 skill 能否根据执行包生成合格章节正文。
```

当前测试对象：

```text
P00-C01 我那一无是处的堂哥
```

当前允许使用临时章节级系统资料包，不强制先完成 confirmed 单元上下文包。

原因：

1. 当前重点是正文效果，不是上游资料调用完整性；
2. 临时章节级系统资料包必须只来自系统资料；
3. 测试失败后先判断是否是正文 skill 或执行包问题；
4. 不把单元上下文包拖进同一轮测试，避免问题混杂。

产物：

```text
.codex-tmp/skill-tests/generate-chapter-prose-draft/DRAFT-P00-C01_我那一无是处的堂哥_自动生成测试正文草稿.md
.codex-tmp/skill-tests/generate-chapter-prose-draft/EVAL-P00-C01_我那一无是处的堂哥_正文测试评估报告.md
```

### 阶段 2：反推章节执行包

目标：

```text
根据正文测试缺口，完善章节执行包模板与 generate-chapter-execution-pack skill。
```

重点判断：

1. 执行包是否给出了足够的章节内核；
2. 是否给了生活压力链、误会机制、对白断裂点；
3. 是否给了读者体验目标、场景导演目标、A/B/C 场景等级和字数预算；
4. 是否只给结构位，而没有塞成品梗；
5. 是否允许正文 skill 自主规划具体表现形式；
6. 是否能让不同章节生成不同结构、不同节奏、不同情绪曲线。

产物：

```text
.codex-tmp/skill-tests/generate-chapter-execution-pack/PACK-P00-C01_我那一无是处的堂哥_自下而上测试执行包.md
```

### 阶段 3：反推章节细纲

目标：

```text
根据执行包需要，完善章节细纲模板与 generate-chapter-detail-outline skill。
```

重点判断：

1. 细纲是否提供章节功能、场景功能、信息边界；
2. 是否给出人物情绪与即时目的；
3. 是否给出误会成立条件；
4. 是否给出读者体验曲线和每场主要读者效果；
5. 是否保留足够空间给执行包和正文 skill 自主设计；
6. 是否避免写成具体正文桥段。

### 阶段 4：反推单元大纲

目标：

```text
根据细纲需要，完善单元大纲模板与单元大纲 skill。
```

重点判断：

1. 单元大纲是否能稳定提供章节功能；
2. 是否能标出读者/主角信息差；
3. 是否能明确章节文风模式；
4. 是否能承接事件卡，但不改写事件事实。

### 阶段 5：反推单元上下文包和资料调用

目标：

```text
根据单元大纲和正文生成需要，完善单元上下文包与资料调用 skill。
```

重点判断：

1. 单元上下文包是否能压缩人物、事件、设定；
2. 是否能保留人物语言、认知边界、信息隐藏；
3. 是否能减少正文生成时全文读取资料库；
4. 是否能稳定服务后续多章生成。

### 阶段 6：正向完整流程验证

当自下而上各层模板稳定后，再执行完整正向链路：

```text
资料调用
    ↓
单元上下文包
    ↓
单元大纲
    ↓
章节大纲
    ↓
章节细纲
    ↓
章节执行包
    ↓
正文生成
    ↓
测试评估
```

---

## 五、第一章测试的评估颗粒度

第一章只是测试对象，不是通用模板。

不得从第一章参照稿中抽出固定结构，强行套到其他章节。

第一章测试只用于判断：

1. 生成链路能否达成同类章节内核；
2. 执行包是否足够支撑正文生成；
3. 正文 skill 是否能自主组织场景和具体表达；
4. 模板字段是否足够通用；
5. 人物卡、事件卡、设定卡是否缺字段。

---

## 六、评估报告归因规则

每次正文测试失败，必须归因到具体层级。

|问题类型|归因方向|
|---|---|
|正文像说明书|正文 skill 去 AI 化规则不足，或执行包缺少场面压力|
|人物语言同质化|正文 skill 角色语言执行不足，或人物卡语言字段不足|
|笑点硬|执行包喜剧机制不足，或正文 skill 没能转成场景行为|
|误会不成立|执行包误会语义链不足，或事件/人物认知边界不足|
|剧情推进偏|执行包章节内核不清，或上游章节功能不清|
|读者体验弱|执行包没有明确章节读者体验，或正文 skill 开场/节奏能力不足|
|和草稿几乎无差别|正文生成缺少场景导演目标，或把生成任务误当后置润色|
|字数失控|执行包缺少 A/B/C 场景等级和字数预算，或正文 skill 没按等级压缩|
|需要整章重写|先判断是否只有单场失败；优先局部替换，只有多场同类结构失败才回写上游|
|信息泄露|执行包信息边界不足，或正文 skill 没执行隐藏规则|
|过度复刻参照稿|测试流程污染，必须重新生成，不得视为合格|

## 七、过程性反哺记录

正文测试不允许只在最终稿完成后一次性反推 skill。

从第二章测试开始，每一轮正文评估、作者修订对比和问题总结之后，都必须同步维护一份 `skills反哺记录.md`。

该记录用于保存“过程中的深度思考”，包括：

1. 当前正文版本暴露出的具体问题；
2. 本轮问题可能对应的流程层级；
3. 当前选择先修正文、暂不修改 skill 的理由；
4. 后续确认该问题是否应回写 skill 的条件；
5. 若确认回写，可能修改的 skill、模板或规则方向。

`skills反哺记录.md` 不是正文修改稿，也不是立即修改 skill 的指令。

它的作用是把每一轮正文修改中的判断保留下来，避免最后只对最终成稿负责，忽略中间反复出现的问题、被排除的误判和逐步收敛出的真实流程缺口。

### 7.1 记录时机

以下节点必须追加记录：

1. 测试正文评估报告完成后；
2. 作者第一版、第二版、第三版等重修稿对比后；
3. 每次问题总结完成后；
4. 决定暂不修改 skill、继续修正文时；
5. 决定正式修改 skill 前。

### 7.2 记录边界

允许记录：

1. 通用流程缺口；
2. 多轮反复出现的问题；
3. 暂不回写的原因；
4. 待观察的归因假设；
5. 未来可能修改的 skill 方向。

禁止记录为通用规则：

1. 某一章的成品句；
2. 某一章的成品梗；
3. 某一章的具体桥段；
4. 某一版正文的临时表达；
5. 只适用于单一章节的特殊结构。

### 7.3 与正式回写的关系

正式修改 skill 前，必须先回看 `skills反哺记录.md`，按以下顺序判断：

1. 该问题是否在多轮或多章中反复出现；
2. 该问题是否能归因到具体 skill、模板或资料字段；
3. 该问题是否能转化为通用规则，而不是成品表达；
4. 是否存在更小的修正方式；
5. 是否会污染后续章节的表现独立性。

只有通过以上判断的问题，才能进入正式回写。

---

## 八、回写规则

允许回写：

1. 通用模板字段；
2. skill 通用规则；
3. 人物卡语言字段缺口；
4. 事件卡信息分层缺口；
5. 设定卡限制字段缺口；
6. 章节执行包字段；
7. 章节细纲字段；
8. 单元大纲字段；
9. 单元上下文包摘要规则。

禁止回写：

1. 参照稿中的成品梗；
2. 参照稿中的成品对白；
3. 参照稿中的具体段落顺序；
4. 参照稿中的高识别度表达；
5. 只适用于第一章的特殊结构。

## 九、当前下一步

当前不进入单元上下文包测试。下一步重点改造前三个测试 skill，而不是继续测试独立正文修订 skill。

下一步执行：

1. 将 `generate-chapter-detail-outline` 改为输出 `读者体验曲线` 和 `场景导演目标`；
2. 将 `generate-chapter-execution-pack` 改为输出 `分场景导演表`、A/B/C 场景等级和字数预算；
3. 将 `generate-chapter-prose-draft` 改为按导演表一次生成，内置去 AI 感、文风一致性和对白幽默；
4. 删除独立后处理 skill：`revise-draft-de-ai`、`revise-draft-style-calibration`、`revise-draft-dialogue-humor`；
5. 用第三章重新测试前三个 skill 的正向链路；
6. 仅对失败场景执行局部替换测试。

## 十、判断标准

当前阶段不以“流程完整”为合格标准。

当前阶段合格标准是：

1. 测试正文不读取合格正文参照稿；
2. 测试评估必须读取合格正文参照稿；
3. 测试正文与参照稿内核一致；
4. 测试正文与参照稿表现独立；
5. 差距能归因到具体流程层级；
6. 回写内容是通用规则或模板字段，不是参照稿成品表达。
