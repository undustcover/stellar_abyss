# 单元上下文包生成需求

## 一、定位

本文件属于 Skill 系统的资料调用需求文档，用于定义单元上下文包如何生成。

它不保存具体小说内容，也不替代单元上下文包本体。

```text
本文件定义“怎么生成”
单元上下文包保存“生成后的确认内容”
```

---

## 二、输入资料

单元上下文包生成时，候选资料来自：

```text
单元大纲
章节大纲
章节细纲
人物卡
人物经历线
事件卡
设定卡
已有正文摘要
资料调用 Skill
正文生成 Skill
```

V1.2 样板验证中，可先使用：

```text
人物卡
事件卡
章节大纲卡
章节细纲卡
必要设定卡
```

---

## 三、脚本负责的内容

脚本适合生成候选资料包，不负责最终判断。

脚本可负责：

```text
读取 YAML 字段
按 system、type、unit、chapter、line、role、event_type 筛选文件
读取 rel 中的关联对象
抽取固定 Markdown 标题下的内容
拼接候选资料
标记来源文件
统计候选资料长度
检查缺失字段
检查未解析 rel
```

建议脚本抽取内容：

```text
人物卡：LLM调用摘要、核心定位、当前状态、已知信息范围、未知信息范围、误解信息、写作禁忌
事件卡：事件摘要、世界时间、事件性质、时间线、结果、后续影响、逻辑限制
章节大纲卡：章节目标、相关事件、事件使用方式、读者知道的信息、主角知道的信息、本章释放/隐藏/误导的信息
章节细纲卡：本章执行目标、场景列表、必须写出的内容、禁止写出的内容
设定卡：核心规则、当前阶段可暴露内容、限制条件、写作禁忌
```

---

## 四、LLM 负责的内容

高价值 LLM 负责根据候选资料包生成单元上下文包草案。

LLM 应负责：

```text
判断哪些人物信息对本单元重要
判断哪些事件内容必须进入本单元
判断哪些设定规则必须保留
压缩重复信息
保留逻辑限制
区分读者已知、主角已知、主角未知、主角误解
整理本单元信息释放计划
提出待作者确认问题
```

LLM 不得直接把草案视为最终结论。

---

## 五、作者确认

单元上下文包必须经过作者确认。

推荐流程：

```text
1. 脚本或人工生成候选资料包
2. LLM 生成单元上下文包草案
3. 作者检查人物、事件、设定、信息释放和禁忌是否准确
4. 作者补充遗漏内容
5. 作者删除不应进入本单元的内容
6. 状态改为 confirmed
```

正文生成 Skill 默认只读取 confirmed 状态的单元上下文包。

---

## 六、输出要求

单元上下文包必须满足：

```text
完整：覆盖本单元正文生成必须知道的信息
简洁：不搬运原始资料全文
可确认：作者能逐段检查和修改
可追溯：关键内容能通过 rel 或双链追溯到原始文件
可复用：同一单元内多个章节可以复用
```

---

## 七、暂不纳入 V1.2

```text
自动化 CLI
向量数据库
embedding 检索
低价值 LLM 摘要
分卷上下文包正式模板
章节调用摘要正式模板
```
