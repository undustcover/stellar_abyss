# 资料调用需求文档

## 一、定位

资料调用属于 Skill 系统中的前置工作层。

它不负责生成正文，也不负责保存小说设定本身，而是负责在生成大纲、章节细纲或正文之前，从设定系统、事件数据库和叙事系统中筛选、提取、压缩并组织任务所需资料。

资料调用的核心目标是：

```text
避免 LLM 事无巨细地读取全部 Markdown 文档
在生成任务前形成完整、简洁、节约 token 的单元上下文包
保证正文生成 Skill 使用的是经过筛选和摘要后的资料
```

---

## 二、核心原则

```text
1. 资料库可以详细，但任务调用必须分层。
2. LLM 不默认全文读取人物卡、事件卡、设定卡或叙事文件。
3. 每个单元正式生成正文前，应先形成作者确认的单元上下文包。
4. 脚本可负责确定性筛选、字段读取、段落抽取和资料拼接。
5. 高价值 LLM 可负责最终摘要、取舍和压缩。
6. 摘要结果必须保存为 Markdown 文件，避免重复计算。
7. 正文生成 Skill 默认读取单元上下文包，不直接读取全量资料库。
```

---

## 三、资料调用分层

资料调用采用分层机制：

```text
原始资料层
    人物卡、事件卡、设定卡、章节大纲、章节细纲、已有正文等

候选资料层
    由脚本或人工根据 YAML、rel、tags、unit、chapter、line 等字段筛选出来

摘要资料层
    对候选资料进行压缩，形成可供单元使用的精简摘要

单元上下文包
    面向某一固定单元的作者确认型稳定输入资料包
```

---

## 四、资料调用产物

资料调用产物分为两类：

```text
候选资料包：脚本或人工根据字段与关联抽取出的原始候选内容
单元上下文包：LLM 整理、作者确认后的稳定上下文文件
```

单元上下文包是 V1.2 的核心产物。

它应回答：

```text
本单元要写什么
本单元必须使用哪些人物
本单元必须使用哪些事件
本单元必须遵守哪些设定
本单元哪些信息可以释放
本单元哪些信息必须隐藏
本单元有哪些写作禁忌
```

单元上下文包建议保存在叙事系统中，而不是保存在资料调用 Skill 文件夹中。

建议位置：

```text
01_叙事系统/00_叙事总控/
01_叙事系统/05_章节/P0_样板验证/C00_样板章节/
```

---

## 五、资料调用 Skill 的职责

资料调用 Skill 负责定义：

```text
1. 当前任务需要读取哪些资料类型
2. 如何根据 YAML 字段和 rel 关系筛选候选资料
3. 哪些正文段落属于高优先级调用内容
4. 哪些内容只能摘要，不能全文放入上下文
5. 如何区分读者可知信息、主角可知信息和隐藏信息
6. 如何先生成候选资料包，再生成单元上下文包
7. 如何控制上下文长度
```

资料调用 Skill 不负责：

```text
1. 生成最终正文
2. 设计章节节奏
3. 执行完整一致性校验
4. 替代人物卡、事件卡或设定卡
5. 直接充当 Prompt 模板
```

---

## 六、脚本与 LLM 的分工

脚本适合负责：

```text
读取 YAML 字段
按 system、type、unit、chapter、line、role、event_type 等字段筛选文件
读取 rel 中的关联对象
抽取指定 Markdown 标题下的内容
拼接候选资料包
统计长度
检查模板缺字段
```

高价值 LLM 适合负责：

```text
判断哪些资料对当前任务真正重要
压缩人物、事件和设定信息
保留不可丢失的逻辑限制
识别信息释放边界
生成完整、简洁、节约 token 的单元上下文包草案
```

V1.2 不引入低价值 LLM。

## 七、P0 样板验证中的资料调用流程

P0 样板验证建议采用以下流程：

```text
1. 确定样板章节任务
2. 读取章节大纲卡
3. 读取章节细纲卡
4. 根据 rel 筛选出场人物卡、相关事件卡和必要设定卡
5. 抽取人物卡中的调用摘要、当前状态、已知信息范围、写作禁忌
6. 抽取事件卡中的事件摘要、世界时间、时间线、结果、逻辑限制
7. 抽取章节大纲中的事件使用方式、读者知道的信息、主角知道的信息
8. 生成候选资料包
9. LLM 根据候选资料包生成单元上下文包草案
10. 作者确认并修订单元上下文包
11. 正文生成 Skill 读取 confirmed 单元上下文包生成正文
12. 记录资料调用缺口
```
