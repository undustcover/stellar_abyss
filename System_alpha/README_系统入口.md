# System_alpha 系统入口

## 1. 系统定位

`System_alpha` 是本仓库唯一的正式系统目录。系统面向两类参与者：

- 面向用户：通过讨论、确认、修改和反馈逐步表达创作目标与审美判断；
- 面向 Codex：通过稳定的目录、状态、模板、Skill 契约和路由规则执行任务；
- 共同目标：把用户需求转化为可验证、可回退、可继续迭代的系统能力，而不是一次性文稿。

当前产品形态是“以 Markdown 文件为状态与知识载体、以 Codex 对话和 Skill 为执行界面”的 AI 协作式小说创作系统。它还不是独立 GUI 软件。

## 2. 开发前必读顺序

任何新窗口、新模型或新开发任务，先按顺序读取：

1. 本文件：确认目录边界和入口；
2. `00_总控台/DEV-ARCH_系统架构与边界.md`：确认系统分层与职责；
3. `00_总控台/DEV-ROADMAP_当前开发路线.md`：确认唯一活跃阶段和下一步；
4. `00_总控台/DEV-GOV_迭代与防漂移规范.md`：确认修改、测试、晋升和提问规则；
5. 与任务直接相关的正式需求、模板和已确认产物；
6. 只有需要理解设计理由时，才读取 `../设计流程与方案/`；
7. 只有需要复现或评估时，才读取 `.codex-tmp/` 和 `../测试文本/`；
8. `../历史归档/` 默认禁止进入当前执行上下文。

## 3. 仓库目录边界

|目录|定位|能否直接作为正式执行依据|
|---|---|---|
|`System_alpha/`|正式系统：规则、模板、系统事实、叙事产物、正式需求与开发控制文本|可以，但仍须检查文件状态|
|`.codex-tmp/skills/`|可执行 Skill 的测试实现|不可以自动视为正式；必须标明测试版本|
|`.codex-tmp/skill-tests/`|测试输入、输出、问题和归档|不可以|
|`设计流程与方案/`|设计理由、提案、实施记录与版本演进|仅作设计依据；不能覆盖正式文件|
|`测试文本/`|人工与 Skill 测试材料|不可以；经用户确认后按流程晋升|
|`核心知识/`|待评估的跨项目知识或素材|不可以自动进入系统|
|`架构设计/`|上位架构思考|不可以自动覆盖本系统架构|
|`历史归档/`|停止、失败、替代或仅供追溯的资料|禁止作为当前默认输入|
|`todolist.md`|历史讨论与用户临时记录|可提供线索，不再充当唯一任务看板|

`DEV-MASTER` 保留历史信息和模块索引，但其中日期早于当前审计的状态、Todo 和“下一步”不得覆盖当前路线图。

## 5. 冲突裁决顺序

同一问题出现多个答案时，按以下顺序判断：

1. 用户本轮明确指令；
2. 当前路线图中已确认的决策；
3. 正式系统内状态为 `active` 或 `confirmed` 的需求、规则和产物；
4. 正式模板与接口契约；
5. 处于 active 状态的设计文档；
6. 测试 Skill 与测试记录；
7. 旧版本、讨论稿和历史归档。

若高优先级资料之间仍冲突，Codex 必须向用户提问，不得静默选择。

## 6. Skill 源码边界（已确认）

`DEC-001` 已于 2026-07-22 由用户确认：正式 Skill 的源码最终归入 `System_alpha/04_Skill系统/`；Codex 安装目录中的副本属于部署产物，不属于系统源码。

当前阶段规则：

- `System_alpha/04_Skill系统/` 保存正式需求、正式规则和晋升记录；
- `.codex-tmp/skills/` 保存可执行测试实现；
- Draft 生成 Skill 仍在设计和验证中，不迁移、不安装、不晋升；
- 只有通过 `DEV-GOV_迭代与防漂移规范.md` 中的晋升门禁后，才把确认版本迁入正式源码目录。

## 7. 全系统复杂度原则（已确认）

`DEC-MOD-001` 已于 2026-07-22 由用户确认：

> 后台模块化、知识最小单元化；前台保持短流程，由 Skill 内部按需调用，只有高风险、测试、恢复或故障诊断时才展开详细控制。

该原则适用于设定、资料调用、大纲、细纲、人物意图、正文、质量检查及未来模块。具体规范见 `04_Skill系统/00_Skill总控/SKL-ARCH_模块化与最小控制单元规范.md`。
