# 单元上下文包说明

## 一、功能定位

单元上下文包是叙事系统中的单元级资料压缩与确认文件。

它用于把人物卡、事件卡、设定卡、章节大纲等原始资料，整理为某一个固定单元内可复用的写作上下文。

单元上下文包不是原始资料库，也不是正文文件。它是作者确认后的中间资料层，用于帮助正文生成 Skill 节约 token，并降低 LLM 误读全量资料的风险。

---

## 二、为什么使用单元上下文包

相比全量读取 Markdown 文档，单元上下文包可以：

```text
减少 token 消耗
集中本单元真正需要的信息
明确人物、事件、设定在本单元中的使用边界
让作者提前确认 LLM 将使用的核心资料
避免正文生成时读取过多无关细节
```

相比分卷上下文包，单元上下文包更适合作为 V1.2 的主机制：

```text
单元范围更稳定
单元目标更明确
人物状态变化更少
事件使用方式更集中
信息释放边界更容易确认
```

分卷上下文摘要可作为后续版本的高层总览，但 V1.2 先以单元上下文包为核心。

---

## 三、归属关系

单元上下文包属于叙事系统。

建议保存位置：

```text
01_叙事系统/04_单元/
```

P0 样板验证建议保存位置：

```text
01_叙事系统/04_单元/P0_样板验证/
```

单元上下文包生成规则属于 Skill 系统中的资料调用部分：

```text
04_Skill系统/01_资料调用/01_单元上下文包/
```

---

## 四、状态规则

单元上下文包必须由作者确认后，才能作为正文生成 Skill 的默认调用资料。

建议状态：

```text
draft：初稿，尚未确认
review：待作者确认
confirmed：作者已确认，可供正文生成调用
archived：废弃或历史版本
```

正文生成 Skill 默认只调用 `status: confirmed` 的单元上下文包。

---

## 五、生成流程

推荐流程：

```text
1. 作者确定单元范围
2. 脚本或人工根据 YAML、rel、unit、line 等字段生成候选资料包
3. 高价值 LLM 根据候选资料包整理单元上下文包草案
4. 作者检查、修改、补充
5. 状态改为 confirmed
6. 正文生成 Skill 调用该单元上下文包
```

---

## 六、与章节生成的关系

正文生成时不应默认全文读取人物卡、事件卡和设定卡。

推荐调用关系：

```text
单元上下文包
        ↓
章节大纲卡
        ↓
章节细纲卡
        ↓
正文生成 Skill
        ↓
正文文件
```

如果某章需要超出单元上下文包的信息，可以通过章节大纲卡或章节细纲卡显式补充必要原文片段。
