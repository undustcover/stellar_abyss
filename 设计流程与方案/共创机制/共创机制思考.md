当前系统确实缺少一个关键层，而且不是小修小补，而是**正文生成链路中的核心交互层缺失**。

现在的系统虽然有“章节级意图理解”“人物意图轨道”“ASK_USER中断”，但这些机制主要解决的是：

> 模型已经开始生成以后，发现人物意图存在分歧，再向作者提问。

它仍然属于一种**风险纠偏机制**，不是正文创作前的**共同创作机制**。

当前链路本质上还是：

```text
作者提供大纲、细纲、人物资料
→ 系统自动理解
→ 自动生成执行包
→ 自动生成正文
→ 发现明显分歧时才中断
→ 作者事后纠偏
```

问题就在这里：

> 系统默认作者已经通过大纲、细纲把正文层面的创作意图表达清楚了。

但实际上没有。

大纲和细纲只能确定：

- 这一章发生什么；
    
- 谁参与；
    
- 冲突如何推进；
    
- 信息释放到哪里；
    
- 结尾落在哪里。
    

它们无法充分确定：

- 这一场真正要怎么写；
    
- 哪个人先开口；
    
- 哪一句话是场面的核心；
    
- 哪一段要快，哪一段要停；
    
- 哪个地方应该幽默，哪个地方必须压住；
    
- 作者希望读者在这一段具体感受到什么；
    
- 哪些地方可以自由发挥；
    
- 哪些地方作者心里其实已经有模糊画面；
    
- 哪些地方虽然大纲正确，但作者直觉上“不想这么写”。
    

这些都属于正文层面的创作决策。

而当前系统把它们自动推断掉了。

所以正文总会出现一种情况：

> 逻辑是对的，人物意图也可能是对的，但写出来不是作者心里那一场。

这正是当前系统反复对齐、反复返工、正文始终不够满意的根本原因。

现有设计强调动态对齐和关键分歧中断，但交互仍然集中在模型识别出“冲突”之后，没有把作者作为正文创作过程中的持续参与者。

# 一、当前缺少的不是“更多提问”，而是一个正式的正文共创阶段

不能简单理解为：

> 正文生成前多问几个问题。

如果只是多问：

- 这一场什么情绪？
    
- 要不要搞笑？
    
- 谁占主动？
    
- 节奏快还是慢？
    

模型仍然可能把作者答案抽象化，然后继续自动生成。

真正需要新增的是一个独立阶段：

# 正文生成前的“场景共创对齐”

它位于：

```text
章节细纲完成
→ 正文执行包生成
→ 场景共创对齐
→ 正文生成
```

或者更准确地说：

```text
章节细纲
→ 初步执行包
→ 作者与模型讨论正文落地
→ 修订执行包
→ 作者确认
→ 正文生成
```

这个阶段的目的不是再次讨论剧情，而是把抽象剧情转成**作者认可的正文演出方案**。

---

# 二、大纲、细纲、执行包、正文之间缺少了一层“演出设计”

当前系统大致有三层：

```text
大纲：发生什么
细纲：怎么推进
正文执行包：模型如何执行
正文：实际写出来
```

但实际上中间还缺一层：

```text
大纲：发生什么
细纲：场景如何构成
演出设计：这一场具体怎么呈现
执行包：模型按什么规则写
正文：正式落笔
```

这里的“演出设计”不是影视分镜，也不是把细纲继续写长。

它解决的是：

> 同样一个情节，可以有十种写法，作者到底希望采用哪一种。

比如：

```text
陈艾艾质问陈科
```

这只是细纲。

真正进入正文前，还要确定：

- 陈艾艾是直接爆发，还是压着火问；
    
- 她先问钱，还是先问手机；
    
- 陈科是强词夺理，还是先装镇定；
    
- 马木什么时候插话；
    
- 马雯雯在场内是旁观、误会，还是被迫参与；
    
- 场景的核心张力是父女冲突，还是骗局的不对劲；
    
- 喜剧来自鸡同鸭讲，还是陈科维护体面的荒谬；
    
- 哪一句话是这个场景最重要的落点；
    
- 场景结束时读者应该更好笑，还是更不安。
    

这些决定不应该全部交给系统自动推断。

因为这部分已经不是“信息结构”，而是“创作判断”。

---

# 三、作者应当在哪些节点介入

不建议作者每一句都介入，也不建议完全自动化。

最合理的方式是建立三类介入节点。

## 第一类：章节正文启动前介入

这是强制介入。

系统在完成章节资料读取和初步执行包后，不应立刻生成正文，而应先和作者进行一次章节级正文讨论。

讨论内容不再是大纲是否正确，而是：

```text
这一章准备怎么写
哪些场景是重点
哪些场景可以略写
正文的主要气质
本章最重要的一场
本章最重要的一句或一个动作
读者情绪如何变化
哪些地方作者已有明确画面
哪些地方可以交给模型发挥
```

这个阶段解决的是整章正文方向。

输出不是正文，而是：

```text
章节正文创作约定
```

例如：

```yaml
chapter_prose_contract:
  core_reading_experience:
  dominant_tone:
  main_scene:
  secondary_scene:
  detail_distribution:
  humor_intensity:
  emotional_pressure:
  narrator_distance:
  author_fixed_images:
  model_free_space:
  forbidden_execution:
```

这相当于作者和模型对这一章签一份“正文创作合同”。

---

## 第二类：重点场景生成前介入

不是每个场景都强制。

重点场景必须进入讨论，普通过渡场景可以自动生成。

重点场景包括：

- 章节核心冲突；
    
- 重要人物首次立住；
    
- 重大误会；
    
- 强笑点；
    
- 人物关系变化；
    
- 关键信息释放；
    
- 章节高潮；
    
- 结尾钩子；
    
- 作者特别重视的场景。
    

进入重点场景前，系统不应只输出人物意图轨道，而应先给出一个很短的“场景演出方案”。

例如：

```text
本场建议这样写：

1. 开场由陈科先抢回体面，不由陈艾艾直接质问。
2. 陈艾艾先问手机，不先问钱。
3. 马木前半段少说，听到“手机离手”后才介入。
4. 喜剧来自陈科不断替对方解释，不使用明显段子。
5. 场景后半段转冷，工作人员出现后压住笑感。
6. 本场不直接下诈骗结论。
```

然后向作者确认：

- 是否同意；
    
- 哪一点要改；
    
- 是否有作者自己的画面；
    
- 是否要先看一小段示例。
    

这就是正文生成前真正的互动。

---

## 第三类：正文片段生成后介入

这是作者审美介入，而不是只检查错误。

每个重点场景生成后，不应自动连续生成下一场。

系统应该暂停，提供：

```text
当前场景正文
+
极短自检
+
需要作者判断的1—3个点
```

例如：

```text
请重点判断：

1. 陈科的语言是否像本人；
2. 陈艾艾的压迫感是否过重；
3. 这场喜剧和不安的比例是否正确。
```

作者可以：

- 确认继续；
    
- 指出某个方向不对；
    
- 直接修改一句；
    
- 要求重写整个场景；
    
- 调整后续场景规则。
    

这样作者不是等整章生成完才审阅，而是在正文形成过程中逐步塑形。

---

# 四、当前ASK_USER机制为什么不够

现有ASK_USER逻辑是：

> 只有模型发现关键分歧时才中断。

这个机制有两个局限。

## 1. 模型不知道自己理解错了

很多正文问题不是模型感知到“有两个合理方向”，而是模型自信地选择了一个方向。

例如：

- 模型认为这一段应该持续盘问；
    
- 模型认为需要补一个漂亮总结；
    
- 模型认为陈灏登场可以增强趣味；
    
- 模型认为工作人员应当表现得滴水不漏。
    

这些在模型内部可能没有分歧。

于是它会直接CONTINUE。

但作者看完以后会说：

> 不对，这场根本不该这么写。

所以仅靠“分歧触发”无法替代“作者参与”。

## 2. ASK_USER偏向纠错，不偏向创作

它的问题通常是：

```text
A方向还是B方向？
人物为什么说这句话？
这个行为是否成立？
```

这些问题是在确定正确性。

但正文质量还依赖大量无法用对错判断的选择：

- 哪一种更有味道；
    
- 哪一种更像作者；
    
- 哪一种更有趣；
    
- 哪一种更克制；
    
- 哪一种读起来更顺；
    
- 哪一种更符合本章气质。
    

这些属于审美决策。

审美决策不能完全等到模型发现冲突以后才交给作者。

---

# 五、正文执行包不应自动成为最终输入

当前流程里，正文执行包容易变成系统自动生成后，直接喂给正文生成Skill。

这一步需要改变。

正文执行包应分成两版：

```text
初始执行包
→ 作者讨论
→ 确认执行包
```

初始执行包只能是系统建议。

它应明确标注：

```text
系统推断内容
作者已确认内容
待讨论内容
允许自由发挥内容
```

例如：

```yaml
scene_execution:
  confirmed:
    - 陈科核心目标是维护自尊
    - 马雯雯不能识破诈骗
  inferred:
    - 场景先轻后冷
    - 马木在手机问题后介入
  pending:
    - 陈艾艾是否当场爆发
    - 陈灏是否在本场登场
  free:
    - 便利店环境细节
    - 普通过渡动作
```

这样作者能一眼看到：

> 哪些是系统自己猜的。

否则执行包写得越完整，越容易制造一种假象：

> 好像所有问题都已经解决了。

实际上只是模型替作者做了决定。

---

# 六、建议新增一个正式状态：PROSE_WORKSHOP

当前状态机里有：

```text
CHAPTER_PLAN_ALIGNED
RUNTIME_STATE_READY
V1_IN_PROGRESS
INTENT_CHECKPOINT
FRAGMENT_WRITING
V1_ASSEMBLED
```

建议增加：

```text
PROSE_WORKSHOP
```

完整状态变成：

```text
CHAPTER_CONTEXT_READY
    ↓
CHAPTER_PLAN_ALIGNED
    ↓
EXECUTION_PACK_DRAFTED
    ↓
PROSE_WORKSHOP
    ↓
EXECUTION_PACK_CONFIRMED
    ↓
RUNTIME_STATE_READY
    ↓
V1_IN_PROGRESS
```

重点场景中还可以增加：

```text
SCENE_WORKSHOP
    ↓
SCENE_EXECUTION_CONFIRMED
    ↓
FRAGMENT_WRITING
    ↓
SCENE_REVIEW
```

也就是说，正文生成不再是：

```text
准备完成
→ 自动执行
```

而是：

```text
准备完成
→ 作者参与设计
→ 确认执行方式
→ 生成
→ 作者判断
→ 继续
```

# 七、正文共创讨论具体讨论什么

必须避免讨论无限扩张。

每次讨论只围绕正文落地，而不是重新讨论世界观和大纲。

建议固定六项。

## 1. 场景中心

```text
这一场最重要的是什么？
```

只能有一个主要答案。

比如：

- 陈科维护体面；
    
- 马雯雯继续误解；
    
- 陈艾艾发现父亲已被深度影响。
    

不能每一项都同等重要。

## 2. 场景主动权

```text
谁在推动这场？
谁在抵抗？
谁在旁观？
谁在暗中改变场面？
```

## 3. 人物语言策略

```text
谁直接说？
谁绕着说？
谁说多了？
谁不说？
谁故意不回答？
```

## 4. 场景情绪轨迹

```text
轻松
→ 尴尬
→ 对抗
→ 短暂缓和
→ 不安
```

不要只写“轻喜剧”。

## 5. 重点与留白

```text
哪段详写？
哪段略写？
什么必须让读者自己发现？
什么不能解释？
```

## 6. 作者审美判断

```text
作者最希望看到什么？
最不希望出现什么？
有没有参考句、参考画面或参考节奏？
```

这一项最重要。

因为很多作者意图并不是抽象规则，而是：

> 我脑子里大概是这种感觉。

系统必须允许作者用模糊、感性、示例化的方式参与，而不是强迫作者把一切翻译成结构化字段。

---

# 八、交互应该是一轮还是多轮

两种都要支持。

## 一轮交互

适合：

- 过渡场景；
    
- 作者意图明确；
    
- 已经有成熟人物规则；
    
- 场景风险较低。
    

流程：

```text
系统给出演出建议
→ 作者一次确认或修改
→ 生成
```

## 多轮交互

适合：

- 黄金章节；
    
- 重要人物首次出场；
    
- 强喜剧场景；
    
- 复杂误会；
    
- 人物关系关键变化；
    
- 作者自己也没想清楚；
    
- 历史上反复重写的场景。
    

流程：

```text
系统提出初步演出方案
→ 作者纠偏
→ 系统给出最小示例
→ 作者判断
→ 系统更新方案
→ 必要时再给第二个局部示例
→ 作者确认
→ 正式生成
```

这里最关键的是：

> 最小示例不是正文成品，而是用来验证双方理解是否一致。

比如只生成100—300字：

- 场景开头；
    
- 一轮核心对话；
    
- 强笑点落点；
    
- 情绪转折点。
    

作者确认“就是这个味道”以后，再生成完整场景。

这比生成整章后再说“不对”有效得多。

---

# 九、作者介入不应只通过回答问题，还应允许直接改写

系统当前的中断协议已经允许作者直接改写人物意图轨道，这是正确的。

正文共创层还需要进一步允许作者直接操作四类内容：

```text
1. 直接改场景目标
2. 直接改人物语言策略
3. 直接给出示例台词
4. 直接指定节奏与重点
```

系统收到作者示例后，不应只复用原句，而应提炼其背后的规则。

例如作者写：

> 陈科这里不是解释项目，而是抓着“你们不尊重老人”反复说。

系统应提炼为：

```text
本场陈科不参与事实辩论。
他将所有质疑重新定义为子女对老人的不尊重。
语言表现为重复、回避和情绪性反击。
```

然后用这条规则生成其余对白。

这样作者只需要提供一个抓手，不需要亲自写完整正文。

---

# 十、自动化系统的定位必须调整

目前系统隐含的目标是：

> 尽量自动生成高质量正文，作者只在发生分歧时介入。

这个定位对大纲、资料整理、执行包生成是合理的，但对小说正文不完全合理。

正文阶段更适合的定位是：

> 系统负责组织、推演、生成和检查；作者负责关键审美决策和重点场景定向。

所以正文模块不应该追求：

```text
全自动生成整章
```

而应追求：

```text
半自动共创整章
```

自动化部分负责：

- 读取资料；
    
- 维护人物状态；
    
- 防止信息越界；
    
- 检查人物意图；
    
- 记录作者确认；
    
- 保证连续性；
    
- 生成候选文本；
    
- 检查重复和AI模式；
    
- 整章组装。
    

作者负责：

- 选择场景写法；
    
- 确定重点；
    
- 判断人物是否有味道；
    
- 确定喜剧尺度；
    
- 决定哪些地方留白；
    
- 判断文本是否符合自己的审美。
    

这不是系统失败。

相反，这是对小说创作本质的正确划分。

# 十一、推荐的新完整链路

```text
一、章节资料加载
    ↓
二、章节级剧情与人物意图理解
    ↓
三、生成初始正文执行包
    ↓
四、章节正文共创讨论
    - 本章阅读体验
    - 重点场景
    - 详略
    - 情绪曲线
    - 作者已有画面
    - 禁止方向
    ↓
五、更新并确认正文执行包
    ↓
六、建立章节运行态
    ↓
七、进入场景
    ↓
八、判断场景等级
    ├─ 普通场景：自动生成
    └─ 重点场景：进入场景共创
                     ↓
九、场景共创讨论
    - 场景中心
    - 主动权
    - 人物语言策略
    - 节奏
    - 笑点
    - 留白
    ↓
十、必要时生成最小验证片段
    ↓
十一、作者确认或纠偏
    ↓
十二、更新场景执行规则
    ↓
十三、人物意图轨道检查
    ↓
十四、自然度预检
    ↓
十五、正文生成
    ↓
十六、重点场景作者审阅
    ├─ 确认：继续
    ├─ 局部修订
    └─ 重回场景共创
    ↓
十七、整章组装
    ↓
十八、整章一致性与AI模式检查
    ↓
十九、输出V1
    ↓
二十、作者整章审阅
    ↓
二十一、V2精修
```

# 十二、最关键的制度变化

需要把当前系统的一条默认规则：

```text
没有发现分歧，就自动继续。
```

改成两条：

```text
普通场景：
没有发现分歧，可以自动继续。

重点场景：
即使模型没有发现分歧，也必须经过作者正文演出确认。
```

这是最核心的变化。

因为重点场景的问题往往不是“有没有逻辑分歧”，而是：

> 这一场是不是作者想要的那种写法。

# 最终结论

当前系统确实欠缺正文创作前和正文生成过程中的作者交互。

缺少的不是简单的“提问功能”，而是一个独立的：

> **正文共创与场景演出对齐层。**

大纲和细纲负责确定故事。

人物意图控制负责保证人物不失真。

去AI化控制负责保证语言不机械。

而正文共创层负责解决最关键的问题：

> **这一场到底要怎么写，才是作者真正想看的正文。**

正文模块因此不应继续以全自动化为目标，而应转为：

> **重点场景强交互、普通场景弱交互、整章连续交付的半自动创作系统。**
