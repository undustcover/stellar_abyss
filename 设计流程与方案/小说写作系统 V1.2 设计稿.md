## 一、V1.2 版本定位

V1.2 是小说写作系统从“架构规范”进入“最小可运行写作系统”的版本。

V1.1 已经完成：

```text
文件夹结构
文件命名规则
YAML 字段原则
双链关系原则
四类核心模板的架构级规范
```

V1.2 的目标是把这些原则落到可执行模板、资料调用机制和验证流程上，使作者能够围绕 P0 单元创建资料、组织章节、提取上下文、生成正文，并通过一次样板流程验证系统是否真的可用。

V1.2 不追求覆盖全书所有模板，也不追求自动化工具开发。它关注的是：

```text
模板是否可填写
资料是否可检索
资料是否可提取和摘要
上下文包是否可形成
章节是否可组织
Skill 是否可调用
正文是否可生成
问题是否可回写
```

```text
1. 作者可以按模板创建人物卡、事件卡、章节大纲卡、章节细纲卡、正文文件、正文生成 Skill。

```

## 三、资料调用与单元上下文包机制

### 1. 设计原因

随着系统扩展，人物卡、事件卡、设定卡、章节文件和 Skill 文件都会迅速增多。

如果 LLM 在每次生成大纲、细纲或正文时都全文读取 Markdown 文档，会造成大量 token 消耗，也会增加误读和上下文污染风险。

因此，V1.2 必须明确：

```text
LLM 不默认全文读取资料库。
任何生成任务开始前，必须先经过资料筛选、提取和摘要环节。
正文生成 Skill 默认读取已确认的单元上下文包，而不是直接读取全量资料。
```

### 2. 系统归属

资料调用机制归属于 Skill 系统中已有的资料调用文件夹：

```text
04_Skill系统/01_资料调用/
```

已建立需求文档：

```text
SKL-CALL-REQ_资料调用需求文档.md
SKL-CALL-UNIT-REQ_单元上下文包生成需求.md
```

资料调用 Skill 负责定义“如何调用资料”，不负责生成正文。

单元上下文包是资料调用后的作者确认型产物，建议保存在叙事系统中，服务同一单元内的大纲、细纲和正文生成。分卷上下文包暂不作为 V1.2 主机制，只作为后续扩展。

### 3. 分层模型

资料调用采用四层模型：

```text
原始资料层
    人物卡、事件卡、设定卡、章节大纲、章节细纲、已有正文等

候选资料层
    根据 YAML、rel、tags、unit、chapter、line 等字段筛选出来的相关资料

摘要资料层
    对候选资料进行压缩，保留当前任务必须知道的信息

单元上下文包
    面向某一固定单元的稳定输入资料包
```

### 4. 脚本与 LLM 的分工

V1.2 不引入低价值 LLM。

脚本可以负责确定性工作：

```text
读取 YAML 字段
按 system、type、unit、chapter、line、role、event_type 等字段筛选文件
读取 rel 中的关联对象
抽取指定 Markdown 标题下的内容
拼接候选资料包
统计长度
检查模板缺字段
```

高价值 LLM 可以负责判断和压缩：

```text
判断哪些资料对当前任务真正重要
压缩人物、事件和设定信息
保留不可丢失的逻辑限制
识别信息释放边界
生成完整、简洁、节约 token 的单元上下文包
```

### 5. 单元上下文包原则

单元上下文包应回答：

```text
本单元要写什么
本单元必须使用哪些人物
本单元必须使用哪些事件
本单元必须遵守哪些设定
本单元哪些信息可以释放
本单元哪些信息必须隐藏
本单元有哪些写作禁忌
```

单元上下文包不是最终正文，也不是设定卡替代品。它是正文生成 Skill 的主要输入之一，必须经过作者确认后才能默认调用。

## 五、模板通用规则

所有模板遵守以下规则：

```text
YAML 只做索引和筛选
复杂内容写入 Markdown 正文
rel 只写 ID，不写 Obsidian 双链
双链只写在正文中，用于解释语义关系
字段名保持稳定
正式模板与示例内容分离
正文生成任务默认读取已确认的单元上下文包，不默认全文读取资料库
```

所有模板默认包含：

```yaml
id:
type:
system:
status:
title:
rel: []
tags: []
```

可按模板需要增加：

```yaml
alias:
line:
role:
world_time:
event_type:
unit:
chapter:
pov:
task:
```

## 六、人物卡正式模板

### 定位

人物卡是设定系统中的完整档案卡。

它记录人物基础档案、身份关系、经历摘要、当前状态和关键设定。阶段变化明细不写在人物卡中，而写入人物经历线。

人物卡可以详细，但后续生成任务不默认全文读取人物卡，而是通过资料调用机制提取人物摘要、当前状态、已知信息范围和写作禁忌。

### YAML

```yaml
id: CHR-0000
type: character
system: setting
status: draft
title:
alias:
line:
role:
rel: []
tags: []
```

### 正文结构

```markdown
## LLM调用摘要

## 核心定位

## 基础信息

## 外貌与识别特征

## 身份与社会关系

## 人物性格

## 说话方式

## 行为习惯

## 能力与资源

## 经历摘要

## 当前状态

## 已知信息范围

## 未知信息范围

## 误解信息

## 主要关系

## 关联事件

## 关联设定

## 人物经历线

## 写作禁忌
```

### 验证重点

```text
是否能让作者快速理解人物
是否能让 LLM 把握人物说话和行为
是否避免与人物经历线重复
是否能通过摘要区节约 token
是否能支持章节大纲和正文生成
```

## 七、事件卡正式模板

### 定位

事件卡是事件数据库中的完整事实卡。

它记录小说世界中真实发生的故事级大事件，是世界事实权威来源。事件卡不记录章节使用方式，章节如何使用事件由章节大纲卡负责。

事件卡可以详细记录事实，但生成任务不默认全文读取事件卡，而是通过资料调用机制提取事件摘要、世界时间、时间线、结果、影响和逻辑限制。

### YAML

type: event
system: event
status: draft
title:
world_time:
event_type:
line:
rel: []
tags: []

## 世界时间

## 事件性质

## 事件背景

## 起因

## 时间线

### 节点一

### 节点二

### 节点三

## 结果

## 后续影响

## 涉及人物

## 涉及组织

## 涉及地点

## 涉及设定

## 逻辑限制

### 定位

章节细纲卡是叙事系统中的场景级执行卡。

它承接章节大纲，把章级设计拆成具体场景、场景顺序、行动推进、对白重点和节奏安排。

### YAML

```yaml
id: OUT-P00-C00
type: detail_outline
system: narrative
status: draft
title:
unit: P00
chapter: C00
pov:
rel: []
tags: []
```

### 正文结构

```markdown
## 调用依据

## 本章执行目标

## 单章注意力结构

用于标明本章正文内部的阅读重点、详略关系和波峰安排。

它不是章节大纲结构，也不是剧情顺序替代品，而是告诉正文生成环节：

```text
哪些地方需要停下来写；
哪些地方只负责过渡；
哪些地方必须形成阅读波峰；
哪些地方不能平均用力；
哪些地方需要留白、省略或快速带过。
```

建议至少说明：

|项目|说明|
|---|---|
|本章阅读波峰|本章最需要详写、最影响读者体验的核心场面。|
|重点详写段|需要展开动作、对话、情绪或流程压力的段落。|
|中等描写段|需要建立场景或推进关系，但不能抢主戏的段落。|
|省略/压缩段|只承担转场、信息衔接或轻收束的段落。|
|不可平均用力提醒|提醒正文生成时避免所有场景同等篇幅、同等密度。|

## 场景列表

## 场景一

### 场景功能

### 出场人物

### 地点与时间

### 行动推进

### 对话重点

### 信息释放

### 情绪变化

### 场景出口

## 场景二

## 场景三

## 节奏安排

## 必须写出的内容

## 禁止写出的内容

## 与章节大纲的对应关系
```

### 验证重点

```text
是否能承接章节大纲
是否能支撑正文生成
是否避免写成正文
是否保留足够创作空间
是否能进入单元上下文包
是否明确单章注意力结构，避免正文平均用力、平铺直叙
```

## 十、正文文件模板

### 定位

正文文件是叙事系统中的正式文本承载文件。

它存放章节正文，不存放复杂设定，不替代人物卡、事件卡和 Skill。

### YAML

```yaml
id: DRAFT-P00-C00
type: draft
system: narrative
status: draft
title:
unit: P00
chapter: C00
pov:
rel: []
tags: []
```

### 正文结构

```markdown
## 调用依据

## 生成说明

## 正文

## 修订记录

## 待检查问题
```

### 验证重点

```text
正文是否能追溯到章节大纲和细纲
正文是否独立承载最终文本
修订问题是否能回写到修订记录或 Skill
正文生成时是否基于单元上下文包，而不是全量资料
```

## 十一、正文生成 Skill 正式模板

### 定位

正文生成 Skill 是 Skill 系统中的执行流程与写作规则文件。

它规定 LLM 如何读取单元上下文包、组织正文、生成文本并做基础自检。它不是 Prompt，Prompt 在后续版本中再设计。

正文生成 Skill 不负责全量资料检索，也不负责资料摘要生成。资料检索和摘要由资料调用 Skill 负责。

### YAML

```yaml
id: SKL-WR-000
type: skill
system: skill
status: draft
title:
task: write_draft
rel: []
tags: []
```

### 正文结构

```markdown
## 适用任务

## 默认输入资料

## 资料调用前置要求

## 单元上下文包读取规则

## 正文生成规则

## 人物表现规则

## 对话规则

## 节奏规则

## 信息释放规则

## 写作层禁用规则

## 输出格式

## 生成后基础自检

## 关联 Skill
```

### 默认输入资料

```text
单元上下文包
章节大纲卡
章节细纲卡
必要原文片段
```

必要人物、事件和设定资料应优先通过单元上下文包进入正文生成流程。

### 输出格式

```markdown
## 调用摘要

## 正文

## 生成后自检
```

### 基础自检

```text
人物是否崩坏
事件事实是否冲突
是否提前暴露信息
是否出现 AI 腔、总结腔、解释腔、概念堆砌
```

## 十四、P0 样板验证流程

V1.2 应选择 P0 中的一个样板章节进行验证。

建议流程：

```text
[部分完成] 1. 创建 2-3 张人物卡
   已完成：马木人物卡样板
   草稿完成：徐乂、林薇、陈灏、苏芳菲外部世界书人物卡
   待办：如需进入 System_alpha，需要转化为本系统人物卡或获得外部目录写入权限

[已完成] 2. 创建 1-2 张事件卡
   已完成：EVT-0001~EVT-0007 事件卡样板
   注意：事件卡已按故事级颗粒度返工校正

[待办] 3. 创建 1 张章节大纲卡

[待办] 4. 创建 1 张章节细纲卡

[已完成] 5. 创建 1 份资料调用需求文档或资料调用规则
   已完成：SKL-CALL-REQ_资料调用需求文档.md
   已完成：SKL-CALL-UNIT-REQ_单元上下文包生成需求.md

[待办] 6. 根据资料调用规则生成候选资料包

[待办] 7. 根据候选资料包生成 1 个单元上下文包草案

[待办] 8. 作者确认并修订单元上下文包

[待办] 9. 创建 1 个正文生成 Skill

[待办] 10. 创建 1 个正文文件

[待办] 11. 使用正文生成 Skill 读取 confirmed 单元上下文包生成正文

[待办] 12. 检查正文是否符合人物、事件、信息释放和节奏要求

[待办] 13. 记录模板缺口和资料调用缺口

[待办] 14. 决定哪些问题进入 V1.2 修订，哪些进入 V1.3
```

P0 样板验证不要求完成整个 P0 单元，只要求跑通一个章节闭环。

验证链路：

```text
人物卡 + 事件卡 + 设定资料
        ↓
资料调用 Skill / 资料调用规则
        ↓
候选资料包
        ↓
单元上下文包
        ↓
章节大纲卡
        ↓
章节细纲卡
        ↓
正文生成 Skill
        ↓
正文文件
        ↓
skills 反哺记录
        ↓
问题回写
```

### 过程性 skills 反哺记录

P0 样板验证期间，正文测试不得只在最终稿完成后一次性修改 skills。

每一轮正文评估、作者修订对比和问题总结之后，必须同步维护 `skills反哺记录.md`，记录：

```text
1. 本轮正文版本暴露出的具体问题；
2. 正文层面如何修正；
3. 该问题可能暴露出的 skill、模板或资料字段缺口；
4. 当前是否暂不修改 skill，以及原因；
5. 后续确认该问题是否需要正式回写的条件；
6. 如果确认回写，可能修改哪个 skill 或模板。
```

`skills反哺记录.md` 不直接等同于 skill 修改清单。

它是正文修订与系统回写之间的缓冲层，用于保留渐进修正过程中的深度思考，避免最终只对成稿结果负责，忽略中间反复出现的流程问题。

正式修改 skill 前，必须先回看反哺记录，并确认问题是否具备通用性、是否反复出现、是否能归因到明确流程层级，以及是否能用更小改动解决。

## 十五、暂不纳入本阶段的内容

V1.2 暂不纳入：

```text
完整 Dataview 查询语法
自动化脚本或 CLI 工具
Prompt 正式模板
正文修订 Skill
一致性校验 Skill
世界观卡完整模板
能力卡完整模板
组织卡完整模板
地点卡完整模板
全书级 Database 仪表盘
多章节连续生成流程
完整 P0 单元验证
分卷上下文包正式模板
章节调用摘要正式模板
向量数据库或 embedding 检索
低价值 LLM 摘要流程
```

这些内容可进入 V1.3 或后续版本。

```text
