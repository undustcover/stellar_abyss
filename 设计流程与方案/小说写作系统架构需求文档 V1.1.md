## 一、系统目标

本系统用于搭建一套面向长篇小说创作的结构化写作系统。

系统以 Obsidian 作为主要资料承载平台，通过双链、标签、属性、看板、文件夹结构等方式，对小说创作中的叙事内容、设定内容、事件内容和 Skill 规则进行模块化管理。

系统最终目标是：在 Skill 系统的控制下，调用设定系统与事件数据库中的资料，支撑叙事系统完成从总纲、卷纲、单元纲、章纲、细纲到正文的生产。

---

## 二、系统总体架构

小说写作系统在 Obsidian 中采用五个一级文件夹：

```text
00_总控台
01_叙事系统
02_设定系统
03_事件数据库
04_Skill系统
```

其中，四大核心系统为：

```text
01_叙事系统
02_设定系统
03_事件数据库
04_Skill系统
```

四大系统的核心定位如下：

|系统|核心定位|主要作用|
|---|---|---|
|叙事系统|正文生产线|面向读者，负责大纲、章纲、细纲、正文等内容生产|
|设定系统|世界规则库|记录人物、世界观、组织、道具、能力、种族、地区等硬规定|
|事件数据库|世界时间轴事实库|记录小说世界中真实发生的大事件、历史阶段、人物经历和组织行动|
|Skill系统|LLM创作控制系统|定义 LLM 如何调用资料、组合资料、校验资料和生成高质量文本|

总控台的核心定位如下：

|系统|核心定位|主要作用|
|---|---|---|
|总控台|系统入口与管理页|集中管理系统总览、当前进度、待处理问题、Database视图、模板和归档索引|

四大系统的逻辑关系为：

```text
设定系统 + 事件数据库
        ↓
Skill系统调用、加工、校验
        ↓
叙事系统生成大纲、章纲、细纲、正文
```

## 三、核心设计原则

### 1. 模块化原则

四大系统相对独立，各自承担不同职责。

设定系统负责规则，事件数据库负责事实，叙事系统负责讲法，Skill系统负责调用与生成。

任一模块可以单独修改、扩展和维护，但不得混淆职责。

---

### 2. 世界时间轴原则

系统底层以世界时间轴为基础。

事件数据库、人物经历、组织行动、历史事件等内容，必须使用小说世界内部时间进行定义。

例如：

```text
2026年5月—2026年6月
十几年前
灾变后第3年
星历XXX年
六合集市开放前7天
```

事件数据库不得使用 P0、C1、B1 等叙事编号作为主要时间定义。

---

### 3. 叙事结构编号原则

叙事系统可以使用 B / V / P / C 等小说结构编号。

例如：

```text
B0：前置部
V1：第一卷
P0：命运的馈赠
C01：我那一事无成的堂哥
C02：女儿先慌了
```

B：定义小说第几部  
V：定义小说第几卷  
P：定义小说第几篇 / 单元  
C：定义小说第几章

B / V / P / C 等编号只用于定义小说的部、卷、单元、章节和正文结构，不用于定义世界事实本身。

---

### 4. 双时间轴原则

系统中必须区分两种时间轴：

```text
世界时间轴：事情在小说世界中真实发生的顺序
叙事结构轴：读者在小说中看到这些事情的顺序
```

世界时间轴服务于逻辑一致性。

叙事结构轴服务于读者体验。

二者可以不一致。倒叙、插叙、伏笔、误导、解密、反转、回收等写作手法，均依赖二者的分离。

---

### 5. 大事件颗粒度原则

事件数据库只记录故事级大事件，不拆分章节，不设计场景，不安排爽点，不承担章纲和细纲职责。

例如，事件数据库可以记录：

```text
事件：《陈科被高端分享局诈骗事件》
时间：2026年5月—2026年6月
```

但不应细分为：

```text
陈科接电话
陈科去会所
马木赶到会场
陈科填写登记表
任战设置门槛
陈科反咬马木
```

上述内容属于叙事系统中的大纲、章纲或细纲设计，不属于事件数据库的基础记录范围。

---

### 6. Skill服务LLM原则

Skill系统不是资料库，而是面向 LLM 的创作控制系统。

Skill系统负责定义：

```text
LLM应该调用哪些资料
LLM应该如何理解资料
LLM应该如何组合资料
LLM应该如何校验资料
LLM应该如何生成大纲、章纲、细纲和正文
LLM应该如何保证正文质量、风格和去AI化
```

---

### 7. Obsidian索引简洁原则

Obsidian 中的 YAML 只作为简洁索引使用。

YAML 负责帮助 Skill、Database、脚本和 LLM 精确定位文件，不承载复杂内容。

复杂内容应写入 Markdown 正文，包括：

```text
人物性格详细说明
人物说话方式详细说明
事件完整过程
能力机制详细解释
章节节奏设计
正文写作规则
信息释放细节
复杂因果链
人物关系解释
```

---

## 四、叙事系统

### 1. 系统定位

叙事系统是正文生产线。

它面向读者体验，负责在 Skill 系统控制下，将设定系统和事件数据库提供的资料，加工为小说的总纲、卷纲、单元纲、章纲、细纲、正文和修订稿。

叙事系统使用 B / V / P / C 等小说结构编号。

---

### 2. 核心职责

叙事系统负责：

```text
总纲设计
分部设计
分卷设计
单元设计
章节设计
场景设计
正文生成
正文修订
信息释放控制
伏笔埋设与回收
悬念设计
反转设计
情绪曲线设计
读者爽点设计
章节节奏控制
POV安排
```

---

### 3. 内容层级

叙事系统建议包含以下层级：

```text
叙事系统
├── 全书总纲
├── 分部大纲
├── 分卷大纲
├── 单元大纲
├── 章节大纲
├── 章节细纲
├── 正文
├── 修订稿
└── 废稿/版本记录
```

---

### 4. 工作重点

叙事系统关注的是：

```text
读者先看到什么，后看到什么
哪些事实提前展示
哪些事实暂时隐藏
哪些内容用于制造误导
哪些内容用于埋伏笔
哪些内容用于后续解密
每章提供什么读者奖励
每章如何形成推进感
每章如何结尾留钩子
```

叙事系统不直接承担世界事实记录职责。

---

## 五、设定系统

### 1. 系统定位

设定系统是世界规则库。

它负责记录小说中的人物、世界观、组织、道具、能力、种族、地区、术语等硬规定，为事件数据库和叙事系统提供规则约束。

---

### 2. 核心职责

设定系统负责：

```text
定义人物设定
定义世界观规则
定义能力边界
定义组织结构
定义道具规则
定义种族文明
定义地区场所
定义术语体系
定义信息暴露限制
定义禁用设定
定义设定冲突校验规则
```

---

### 3. 建议分类

设定系统建议包含：

```text
设定系统
├── 世界观
├── 人物
├── 组织/派系
├── 地区/场所
├── 道具/技术
├── 种族/文明
├── 能力系统
├── 术语表
├── 信息暴露规则
└── 禁用设定/冲突记录
```

---

### 4. 工作重点

设定系统关注的是：

```text
人物是否崩坏
能力是否越界
世界规则是否冲突
组织逻辑是否成立
术语是否统一
某个设定是否可以在当前阶段暴露
某个角色是否应该知道某条信息
```

设定系统不负责章节拆分，也不负责正文节奏设计。

---

## 六、事件数据库

### 1. 系统定位

事件数据库是基于世界时间轴的故事事实库。

它记录小说世界中真实发生的大事件、历史阶段、人物经历和组织行动。

事件数据库使用世界内时间定义，不使用 B / V / P / C 等叙事编号作为核心时间标识。

---

### 2. 核心职责

事件数据库负责：

```text
记录世界大事件
记录历史阶段
记录人物经历
记录组织行动
记录事件时间跨度
记录事件起因
记录事件结果
记录事件影响
记录事件涉及的人物、组织、地点和设定
维护世界时间轴
维护人物状态逻辑
维护组织状态逻辑
维护因果关系
防止故事事实冲突
```

---

### 3. 事件颗粒度

事件数据库采用故事级大事件颗粒度。

事件数据库记录的是：

```text
一段完整剧情事件
一段人物经历
一次组织行动
一次历史转折
一个世界阶段
```

不记录：

```text
章节节点
场景节点
对话安排
爽点设计
反转设计
悬念设计
章节结尾钩子
```

---

### 4. 示例事件颗粒度

正确颗粒度示例：

```text
事件：《马木当年合伙人捐款跑路事件》
时间：十几年前
性质：主角前史事件
影响：导致马木从人生巅峰跌落谷底，形成对“天降机会”的深层不信任。
```

```text
事件：《陈科被高端分享局诈骗事件》
时间：2026年5月—2026年6月
性质：现实诈骗事件 / 主角低谷事件
影响：陈科被诱导入局，马木介入失败，最终被反咬，触发断卡与开除。
```

```text
事件：《张凡与小满分离事件》
时间：2026年6月
性质：现实拯救线事件
影响：张富贵为孩子筹钱卖猫，张凡情绪崩落，形成马木“救不了所有人”的现实挫败。
```

---

### 5. 事件数据库字段建议

事件数据库中的大事件卡采用简洁 YAML 字段。

示例：

```yaml
id: EVT-0002
type: event
system: event
status: confirmed
title: 陈科被高端分享局诈骗事件
world_time: 2026-05~2026-06
event_type: 现实诈骗事件
line: 现实线
rel: [CHR-0001, CHR-0003, CHR-0005, ORG-0001, POW-0002]
tags: [现实线, 诈骗线, P00]
```

事件摘要、世界时间、事件性质、时间线、结果、影响、逻辑限制、关联说明等内容写入 Markdown 正文。

事件卡只记录世界事实，不记录事件被哪些章节使用。事件在叙事系统中的展示、隐藏、误导和回收方式，应写入章节大纲卡。

正文建议结构如下：

```text
事件摘要
世界时间
事件性质
事件背景
起因
时间线
结果
后续影响
涉及人物
涉及组织
涉及地点
涉及设定
逻辑限制
```

## 七、Skill系统

### 1. 系统定位

Skill系统是面向 LLM 的创作控制系统。

它负责定义 LLM 如何调用、筛选、组合、加工和校验资料，以及如何生成高质量文本。

Skill系统不是资料库，不存放具体小说内容，而是存放 LLM 使用资料和生成内容的方法。

---

### 2. 核心职责

Skill系统负责：

```text
定义资料调用规则
定义资料优先级
定义资料冲突处理规则
定义大纲生成流程
定义章纲生成流程
定义细纲生成流程
定义正文生成流程
定义正文修订流程
定义设定校验规则
定义事件校验规则
定义人设校验规则
定义信息释放规则
定义文风规则
定义去AI化规则
定义章节节奏规则
定义禁用写法
```

---

### 3. Skill系统内部分类

Skill系统分为两大类：

```text
Skill系统
├── 资料调用Skill
└── 写作生成Skill
```

---

## 八、资料调用Skill

### 1. 系统定位

资料调用Skill负责告诉 LLM 在执行某项写作任务时，应该读取哪些资料、如何筛选资料、如何判断资料优先级、如何处理资料冲突。

---

### 2. 核心职责

资料调用Skill负责：

```text
调用设定系统
调用事件数据库
调用叙事系统已有内容
调用上一章正文摘要
调用相关人物卡
调用相关事件卡
调用相关世界观设定
调用相关术语表
调用伏笔与信息释放记录
判断资料优先级
处理资料冲突
限定资料使用范围
防止提前暴露信息
```

---

### 3. 调用原则

LLM生成内容前，应根据任务类型调用不同资料。

生成总纲时，优先调用：

```text
核心世界观
主角设定
核心冲突设定
主要大事件
全书主题
类型文定位
```

生成单元大纲时，优先调用：

```text
单元目标
相关大事件
相关人物设定
相关组织设定
单元情绪曲线
单元结局要求
```

生成章纲时，优先调用：

```text
单元大纲
相关大事件
出场人物卡
相关伏笔
信息释放限制
上一章摘要
下一章目标
```

生成正文时，优先调用：

```text
章节细纲
出场人物卡
当前场景信息
上一章正文摘要
文风规则
对话规则
节奏规则
禁用写法
```

---

## 九、写作生成Skill

### 1. 系统定位

写作生成Skill负责定义 LLM 应该如何写。

它关注正文质量、文风一致性、人物表现、节奏控制、去AI化、网络小说可读性和读者体验。

---

### 2. 核心职责

写作生成Skill负责：

```text
定义正文文风
定义人物对话方式
定义章节结构
定义网文节奏
定义爽点设计
定义悬念设计
定义反转设计
定义情绪推进
定义信息差
定义伏笔埋设与回收
定义去AI化规则
定义禁止写法
定义修改标准
```

---

### 3. 写作规则方向

写作生成Skill应包含以下规则：

```text
对话必须口语化、生活化
禁止人物用说明文口吻解释设定
人物关系通过动作、停顿、寒暄和留白表现
少写心理，少写大段环境
正文主要靠动作和对话推进
每章必须有明确读者奖励
每章应形成压制、触发、兑现、余波
主角不能降智
读者可以领先主角，但主角不能显得愚蠢
异常内容可被读者感知，但不能让主角过早理解
人物台词必须符合人物身份、年龄、经验和当下处境
禁止AI腔、总结腔、解释腔、概念堆砌
```

## 十、四大系统边界

### 1. 设定系统与事件数据库的边界

设定系统记录规则。

事件数据库记录事实。

例如：

```text
任战的能力机制 → 设定系统
任战在某一时间段利用该能力布局现实骗局 → 事件数据库
```

---

### 2. 事件数据库与叙事系统的边界

事件数据库记录世界真实发生的大事件。

叙事系统决定这些大事件如何被拆分、重组、隐藏、揭露并写给读者。

例如：

```text
《陈科被高端分享局诈骗事件》 → 事件数据库
该事件被拆成若干章节并安排悬念、反转、节奏 → 叙事系统
```

---

### 3. 叙事系统与Skill系统的边界

叙事系统存放输出结果。

Skill系统存放生成规则。

例如：

```text
P0十五章大纲 → 叙事系统
如何根据事件生成P0十五章大纲 → Skill系统
```

---

### 4. 设定系统与Skill系统的边界

设定系统记录小说内容本身。

Skill系统规定 LLM 如何使用这些内容。

例如：

```text
马木的人物设定 → 设定系统
写马木时如何表现嘴直但克制、热心但不圣母 → Skill系统
```

---

## 十一、Obsidian落地原则

### 1. 文件标准化

所有资料必须采用标准文件模板存储，便于 LLM 调用、检索、筛选和校验。

文件应包含：

```text
唯一ID
标题
类型
状态
所属系统
核心关联对象
标签
正文内容
双链关系
```

文件内容分为两层：

```text
YAML：用于简洁索引和精确定位
Markdown正文：用于承载具体内容和关系说明
```

---

### 2. 文件夹结构

Obsidian 根目录采用五个一级文件夹：

```text
Novel_System/
├── 00_总控台/
├── 01_叙事系统/
├── 02_设定系统/
├── 03_事件数据库/
└── 04_Skill系统/
```

文件夹可以多层嵌套，但文件夹只服务作者理解。

文件最终分类通过 YAML 和 Database 确认。

不得将同一内容的正式文档分别存放在不同文件夹中。

---

### 3. 文件命名规则

普通文件采用：

```text
ID_标题.md
```

示例：

```text
CHR-0001_马木.md
EVT-0002_陈科被高端分享局诈骗事件.md
POW-0002_任战-入席.md
SKL-WR-001_正文生成总规则.md
```

章节类文件采用稳定编号命名，文件名不写章节标题：

```text
NAR-P00-C01_章节大纲.md
OUT-P00-C01_章节细纲.md
DRAFT-P00-C01_正文.md
REV-P00-C01_修订记录.md
```

章节标题写入 YAML 的 `title` 字段和 Markdown 正文中。

---

### 4. YAML属性

所有关键文档应使用 YAML 属性，方便后续通过 Database、脚本或 LLM 检索。

YAML 只保留简洁索引字段。

通用字段如下：

```yaml

type:
system:
status:
title:
rel:
tags:
```

可选字段如下：

```yaml

unit:
chapter:
world_time:

role:
pov:
event_type:
task:
owner:
```

字段使用规则：

```text
system 使用英文
type 使用英文
rel 只写 ID，不写 Obsidian 双链
alias 用于别名、简称、旧称、误称
复杂内容写入 Markdown 正文
```

system 字段建议枚举：

```text

narrative
setting
event

```

```text

character
event

detail_outline
draft

organization
location

```

status 字段建议枚举：

```text

draft
review
active
confirmed
deprecated

```

### 5. 双链关联

系统应充分利用 Obsidian 双链能力。

双链写在 Markdown 正文中，不写在 YAML 的 `rel` 字段中。

常见关联包括：

```text
人物 ↔ 事件
人物 ↔ 组织
人物 ↔ 地点
事件 ↔ 设定
事件 ↔ 叙事单元
设定 ↔ 术语
章节 ↔ 伏笔
章节 ↔ 事件
章节 ↔ 人物
Skill ↔ 调用规则
Skill ↔ 校验规则
```

YAML 负责说明“有关联”。

Markdown 正文中的双链负责说明“是什么关系”。

示例：

```markdown
## 关联说明

- [[CHR-0001_马木]]：事件介入者，最终被反咬。
- [[CHR-0003_陈科]]：被骗者，也是反咬马木的关键人物。
- [[EVT-0002_陈科被高端分享局诈骗事件]]：P0现实崩塌主线事件。
- [[POW-0002_任战-入席]]：事件背后的隐藏诱导机制。
```

## 十二、核心模板规范

### 1. 核心模板设计目标

本阶段先确定四类核心模板的架构规范：

```text
人物卡
事件卡
章节大纲卡
正文生成 Skill
```

这四类模板构成小说写作系统的最小可运行链路：

```text
人物卡提供人物档案和角色规则
事件卡提供世界事实和事件时间线
章节大纲卡把人物与事件转化为章节叙事设计
正文生成 Skill 规定 LLM 如何调用资料、生成正文并做基础自检
```

本阶段只定义模板规范，不创建具体 Obsidian 模板文件。

模板规范用于说明：

```text
1. 该类文件的系统定位
2. YAML 应保留哪些简洁索引字段
3. Markdown 正文应承载哪些复杂内容
4. 该文件如何与其他文件建立关联
5. LLM 调用该文件时应重点读取什么
```

---

### 2. 人物卡模板规范

#### 2.1 模板定位

人物卡属于设定系统，定位为人物的完整档案卡。

人物卡用于集中记录人物基础档案、人物定位、关系摘要、经历摘要、当前状态和关键设定。

人物卡不负责完整记录阶段变化明细。人物在不同阶段的变化，应主要记录在事件数据库中的人物经历线中。

人物卡与人物经历线的分工如下：

```text
人物卡：记录人物档案、经历摘要、当前状态和作者阅读入口
人物经历线：记录阶段变化、状态变化和经历节点明细
```

同一人物只能有一个正式人物卡，不得因为不同阶段、不同主线或不同用途重复创建多个人物正式文件。

#### 2.2 YAML 字段

```yaml
id: CHR-0000
type: character
system: setting
status: draft
title:
alias:
line:
role:
rel: []
tags: []
```

字段说明：

|字段|说明|
|---|---|
|`id`|人物唯一 ID|
|`type`|固定为 `character`|
|`system`|固定为 `setting`|
|`status`|人物卡状态|
|`title`|人物正式名称|
|`alias`|别名、简称、旧称、误称；没有可不写|
|`line`|人物所属故事线，如现实线、异常线、宇宙线|
|`role`|人物叙事角色，如主角、配角、反派、关键见证人|
|`rel`|核心关联对象 ID，如关键事件、组织、能力|
|`tags`|补充分类标签|

#### 2.3 正文结构

人物卡正文采用百科档案型结构，优先服务作者阅读和资料总览。

```markdown
## 核心定位

## 基础信息

## 外貌与识别特征

## 身份与社会关系

## 人物性格

## 说话方式

## 行为习惯

## 能力与资源

## 经历摘要

## 当前状态

## 已知信息范围

## 主要关系

## 关联事件

## 关联设定

## 人物经历线

## 写作禁忌
```

#### 2.4 调用重点

LLM 在生成章节大纲或正文时，读取人物卡应重点关注：

```text
人物当前身份与状态
人物在本故事线中的作用
人物和其他角色的关系
人物说话方式
人物行为习惯
人物已知与未知的信息边界
人物关联的人物经历线
人物写作禁忌
```

---

### 3. 事件卡模板规范

#### 3.1 模板定位

事件卡属于事件数据库，定位为完整事实卡。

事件卡用于记录小说世界中真实发生的故事级大事件，是事件事实的权威来源。

事件卡基于世界时间轴，不基于读者阅读顺序，也不基于章节编号。

事件卡不拆分章节节点，不设计场景，不安排爽点、悬念、反转和结尾钩子。

事件卡不记录“被哪些叙事内容使用”。事件如何被章节展示、隐藏、误导或回收，应记录在章节大纲卡中。

#### 3.2 YAML 字段

```yaml
id: EVT-0000
type: event
system: event
status: draft
title:
world_time:
event_type:
line:
rel: []
tags: []
```

字段说明：

|字段|说明|
|---|---|
|`id`|事件唯一 ID|
|`type`|固定为 `event`|
|`system`|固定为 `event`|
|`status`|事件状态|
|`title`|事件名称|
|`world_time`|世界内时间或时间范围|
|`event_type`|事件性质，如现实诈骗事件、人物前史事件、组织行动事件|
|`line`|事件所属故事线，如现实线、异常线、宇宙线|
|`rel`|核心关联对象 ID，如人物、组织、地点、能力|
|`tags`|补充分类标签|

#### 3.3 正文结构

事件卡正文采用时间线型结构，优先记录事件真实发生过程。

```markdown
## 事件摘要

## 世界时间

## 事件性质

## 事件背景

## 起因

## 时间线

### 节点一

### 节点二

### 节点三

## 结果

## 后续影响

## 涉及人物

## 涉及组织

## 涉及地点

## 涉及设定

## 逻辑限制
```

#### 3.4 调用重点

LLM 在生成章节大纲或正文时，读取事件卡应重点关注：

```text
事件在世界中真实发生了什么
事件在世界时间轴中的位置
事件的起因、时间线、结果和影响
事件涉及哪些人物和设定
事件有哪些不能违背的逻辑限制
```

---

### 4. 章节大纲卡模板规范

#### 4.1 模板定位

章节大纲卡属于叙事系统，定位为叙事设计卡。

章节大纲卡用于把人物卡、事件卡和必要设定加工为本章的叙事安排。

章节大纲卡只做章级设计，不拆分具体场景。场景列表、场景顺序、对白重点和动作细节，应留给章节细纲。

章节大纲卡重点回答：

```text
本章承担什么叙事功能
本章调用哪些世界事实
本章让读者知道什么
本章让主角知道什么
本章隐藏、误导或回收哪些信息
本章给读者什么奖励
本章如何结尾
```

#### 4.2 YAML 字段

```yaml
id: NAR-P00-C00
type: chapter_outline
system: narrative
status: draft
title:
unit: P00
chapter: C00
pov:
rel: []
tags: []
```

字段说明：

|字段|说明|
|---|---|
|`id`|章节大纲唯一 ID|
|`type`|固定为 `chapter_outline`|
|`system`|固定为 `narrative`|
|`status`|章节大纲状态|
|`title`|章节标题；标题变化时改 YAML 和正文，不改文件名|
|`unit`|所属单元编号|
|`chapter`|章节编号|
|`pov`|本章主要视角人物 ID；多视角时可写多个 ID|
|`rel`|核心关联对象 ID，如人物、事件、设定、章节细纲|
|`tags`|补充分类标签，如 P00、C01、章节大纲|

#### 4.3 正文结构

章节大纲卡正文采用功能驱动型结构。

```markdown
## 本章功能

## 章节目标

## POV

## 出场人物

## 相关事件

## 事件使用方式

## 相关设定

## 读者知道的信息

## 主角知道的信息

## 本章释放的信息

## 本章隐藏的信息

## 本章误导的信息

## 伏笔与回收

## 读者奖励

## 情绪推进

## 结尾钩子

## 禁止事项
```

#### 4.4 调用重点

LLM 在生成正文前，读取章节大纲卡应重点关注：

```text
本章要完成什么叙事功能
哪些人物必须出场
本章调用哪些事件事实，以及如何使用这些事实
读者知道什么
主角知道什么
哪些信息可以释放
哪些信息需要隐藏或误导
本章给读者的奖励是什么
本章结尾要把读者推向哪里
```

### 5. 正文生成 Skill 模板规范

#### 5.1 模板定位

正文生成 Skill 属于 Skill 系统，定位为执行流程与写作规则文件。

正文生成 Skill 用于规定 LLM 如何调用资料、组织上下文、生成正文，并完成基础自检。

正文生成 Skill 不存放具体小说正文内容，不存放人物、事件和世界观设定本身，也不直接写成 Prompt。

Skill 与 Prompt 的分工如下：

```text
Skill：规定流程、规则、禁用项和自检要求
Prompt：后续将 Skill 规则转化为可投喂 LLM 的表达格式
```

#### 5.2 YAML 字段

```yaml
id: SKL-WR-000
type: skill
system: skill
status: draft
title:
task: write_draft
rel: []
tags: []
```

字段说明：

|字段|说明|
|---|---|
|`id`|Skill 唯一 ID|
|`type`|固定为 `skill`|
|`system`|固定为 `skill`|
|`status`|Skill 状态|
|`title`|Skill 名称|
|`task`|任务类型，正文生成固定为 `write_draft`|
|`rel`|关联的资料调用 Skill、校验 Skill、风格 Skill 等 ID|
|`tags`|中文分类标签，如正文生成、写作规则|

#### 5.3 正文结构

```markdown
## 适用任务

## 默认输入资料

## 资料调用顺序

## 上下文组织方式

## 正文生成规则

## 人物表现规则

## 对话规则

## 节奏规则

## 信息释放规则

## 写作层禁用规则

## 输出格式

## 生成后基础自检

## 关联 Skill
```

#### 5.4 调用重点

正文生成 Skill 默认调用以下资料：

```text
章节大纲
章节细纲
出场人物卡
相关事件卡
必要设定卡
```

必要设定卡不默认全量调用，应根据章节大纲、人物卡和事件卡中的 `rel` 字段补充读取。

LLM 执行正文生成 Skill 时，应按以下顺序组织资料：

```text
1. 读取正文生成 Skill
2. 读取章节大纲卡
3. 读取章节细纲或场景安排
4. 读取出场人物卡
5. 读取相关事件卡
6. 读取必要设定卡
7. 读取信息释放规则
8. 生成正文
9. 检查人物是否崩坏
10. 检查事件事实是否冲突
11. 检查是否提前暴露信息
12. 检查是否出现明显 AI 腔、总结腔、解释腔和概念堆砌
```

正文生成 Skill 的核心约束为：

```text
正文主要通过动作和对话推进
人物不能用说明文口吻解释设定
人物台词必须符合身份、经验和当前处境
读者可以获得信息差，但主角不能显得愚蠢
异常内容可以被读者感知，但不能让主角过早理解
每章必须提供明确读者奖励
每章结尾应形成继续阅读的推动力
```

正文生成 Skill 的输出格式采用基础格式：

```markdown
## 调用摘要

## 正文

## 生成后自检
```

其中：

```text
调用摘要：简要列出本次调用的章节大纲、章节细纲、人物卡、事件卡和必要设定
正文：正式正文内容
生成后自检：检查人物、事件、信息释放和明显 AI 腔问题
```

---

### 6. 四类模板之间的关系

四类核心模板的调用关系如下：

```text
人物卡 + 事件卡
        ↓
章节大纲卡进行叙事设计
        ↓
正文生成 Skill 调用章节大纲卡、人物卡、事件卡和必要设定
        ↓
生成正文
```

其中：

```text
人物卡回答“这个人是谁、当前是什么状态”
事件卡回答“世界中真实发生了什么”
章节大纲卡回答“本章让读者和主角分别知道什么”
正文生成 Skill 回答“LLM 应该如何调用资料并写成正文”
```

四类模板的职责边界如下：

```text
人物卡写人物档案和经历摘要，不写阶段变化明细
人物经历线写阶段变化明细
事件卡写世界事实，不写叙事使用记录
章节大纲卡写事件在本章中的展示、隐藏、误导和回收方式
正文生成 Skill 写流程和规则，不写 Prompt 正文
完整一致性校验由后续一致性校验 Skill 承担
```

---

## 十三、系统运行流程

系统的基础运行流程如下：

```text
1. 作者在设定系统中维护人物、世界观、能力、组织、道具等硬规则
2. 作者在事件数据库中维护世界时间轴和故事级大事件
3. Skill系统根据写作任务调用相关资料
4. LLM在Skill规则约束下生成叙事系统内容
5. 叙事系统形成总纲、卷纲、单元纲、章纲、细纲和正文
6. 正文生成后，系统根据结果回写必要的事件、设定、伏笔和修订记录
7. Skill系统继续进行一致性校验和正文质量修订
```

## 十四、系统阶段目标

### V1.1阶段目标

V1.1阶段主要完成 Obsidian 落地规范设计，明确五个一级文件夹结构、文件命名规则、YAML字段规则、双链使用规则、文件唯一性原则，以及第一批核心模板规范。

# 结论

小说写作系统应采用五个 Obsidian 一级文件夹：

```text
00_总控台
01_叙事系统
02_设定系统
03_事件数据库
04_Skill系统
```

其中四大核心模块为：

```text
设定系统 = 世界规则库
事件数据库 = 世界时间轴事实库
叙事系统 = 正文生产线
Skill系统 = LLM创作控制系统
```

系统底层以世界时间轴为基础，保证事件、人物经历和组织行动的逻辑一致。

叙事系统使用 B / V / P / C 等小说结构编号，负责将世界事实加工为读者可阅读的小说结构。

Obsidian 中，文件夹服务作者理解，YAML 服务精确定位，Database 服务筛选归类，Markdown 正文承载复杂内容，双链用于说明语义关系。

Skill系统服务于 LLM，既负责资料调用，也负责写作生成，是连接资料系统与正文生产的核心控制层。
