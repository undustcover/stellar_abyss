# 意图控制系统第一版需求设计

## 一、系统定位

意图控制系统用于辅助长篇网络小说正文生成。

系统核心目标不是替代作者完成全部创作决策，而是通过持续识别、验证和校正人物意图，降低大模型在正文生成过程中出现人物失真、行为失因、对白失控和剧情机械推进的概率。

第一版系统的核心定义为：

> 以人物语言意图和人物行为意图为核心，通过关键片段检查识别人物驱动层面的分歧，并在关键分歧出现时中断生成，确保剧情由人物的表达与选择自然产生。

系统重点解决以下问题：

1. 情节走向正确，但人物说话不像本人；
2. 信息没有提前暴露，但对白变成说明书；
3. 场景功能正确，但人物行为只是在配合大纲；
4. 作者无法在正文生成前一次性表达全部人物意图；
5. 模型通过抽象概念复述意图，看似理解，实际落地错误；
6. 修订阶段只润色句面，没有识别语言意图或行为意图偏差；
7. 第一版正文方向性错误，导致第二版需要整体重写；
8. 生成过程重复读取大量资料，token 消耗过高。

---

# 二、核心设计原则

## 2.1 整章交付，人物意图控制

系统最终交付单位为完整章节。

系统结构单位为场景。

系统控制单位为人物意图。

系统落地单位为对话与行为。

系统实际输出单位为关键片段。

即：

```text
交付单位：整章
结构单位：场景
控制单位：人物意图
落地单位：对话与行为
输出单位：关键片段
验证单位：关键对白 / 关键动作 / 最小片段
```

“整章生成”不等于一次调用完成整章。

一章正文可以跨多个交互轮次连续生成，但最终必须形成一份结构完整、语气统一、节奏连贯的整章第一版。

---

## 2.2 人物才是意图主体

场景、伏笔、节奏、信息差和喜剧机制都很重要，但它们不是意图控制的中心。

真正有意图的是人物。

因此，系统必须优先回答：

```text
人物为什么说这句话？
人物为什么做这个动作？
人物说完或做完以后，希望对方发生什么变化？
人物为什么不采取更简单、更直接的做法？
```

剧情应当从以下链条自然产生：

```text
人物长期设定
    ↓
人物当前认知
    ↓
人物当前行为目标
    ↓
人物当前语言目标
    ↓
人物选择表达与行动策略
    ↓
人物语言和行为
    ↓
对方反应及现实后果
    ↓
剧情推进
    ↓
读者体验、喜剧、伏笔、节奏
```

而不是：

```text
章节需要发生某件事
    ↓
安排人物说话和行动
```

---

## 2.3 动态对齐，而非一次性需求确认

系统不要求作者在正文生成前一次性说明全部人物意图。

作者意图会在以下过程中逐步显现：

```text
模型建立人物意图轨道
→ 作者确认或直接改写轨道
→ 模型更新章节运行态
→ 正文生成
→ 正文暴露剩余偏差
→ 作者继续纠偏
```

第一版采用显式人物意图轨道。

成熟后可预留内部检查模式：

```text
高风险片段：展示人物意图轨道
低风险片段：内部检查，直接 CONTINUE
```

---

## 2.3.1 作者意图与正文表现控制边界

本系统第一版以人物语言意图和人物行为意图为核心，但第三章第二次正文测试暴露出一个新的边界问题：

```text
人物意图正确
≠
正文读起来符合作者想要的读者体验
```

例如：

- 马木从人物角度可以问出正确问题，但如果正文把这些问题写得过于清楚，读者会越过马雯雯提前察觉诈骗方向；
- 执行包里的每个控制点都被完整展示，人物没有越界，但正文会变得过满、过累、像说明文；
- 视角人物被误导时，读者是否也应被误导，属于作者对本章表现方式的要求，不等同于某个角色为什么说或为什么做；
- AI 感、板正、没有留白，往往不是人物意图错误，而是正文输出策略错误。

因此，系统需要区分两类控制：

```text
人物意图控制：
控制人物为什么说、为什么做、为什么不采取更简单行动。

作者意图 / 正文表现控制：
控制本章应该如何展现给读者、读者应该知道多少、哪些信息需要被视角误读过滤、哪些内容必须降噪或留白。
```

人物意图控制不应吞并正文表现问题。

如果正文已经理解了章节内容，但问题表现为：

- 内容太多；
- 所有信息都写满；
- 视角过滤失败；
- 读者获得的信息超过视角人物；
- 文本像说明文而不是小说；
- 每段都太精修、太板正；
- 缺少轻重缓急和留白；

则优先归因为 `正文输出控制 / 作者意图修订`，而不是继续强化人物意图控制器。

该结论会影响两处链路：

1. **草稿前**：正文生成前应有最低限度的正文输出策略，防止第一稿天然写满。
2. **草稿后**：当已有草稿内容足够完整时，应允许作者意图修订从表现方式入手，对同一组事实进行减法、视角重排、信息降噪和留白重写。

一句话规则：

```text
当前要后置修，系统要前置防。
```

---

## 2.4 第一版必须达到“基本写对”

第一版正文不能仅作为试错稿。

第一版目标是：

- 人物语言意图基本正确；
- 人物行为意图基本正确；
- 人物认知边界基本正确；
- 情节因果基本正确；
- 信息差基本正确；
- 场景功能基本正确；
- 主要喜剧机制基本正确；
- 伏笔和信息释放基本正确；
- 可直接进入精修，而不是整体推倒重写。

版本功能定义：

```text
V1：写对
V2：写好
V3及以后：精修
```

若 V1 需要全章重写，说明意图控制机制失效。

---

## 2.5 风险触发，而非固定提问

每个关键片段生成前都必须执行人物意图检查。

但并非每个片段都必须向作者提问。

系统只有在发现关键分歧时才中断生成并请求作者确认。

即：

```text
每个关键片段必须检查
≠
每个关键片段必须提问
```

第一版虽然显式列轨道，但只有 ASK_USER 才真正打断正文生成。

---

## 2.6 质量优先，同时清除无效 token

第一阶段以正文质量为硬目标。

但从第一版开始就必须避免明显无效的 token 消耗。

应保留：

- 当前人物认知；
- 当前人物行为目标；
- 当前人物语言目标；
- 当前人物表达策略；
- 章节必要背景；
- 信息差；
- 已确认规则；
- 极短验证片段；
- 作者反馈；
- 当前连续正文。

应删除：

- 重复加载完整大纲；
- 重复加载全部人物卡；
- 重复输出方法论；
- 大量抽象分析；
- 每个片段完整长篇意图报告；
- 已经确认的问题重复询问；
- 多套完整正文方案；
- 无复用价值的历史修订记录。

---

# 三、系统总体流程

```text
读取章节资料
    ↓
建立每章一份章节运行态
    ↓
完成章节级意图理解
    ↓
确认章节关键分歧
    ↓
进入整章 V1 生成
    ↓
进入关键片段
    ↓
显式列人物意图轨道
    ↓
人物语言 / 行为意图检查
    ├─ CONTINUE：生成当前正文片段
    └─ ASK_USER：停止正文并向作者提问
    ↓
作者确认或直接改写轨道
    ↓
系统复述确认结果
    ↓
更新章节运行态
    ↓
从中断点继续
    ↓
完成全部关键片段
    ↓
整章组装与一致性检查
    ↓
输出整章 V1
    ↓
作者整章审阅
    ↓
偏差归因
    ↓
生成 V2
```

动态对齐第一版主要在对话框中完成。

确认结果必须写入章节运行态文档。

正文草稿文档保持干净，不塞入大量过程分析。

---

# 四、章节级意图控制

## 4.1 章节生成前必须明确的内容

系统在正文生成前应理解：

```text
本章核心目的
本章主场景
本章次要场景
本章情绪与节奏曲线
本章信息释放上限
本章主要误会或冲突机制
本章主要笑点机制
人物状态变化
关键伏笔
本章结尾状态
每个场景的独占功能
每个关键人物的基础认知和关系压力
```

## 4.2 章节级对齐的作用

章节级对齐负责：

- 确定整体方向；
- 确定章节边界；
- 确定信息释放总量；
- 确定主次关系；
- 防止局部片段过度优化；
- 防止每个场景都承担相同功能；
- 为人物意图轨道提供上层约束。

章节级对齐不能替代人物意图对齐。

---

# 五、关键片段与人物意图轨道

## 5.1 关键片段定义

关键片段可以是：

- 一个重要对白回合；
- 一个关键行为选择；
- 一个误会升级段；
- 一个信息释放段；
- 一个强笑点落地段；
- 一个关系变化段；
- 一个场景入口或出口段；
- 一个会影响后续剧情的动作链。

关键片段不是整场成品，也不是单句台词。

## 5.2 人物意图轨道

每个关键片段生成前，系统必须显式列出相关人物的意图轨道。

建议结构：

```yaml
character_intent_tracks:
  人物名:
    behavior_goal:
    behavior_motive:
    immediate_action:
    why_not_simpler_action:
    language_goal:
    language_strategy:
    emotional_pressure:
    knowledge:
      knows:
      does_not_know:
      assumes:
      suspects:
    forbidden:
    expected_effect_on_other:
    expected_consequence:
```

作者可以直接改写人物意图轨道。

系统必须将作者修改转化为明确规则，写入章节运行态。

---

# 六、语言意图控制

人物语言意图是第一优先级。

一轮有效语言行为的结构为：

```text
人物受到刺激
→ 形成语言目的
→ 选择表达策略
→ 说出具体内容
→ 对方按照自己的认知理解
→ 产生反应
```

语言意图必须检查：

1. 稳定语言人格；
2. 当前语言目标；
3. 当前认知边界；
4. 当前情绪压力；
5. 表达策略；
6. 语言禁区。

必须明确：

- 人物为什么开口；
- 想让对方知道什么；
- 想让对方误解什么；
- 想从对方口中得到什么；
- 哪些内容不能说；
- 用什么方式说；
- 说出口的话和真实想法之间有什么距离。

---

# 七、行为意图控制

人物行为意图是第二优先级。

关键行为必须形成完整因果链：

```text
外部刺激
→ 人物认知
→ 人物目标
→ 可选行动
→ 人物选择
→ 选择理由
→ 行动结果
→ 状态变化
```

系统必须强制检查：

```text
如果人物没有采取最直接行动，必须有认知、情绪、关系、利益或外部压力理由。
```

如果人物不直接问、不直接走、不直接报警、不直接拒绝、不直接解释，必须能从人物自身处境中解释。

解释不能只来自“剧情需要”。

---

# 八、叙事意图控制

场景、信息差、节奏、伏笔、喜剧机制和读者领先程度仍然重要，但它们属于外层约束。

叙事意图必须转化为人物语言和行为才能进入正文。

例如：

```text
本场要维持误会
```

必须转换为：

```text
陈艾艾为什么不解释？
马雯雯为什么不追问真实情况？
陈科为什么继续强调项目可靠？
```

只有人物自己的目标成立，误会才成立。

同时，叙事意图还包含一类不能完全转化为人物意图的问题：

```text
作者希望读者如何获得信息、误读信息、忽略信息和感受文本。
```

这类问题包括：

- 读者是否只能通过视角人物获得信息；
- 视角人物被误导时，读者是否也必须被误导；
- 正确线索是否需要写成背景噪音；
- 某些人物的有效行动是否不应被读者立即理解；
- 哪些信息只允许作为生活细节、物件、错接、停顿或误会出现；
- 哪些内容虽然在执行包里存在，但正文中必须轻写、略写或不写满。

如果这些问题在草稿前已经明确，应进入正文输出策略；如果在草稿后由作者指出，应进入作者意图修订，而不是回退到人物意图轨道。

例如第三章测试结论：

```text
马雯雯是线索人物。
读者获得的信息应通过马雯雯过滤。
马雯雯被误导，读者也应被误导。
马木的正确追问不能被正文展示成读者可识别的诈骗线索。
```

这种控制不是“马木为什么问”的问题，而是“马木问出来的内容如何被正文呈现给读者”的问题。

---

# 九、分歧分级机制

## 9.1 P0：人物语言意图分歧

出现以下情况必须 ASK_USER：

- 不清楚人物为什么开口；
- 同一句台词可以服务于两个不同目标；
- 不清楚人物是试探、解释、遮掩、反击、敷衍还是转移话题；
- 不清楚人物想让对方知道还是误解；
- 不清楚人物是否会直接说出某件事；
- 不清楚人物在此关系下会采用什么态度；
- 两种语言方向会塑造出不同人物形象；
- 台词虽然能推动剧情，但可能违背人物。

## 9.2 P0：人物行为意图分歧

出现以下情况必须 ASK_USER：

- 不清楚人物当前目标；
- 同一人物存在两个同样合理的行动选择；
- 当前行为只服务剧情，不符合人物；
- 行为会显著改变情节走向；
- 人物主动还是被动不明确；
- 人物为什么不采取更简单行动无法解释；
- 行为会改变人物关系；
- 行为意味着人物态度发生转变。

## 9.3 P1：认知与叙事机制分歧

包括：

- 人物认知边界；
- 信息差；
- 误会机制；
- POV；
- 读者信息量；
- 场景主动权；
- 伏笔；
- 喜剧机制。

当这些问题会影响人物语言或行为时，必须 ASK_USER。

当这些问题不影响人物语言或行为，但会决定整章读者体验时，不应强行包装成人物意图分歧；应转入 `正文输出策略确认` 或 `作者意图修订`。

典型触发：

- 读者是否可以比视角人物更早知道真相；
- 正确线索是否需要被误读包住；
- 本章应该让读者看出风险，还是让读者跟着视角人物误判；
- 执行包中的信息是否应该在正文中写满；
- 本章主阅读体验是“揭示”还是“误导”。

## 9.4 P2：表现层分歧

包括：

- 节奏；
- 详略；
- 环境描写；
- 句子润色；
- 段落长短；
- 一般过渡；
- token 压缩。

P2 问题不得单独中断，除非反过来影响人物语言或行为。

但如果 P2 问题在成稿后成为整章级问题，例如：

- 文章过满；
- 没有留白；
- 所有场景都像重点；
- 句段结构过于精修和板正；
- 阅读体验累；
- 文本像说明文；

则不再视为普通句面润色，而应归入 `正文表现偏差`，进入草稿后作者意图修订流程。

---

# 十、禁止中断条件

以下情况不得向作者提问：

- 普通措辞选择；
- 一般动作设计；
- 无关紧要的环境描写；
- 已有规则可以唯一推导；
- 作者已经明确回答过；
- 不影响人物语言、行为、认知、因果和节奏；
- 只是模型不知道怎样写得更好；
- 只是局部文笔难度较高；
- 两种方式差异极小。

必须明确：

```text
写作难度高
≠
作者意图存在分歧
```

模型不能把正常创作责任转交给作者。

---

# 十一、中断提问协议

当系统触发 ASK_USER 时，必须立即停止正文生成。

提问必须包含：

1. 已确认内容；
2. 唯一关键分歧；
3. 模型推荐方向；
4. 最小验证片段；
5. 允许作者直接改写人物意图轨道。

标准格式：

```markdown
## 人物意图中断：需要确认一个关键分歧

### 已确认

- ...

### 当前分歧

A方向：...

B方向：...

### 我的判断

建议采用...

原因：...

### 最小验证片段

...

请确认 A、B，或直接改写人物意图轨道。
```

输出该问题后，当前轮次必须结束。

不得在问题后继续写正文。

---

# 十二、恢复生成协议

作者回答后，系统必须完成以下步骤。

## 12.1 将自然语言反馈转换为明确规则

例如作者回答：

```text
不是证明项目可靠，而是证明自己不是被人骗的蠢货。
```

系统应转换为：

```text
陈科当前语言目标不是解释项目本身，而是维护自尊，避免承认自己判断失误。
```

## 12.2 判断影响范围

作者反馈可能影响：

```text
当前片段
当前场景
本章后续
长期人物规则候选
```

## 12.3 更新章节运行态

作者确认后的规则必须进入当前章节运行态。

不能只依赖聊天上下文。

## 12.4 复述确认后继续

恢复生成前必须复述：

```markdown
已确认：
- ...

影响范围：
- ...

已更新章节运行态。

继续当前正文片段：
```

不得重复此前正文，不得从章节开头重写，不得擅自修改已确认内容。

---

# 十三、章节运行态

系统必须维护每章一份轻量章节运行态。

建议命名：

```text
CXX_章节运行态_意图控制.md
```

章节运行态保存：

- 当前章临时人物意图；
- 作者确认的本章规则；
- 分歧解决记录；
- 断点；
- 正文片段生成状态；
- 长期规则候选。

章节运行态不保存：

- 完整人物卡；
- 长期世界事实；
- 完整正文全文；
- 未经作者确认的长期人物结论；
- 通用写作方法论长篇复述。

临时人物意图不得自动回写人物卡。

章节结束后，再判断哪些内容值得沉淀为长期人物规则候选。

# 十四、正文生成方式

## 14.1 关键片段不是独立交付成品

每个关键片段生成后，不要求立即完成最终验收。

片段正文在整章完成前属于：

```text
整章 V1 生成中的当前正文
```

而不是独立的：

```text
Fragment Final
```

## 14.1.1 正文输出策略

正文生成前，除人物意图轨道外，还应读取或临时建立一份轻量 `正文输出策略`。

它不替代执行包，不重写细纲，也不生成正文，而是把作者对本章读者体验的要求转换为正文输出约束。

建议字段：

```yaml
prose_output_strategy:
  reader_access:
    viewpoint_filter: 读者只能通过谁获得信息
    reader_should_know:
      -
    reader_should_misread:
      -
    reader_must_not_infer:
      -
  presentation_weight:
    write_full:
      -
    write_light:
      -
    background_noise:
      -
    leave_blank:
      -
  information_noise_control:
    correct_clues_to_hide:
      -
    clues_to_translate_through_viewpoint_misread:
      -
    prohibited_reader_signals:
      -
  prose_density:
    main_effect:
    heavy_scenes:
      -
    light_scenes:
      -
    compression_rule:
    blank_space_rule:
  anti_ai_expression:
    avoid_structure:
      -
    allow_loose_actions:
      -
    sentence_rhythm_rule:
```

正文输出策略回答的是：

```text
这章应该以什么方式展示给读者？
读者应该被谁的认知过滤？
哪些正确内容不能让读者当成正确线索？
哪些地方需要轻写、跳写、留白或背景化？
```

它不回答：

```text
人物为什么说？
人物为什么做？
人物为什么不采取更简单行动？
```

这些仍由人物意图控制器处理。

## 14.2 片段间必须继承信息

后续片段必须继承：

- 前文已生成内容；
- 已确认人物意图；
- 已确认人物状态；
- 已释放的信息；
- 已使用的笑点；
- 已建立的节奏；
- 新增禁止方向；
- 已解决分歧。

## 14.3 全章完成后统一检查

所有片段生成完毕后，必须进行整章检查：

```text
人物语言意图是否持续稳定
人物行为因果是否成立
场景衔接是否自然
语气是否统一
人物声音是否稳定
前后是否重复
主次是否清楚
详略是否合理
笑点是否过密或过平均
信息释放是否超标
伏笔是否过于明显
情绪曲线是否成立
结尾钩子是否有效
```

完成整章检查后，才输出完整 V1。

# 十五、正文修订机制

作者完成整章审阅后，系统必须先进行偏差归因，不能直接改写。

偏差分为六类。

## 15.1 人物语言意图偏差

例如：

- 人物为什么开口不成立；
- 台词服务了作者解释，而不是人物目标；
- 人物表达策略错误；
- 人物语言禁区被突破。

处理：

```text
回到对应关键片段，重建语言意图轨道后局部重写。
```

## 15.2 人物行为意图偏差

例如：

- 行为只服务剧情；
- 人物为什么不采取更简单行动无法解释；
- 行为依据不符合人物认知；
- 行为导致人物关系变化但前文没有铺垫。

处理：

```text
回到对应关键动作链，重建行为意图后局部重写。
```

## 15.3 局部表现偏差

例如：

- 台词不自然；
- 某段节奏拖沓；
- 笑点尴尬；
- 旁白解释过度。

处理：

```text
局部替换或压缩。
```

## 15.4 正文表现 / 作者意图偏差

例如：

- 内容不是不够，而是太多；
- 草稿把执行包内容全部饱满展示出来；
- 读者获得信息的角度错误；
- 视角人物被误导，但读者没有被误导；
- 正确线索没有被视角过滤，导致读者提前识别真相；
- 每个场景都像重点，缺少轻重缓急；
- 文本读起来累，像说明文；
- 句子和段落结构过于精修、板正、呆板；
- 缺少留白、错接、无效动作和自然松动。

处理：

```text
基于已有草稿进行作者意图修订。
不优先回退人物意图控制。
不优先重跑细纲和执行包。
对同一组事实进行视角收紧、信息降噪、详略重排、留白、句段松动和局部删减。
```

作者意图修订的目标不是换词，而是改变正文展示方式：

```text
从“把内容写全”
改为
“让读者按作者希望的方式误读、感受、忽略和追读”
```

第三章测试中的典型修订口径：

```text
马木仍可以问正确问题，
但读者不能直接从马木的问题中读出诈骗方向；
这些问题应被马雯雯的误读包住，
成为“堂哥像在查养生馆违规服务老人”的背景噪音。
```

## 15.5 场景机制偏差

例如：

- 误会机制错误；
- 信息释放过早；
- 场景重心错误；
- 主动权错误。

处理：

```text
重建该场关键人物意图轨道，必要时重写整个场景。
```

## 15.6 上游意图偏差

例如：

- 章节目标理解错误；
- 主次关系错误；
- 情绪曲线错误；
- 章节结尾方向错误。

处理：

```text
回退到章节级意图对齐。
```

# 十六、token 控制要求

## 16.1 第一阶段目标

第一阶段以验证人物意图控制是否能提高 V1 质量为主。

不强行压缩影响质量的必要信息。

## 16.2 第一阶段即时限制

即使在质量验证阶段，也应执行以下限制：

```text
每次人物意图轨道：尽量控制在当前片段必要人物范围内
每次最小验证片段：50—150字以内
每次分歧：1—3个以内
不重复完整章节资料
不输出长篇分析报告
不生成多套完整场景
不重复已确认内容
```

## 16.3 token 评估指标

不能只统计单次调用 token。

核心指标为：

```text
从章节开始到作者认可 V1，
每获得一千字可保留正文所消耗的总 token。
```

# 十七、系统状态机

第一版使用章节级状态机。

```text
CHAPTER_CONTEXT_READY
章节资料已加载

CHAPTER_PLAN_ALIGNED
章节总体方向已确认

RUNTIME_STATE_READY
章节运行态已创建

V1_IN_PROGRESS
整章第一版生成中

INTENT_CHECKPOINT
进入关键片段，显式列人物意图轨道

WAITING_FOR_USER
存在关键人物意图分歧，等待作者确认

FRAGMENT_WRITING
当前正文片段生成中

RUNTIME_STATE_UPDATED
作者确认已转化为规则并写入运行态

V1_ASSEMBLED
整章第一版已完成

V1_REVIEW
作者进行整章审阅

V2_REVISION
根据偏差层级进行修改

AUTHOR_INTENT_REVISION
根据作者对读者体验、视角过滤、详略、留白和文本表现的反馈进行草稿后修订

CHAPTER_CONFIRMED
章节完成
```

状态转换示例：

```text
INTENT_CHECKPOINT
    ├─ CONTINUE → FRAGMENT_WRITING
    └─ ASK_USER → WAITING_FOR_USER

WAITING_FOR_USER
    ↓
作者回答或改写轨道
    ↓
RUNTIME_STATE_UPDATED
    ↓
FRAGMENT_WRITING
```

草稿后作者意图修订状态转换：

```text
V1_REVIEW
    ↓
作者指出正文表现偏差
    ↓
偏差归因为“内容过满 / 视角过滤失败 / 读者误导失败 / AI感 / 留白不足”
    ↓
AUTHOR_INTENT_REVISION
    ↓
基于同一组事实重排展示方式
    ↓
V2_REVISION 或 CHAPTER_CONFIRMED
```

其中：

- `方案/独立意图控制Skill设计稿.md` 作为独立 Skill 本体开发依据；
- `方案/大纲细纲意图控制接入改造方案.md` 作为上游 Skill 改造依据；
- `方案/正文生成Skill改造点.md` 作为正文生成 Skill 改造依据。

旧讨论稿只保留思考过程，不作为最新实施口径。

## 18.3 Skill 主要职责

Skill 负责：

1. 读取章节资料；
2. 建立章节运行态；
3. 执行章节级意图识别；
4. 识别关键片段；
5. 显式列人物意图轨道；
6. 判断人物语言意图分歧；
7. 判断人物行为意图分歧；
8. 输出 CONTINUE 或 ASK_USER；
9. 触发条件中断；
10. 生成最小验证片段；
11. 将作者回答转换为明确规则；
12. 更新章节运行态；
13. 从中断点恢复生成；
14. 完成整章拼接与检查；
15. 对作者反馈进行偏差归因；
16. 选择局部修改、片段重写、场景重写或上游回退。
17. 识别“人物意图正确但正文表现失败”的情况，并将其交给正文输出控制或作者意图修订，而不是错误归因给人物意图控制器。

## 18.4 开发启动顺序

正式开发不应先直接改正文生成 Skill。

第一步应创建独立人物意图控制 Skill：

```text
.codex-tmp/skills/control-character-intent/
```

原因：

```text
意图控制是跨阶段控制层。
如果只放入正文生成 Skill，会导致轨道生成、中断判断、状态恢复和正文写作职责混杂。
```

开发顺序：

```text
1. 创建独立人物意图控制 Skill
2. 改造章节执行包 Skill
3. 改造章节细纲 Skill
4. 改造正文生成 Skill
5. 执行一章完整链路测试
```

# 十九、第一版核心规则

```text
1. 每进入一个关键片段前，必须显式列出相关人物意图轨道。

2. 人物语言意图和人物行为意图是最高优先级。

3. 只要不清楚人物为什么说或为什么做，即使场景目标明确，也必须 ASK_USER。

4. 人物没有采取更简单行动时，必须有认知、情绪、关系、利益或外部压力理由。

5. 作者可以直接改写人物意图轨道。

6. 作者回答后，必须将答案转换为明确规则，判断影响范围，并更新章节运行态。

7. 恢复生成前必须简短复述已确认内容和影响范围。

8. 临时人物意图不得自动写入人物卡。

9. 正文片段必须从人物意图中生成，不能让人物配合大纲说话和行动。

10. 所有关键片段完成后，必须执行整章一致性检查。

11. 人物意图正确不代表正文表现正确。若正文问题是过满、过清楚、视角过滤失败、读者被提前告知、没有留白或 AI 感强，应进入正文表现 / 作者意图修订。

12. 正确线索不一定要被读者当作正确线索。若作者要求读者跟随视角人物误读，正文必须把正确线索翻译成视角人物可误读的生活表层。

13. 执行包里的内容不是正文展示清单。正文生成应根据作者意图决定写满、轻写、背景化、错接或留白。
```

# 二十、第一版验收标准

## 20.1 人物意图控制有效

- 能够识别真正的人物语言意图分歧；
- 能够识别真正的人物行为意图分歧；
- 不会忽略关键分歧继续生成；
- 不会因普通写作问题频繁打断作者。

## 20.2 中断机制有效

- 出现分歧时确实停止正文；
- 问题明确、具体；
- 一次只询问关键问题；
- 提供模型推荐方向；
- 提供最小验证片段；
- 允许作者直接改写人物意图轨道。

## 20.3 恢复机制有效

- 能正确理解作者回答；
- 能将回答转换为规则；
- 能判断影响范围；
- 能更新当前章节运行态；
- 能复述确认后继续；
- 不重复生成前文；
- 不遗忘已确认内容。

## 20.4 第一版质量提升

与传统一次性整章生成相比：

- V1 方向性错误明显减少；
- 人物语言失真明显减少；
- 人物行为无因明显减少；
- 信息提前暴露明显减少；
- 整场或整章重写次数明显减少；
- V1 可保留正文比例明显提高。

## 20.5 token 成本可控

- 增加的对齐 token 小于减少的返工 token；
- 不重复加载无关资料；
- 不生成冗长分析；
- 每千字合格正文 token 成本低于原流程。

```text
3. 完成章节级对齐；
4. 开始整章 V1 生成；

6. 记录每次 CONTINUE / ASK_USER；
```

```text
恢复生成准确性
第一版可保留比例
整章重写比例
每千字合格正文 token 成本
```

# 二十三、最终定义

意图控制系统第一版，是一个：

> 以完整章节为交付单位，以关键片段为输出单位，以人物语言意图和人物行为意图为核心控制对象，通过显式人物意图轨道和风险触发式中断实现动态对齐的交互式正文生成系统。

其核心流程为：

```text
整章理解
→ 关键片段识别
→ 人物意图轨道
→ 语言 / 行为分歧识别
→ 条件中断
→ 作者确认或改写
→ 章节运行态更新
→ 断点续写
→ 整章完成
→ 偏差归因
→ 精修迭代
```

系统的首要目标不是减少单次生成 token，而是减少错误正文、无效正文和整体返工。

最终评价标准不是模型一次写了多少，而是：

> 在作者可接受的交互成本下，第一版正文有多少内容能够直接保留并进入精修。
