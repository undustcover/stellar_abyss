你这次补充后，第一版设计里有一个核心概念需要重新纠正：

> **“整章生成”指最终交付单位是整章，不代表生成过程必须一次完成。**  
> **“场景级对齐”指交互与控制粒度，不代表每个场景都要独立验收、独立形成正式正文。**

第一版设计将“生成单位”和“控制单位”混在了一起，所以才会错误地推导出“逐场景生成、逐场景验收、最后合并整章”的流水线。设计稿目前强调了动态意图识别、短示例探测和纠偏归因，这些方向成立，但将第一版正文拆成场景级成品并不符合作者的真实需求。

# 一、第一版仍然应当整章生成，但生成过程必须可暂停

作者真正需要的流程不是：

```text
场景一生成
→ 作者验收
→ 场景二生成
→ 作者验收
→ 最后拼成一章
```

也不是：

```text
整章需求全部确认
→ 一次性生成整章
→ 发现整体偏差
→ 全章重写
```

而是：

```text
理解整章
→ 建立章节级生成计划
→ 场景一生成前进行意图对齐
→ 继续写场景一
→ 场景二生成前进行意图对齐
→ 继续写场景二
→ 场景三生成前进行意图对齐
→ 继续写场景三
→ 全部完成后交付整章第一版
```

因此，第一版有三个不同层级：

|层级|实际单位|作用|
|---|---|---|
|交付单位|整章|作者最终审阅的是完整章节|
|生成控制单位|场景|每进入一个场景前重新确认意图|
|最小验证单位|示例、关键对白、动作片段|测试模型是否真的理解当前场景|

这三个单位必须分开。

## “整章第一版”不等于“一次回答写完整章”

它更准确的定义应该是：

> **在同一轮章节创作任务中，通过若干次场景级意图校准，连续完成并最终组装出的整章正文。**

从作者视角看，它仍然是第一版整章。

从系统执行视角看，它是一个可以中途暂停、校正、继续生成的过程。

这和传统软件里的“一个事务分阶段执行”很相似：最终提交的是完整结果，但内部可以有多个检查点。

# 二、场景生成前确实应当进行意图对齐

这一点作者的判断是正确的，而且比设计稿中的“先确认场景意图，再生成场景成品”更加准确。

第一版草稿质量差的主要原因，通常不是文笔本身，而是模型在场景转换时自行补全了作者没有表达的内容，例如：

- 自行判断本场景的重点；
    
- 自行决定人物的主动权；
    
- 自行选择信息释放量；
    
- 自行添加心理活动；
    
- 自行加强所谓冲突；
    
- 自行设置笑点；
    
- 自行解释伏笔；
    
- 自行决定详写还是略写。
    

章节开始前，即使已经对齐了整章目标，也无法一次覆盖每个场景的具体表现意图。

因为“章节意图”和“场景意图”并不是上下级重复关系。

例如第三章的章节总意图可能是：

```text
维持三重误会；
完成黄金三章收束；
扩大角色群像；
喜剧外壳下出现暗流；
不揭露诈骗真相。
```

但进入某一个具体场景时，仍然要确认：

```text
这一场谁掌握主动权？
马雯雯此时误会到了哪一步？
读者应该比她多知道多少？
陈科在这一场是可笑，还是可疑？
这一场的喜剧依靠什么产生？
异常只留下细节，还是出现一次明显错位？
本场是主场景还是过渡场景？
```

前者不能替代后者。

所以正确原则是：

> **章节级对齐负责确定方向和边界；场景前对齐负责确定当前怎么写。**

# 三、场景前不是“重新询问所有需求”，而是增量对齐

这里也需要避免走向另一个极端。

如果每个场景开始前，都重新询问：

- 场景目的；
    
- 人物目标；
    
- 人物认知；
    
- 信息差；
    
- 叙事重点；
    
- 喜剧机制；
    
- 情绪变化；
    
- 伏笔；
    
- 禁止内容；
    
- 结尾状态……
    

那么作者会重新承担一遍写细纲的工作，整个流程会非常重。

场景前对齐应采用**增量模式**。

系统已经拥有：

1. 小说长期规则；
    
2. 当前单元大纲；
    
3. 章节细纲；
    
4. 章节总意图；
    
5. 前文已经生成的正文；
    
6. 前面场景刚刚确认的新规则。
    

所以每次只需要寻找当前场景中的**高风险分歧点**。

例如进入第三幕前，系统不应问十几个问题，而应输出：

```text
接下来进入第三幕。根据前文，我准备这样处理：

1. 马雯雯已经不再把陈艾艾当警察，但仍将她理解为会所工作人员；
2. 陈艾艾认为马木已向马雯雯说明真实情况，所以不会主动解释；
3. 两人对话能够继续，是因为双方都省略了自己认为对方已经知道的前提；
4. 读者通过无女迎宾、男工作人员核验名单、身份证登记等细节发现事情不对；
5. 本场仍不出现诈骗、投资、传销等直接指向。

当前最大分歧：
马雯雯是把现场理解为普通高端会所，还是理解为存在灰色业务的会所？

我判断应采用后者。
```

随后附三五句验证性示例。

作者只需指出：

```text
整体正确，但马雯雯不是完全确定，只是开始怀疑。
```

然后就可以继续生成。

这才是高效率的意图对齐。

# 四、第一版的正确执行结构

建议把原设计改为以下流程。

## 阶段一：整章理解

在正式写作前，模型先完成章节级分析，但不需要把全部分析都输出给作者。

内部需要明确：

```text
本章核心作用
本章情绪与节奏曲线
主场景与过渡场景
本章信息释放上限
人物状态变化
关键伏笔
本章结尾钩子
每个场景的独占功能
```

向作者输出的只是精简后的章节理解和真正存在分歧的部分。

作者确认后，进入正文生成。

## 阶段二：场景前检查点

每进入一个新场景，执行一次轻量检查：

```text
继承哪些已确认意图
本场新增什么任务
本场最大的理解风险是什么
模型准备如何表现
用一个极短示例验证
```

作者反馈后，更新当前章节的临时意图状态。

## 阶段三：连续生成正文

场景得到确认后，直接生成该部分正文。

但这时不要求作者对该场景进行完整验收，也不宣布“场景冻结”。

因为后续场景可能反过来影响前面的节奏、铺垫和详略。

生成完当前场景后，系统进入下一个场景检查点。

## 阶段四：完成整章第一版

所有场景写完后，将正文整合为完整章节，执行一次章节级检查：

- 场景之间是否真正连贯；
    
- 语气是否发生断裂；
    
- 前后是否重复表达；
    
- 主次是否清楚；
    
- 是否每个场景都写得过满；
    
- 笑点密度是否均匀得不自然；
    
- 信息释放是否超标；
    
- 结尾是否达到章节钩子；
    
- 前面为了局部正确而影响了整体节奏。
    

完成这些检查后，才形成真正的：

```text
整章正文草稿 V1
```

所以场景生成出来的局部文字，在整章完成前都只能叫：

```text
V1生成中的当前正文
```

不能叫独立的“场景第一版成品”。

# 五、执行中不断对齐，不等于无限打断

作者强调“不断对齐、不断修正”，这是正确的，但“不断”不能理解为每写几百字就询问一次。

否则会产生两个问题：

1. 叙事连续性被破坏；
    
2. 作者的审阅负担比自己写还大。
    

应当采用**风险触发式对齐**，而不是机械的固定频率对齐。

必须对齐的节点主要有四类：

### 1. 场景转换

人物、地点、时间、冲突机制发生变化时。

### 2. 高风险表达前

例如：

- 关键误会即将升级；
    
- 伏笔即将出现；
    
- 新角色即将立住；
    
- 强笑点即将落地；
    
- 重要信息即将释放；
    
- 人物态度即将转变。
    

### 3. 模型存在两个以上合理理解时

不能私自选择，应当展示差异。

### 4. 前文生成结果改变了后续设计时

例如原计划中的一个笑点，已经被前文提前使用；或者前一个场景比预想更沉，后面需要调整。

不需要对齐的情况包括：

- 普通过渡动作；
    
- 已明确的连续对话；
    
- 单纯执行已确认的表现方式；
    
- 不涉及信息、人物和节奏变化的连接段。
    

因此，系统规则不应写成：

```text
每个场景生成前必须向作者提问。
```

而应写成：

```text
每个场景生成前必须执行意图检查。
只有发现未确认的高风险分歧时，才向作者发起对齐。
```

“执行检查”和“向作者提问”不是一回事。

如果当前意图已经非常明确，可以简短报告：

```text
下一场沿用已确认方向，不存在新的关键分歧，我将继续生成。
```

甚至在成熟阶段可以直接继续，不必人为增加交互。

# 六、第二版为什么不应再承担全章重写

按照这个设计，第一版的目标必须提高。

设计稿中将第一版定义为“尽可能生成可用正文，同时暴露理解问题”，这个表述偏弱。

因为作者不是为了用第一版探雷，再让第二版正式重写。

第一版应该定义为：

> **在动态意图对齐基础上，生成结构、人物认知、信息控制和主要表现方向基本正确的完整章节。**

第二版主要解决：

- 局部对话不够自然；
    
- 某些段落节奏拖沓；
    
- 详略仍需调整；
    
- 笑点落点不够准确；
    
- 某些旁白解释过多；
    
- 个别人物语言不够稳定；
    
- 场景衔接存在痕迹；
    
- 全章读感不够统一。
    

也就是说：

```text
V1解决“写对”
V2解决“写好”
V3解决“打磨”
```

当然，现实中V1仍可能存在机制性错误，但系统设计不能把“第一版大概率失败”当成正常目标。

一旦第一版需要全章推倒重写，意味着场景前意图对齐机制没有发挥作用。

# 七、token问题不能等到质量完成后才开始考虑

关于第三个问题，不能简单选择其中一端：

- 不是“第一版就把token压到最低”；
    
- 也不是“完全不考虑token，等质量成熟后再优化”。
    

最合理的策略是：

> **第一阶段以质量为硬目标，同时从第一版设计开始控制明显的无效token；待流程验证有效后，再优化必要token。**

这里要区分两种token。

## 1. 有效token

这些token直接提高正文质量：

- 章节必要背景；
    
- 当前人物状态；
    
- 信息差；
    
- 当前场景关键意图；
    
- 极短验证示例；
    
- 作者反馈；
    
- 已确认的新增规则；
    
- 当前连续正文。
    

这些不能过早压缩。

## 2. 无效token

这些应从第一版就删除：

- 每次重复整章大纲；
    
- 每次重复全部人物卡；
    
- 每次重复完整方法论；
    
- 模型长篇解释自己为什么这样理解；
    
- 为每个场景输出庞大意图表；
    
- 已经确认的内容再次询问；
    
- 同一场景生成三套完整方案；
    
- 每次都把历史纠偏记录全文塞入上下文；
    
- 用大量抽象术语代替一个短示例；
    
- 第一版正文完成后，又重复输出逐段总结。
    

所以正确路线是：

```text
质量机制先完整保留
+
从一开始清除确定无用的token
+
完成真实章节测试
+
统计哪些上下文真正影响质量
+
再压缩有效信息的表示方式
```

# 八、建议采用“三阶段token策略”

## 阶段A：机制验证期

目标：证明动态对齐能显著提高第一版质量。

此时：

- 不限制必要的对齐轮次；
    
- 不压缩关键上下文；
    
- 允许保留短示例；
    
- 保留场景间临时记忆；
    
- 记录每次对齐解决了什么问题。
    

但仍要限制明显膨胀：

- 每次场景理解控制在200—400字；
    
- 验证示例通常控制在50—150字；
    
- 每次最多提出1—3个真正分歧；
    
- 不输出完整分析报告；
    
- 不重复作者已经明确的规则。
    

## 阶段B：流程稳定期

目标：找到哪些对齐是必要的，哪些可以由系统自行判断。

开始统计：

```text
哪些问题作者经常修正？
哪些问题模型已经稳定掌握？
哪些场景必须对齐？
哪些场景可以自动继续？
哪些资料从未真正影响生成？
哪些纠偏规则可以转成长期规则？
```

此时可以把对齐从“每场景一次”优化为“高风险场景触发”。

## 阶段C：token优化期

目标：在不降低第一版合格率的前提下压缩成本。

主要手段：

- 将完整资料压缩成章节运行态；
    
- 只加载当前角色和当前伏笔；
    
- 历史正文采用最近窗口加结构摘要；
    
- 已确认规则用短标签调用；
    
- 相同类型的意图不重复解释；
    
- 低风险场景自动生成；
    
- 只有歧义时才生成对比示例；
    
- 将纠偏记录分为本章临时规则和长期规则；
    
- 章节完成后清理无复用价值的过程信息。
    

这里的评估指标不能只是token总量，而应当是：

```text
每千字合格正文消耗的总token
```

甚至更准确地说：

```text
从章节开始到作者认可V1，
每获得一千字可保留正文所消耗的token。
```

因为一次生成虽然token少，但如果整章废掉重写，实际成本反而更高。

# 九、动态对齐可能比一次生成更节省token

表面上，多轮询问会增加token。

但必须计算返工成本。

假设一次性生成一章：

```text
输入上下文：12k
正文输出：5k
作者反馈：2k
整章重写输入：15k
第二版输出：5k
再次修改：若干
```

总成本可能远高于：

```text
章节理解：3k
场景一对齐：0.5k
生成：1.5k
场景二对齐：0.5k
生成：1.5k
场景三对齐：0.5k
生成：2k
整章调整：2k
```

动态对齐增加了前置成本，但减少了废稿。

因此不能只看：

```text
一次调用用了多少token
```

要看：

```text
获得合格章节之前，总共浪费了多少token。
```

作者当前遇到的核心问题不是单次生成太贵，而是：

> **便宜地生成了一整章错误正文，然后昂贵地全部重写。**

从这个角度看，场景前对齐本身就是一种token优化，只不过它优化的是返工token，而不是单次调用token。

# 十、对第一版设计的具体修正

原设计中以下内容应保留：

- 整章开始前建立章节控制；
    
- 模型主动识别关键歧义；
    
- 用极短示例验证理解；
    
- 作者通过选择、否定、修正表达意图；
    
- 纠偏前先归因；
    
- 有价值的反馈继承到后文；
    
- 不要求作者填写复杂表格。
    

以下内容需要修改：

## 1. 删除“场景逐个验收后进入下一场景”

改为：

```text
场景前对齐
→ 连续生成
→ 必要时即时纠偏
→ 进入下一场景
→ 全章完成后统一验收V1
```

## 2. 删除“场景意图全部确认后才生成正文”

改为：

```text
只确认当前生成所必需的意图；
未出现的后续问题留到后续场景再确认。
```

## 3. 删除“第一版主要用于暴露问题”的弱目标

改为：

```text
第一版必须保证结构和意图基本正确，
其主要功能是形成可继续精修的完整章节。
```

## 4. 删除固定的场景状态机

原来的：

```text
SCENE_INTENT_PROBING
SCENE_INTENT_CONFIRMED
DRAFT_V1_GENERATED
SCENE_CONFIRMED
NEXT_SCENE
```

容易把流程做成逐场景交付。

更合适的是章节级状态：

```text
CHAPTER_CONTEXT_READY
章节资料已加载

CHAPTER_PLAN_ALIGNED
章节总体方向已确认

V1_IN_PROGRESS
整章第一版生成中

SCENE_CHECKPOINT
进入新场景，执行意图检查

SCENE_WRITING
当前场景连续生成

MIDCOURSE_CORRECTION
生成过程中发生必要纠偏

V1_ASSEMBLED
整章第一版已完成

V1_REVIEW
作者从整章角度审阅

V2_REVISION
根据问题层级进行第二版修改

CHAPTER_CONFIRMED
章节通过
```

其中 `SCENE_CHECKPOINT` 是章节生成中的临时子状态，而不是独立生产流程。

# 十一、最终应确定的新范式

可以把整个方法正式定义为：

> **整章交付、场景控制、动态对齐、连续生成。**

完整链路为：

```text
读取章节资料
    ↓
理解整章目的、结构、节奏与信息边界
    ↓
确认章节级关键分歧
    ↓
开始生成整章V1
    ↓
进入场景一前执行意图检查
    ├─ 无关键分歧：直接生成
    └─ 有关键分歧：短示例对齐后生成
    ↓
进入场景二前重新检查
    ├─ 继承前文已确认意图
    ├─ 根据已生成正文修正后续计划
    └─ 对齐本场新增的高风险问题
    ↓
继续完成所有场景
    ↓
整章统一检查与衔接调整
    ↓
输出完整正文V1
    ↓
作者从整章角度反馈
    ↓
偏差归因
    ├─ 局部文字问题：局部修改
    ├─ 场景表现问题：重写该场景
    ├─ 章节节奏问题：重新分配详略
    └─ 上游意图问题：回退重新对齐
    ↓
形成整章V2
```

这套流程解决的是当前最核心的矛盾：

- 保留整章生成，避免场景拼接感；
    
- 在执行过程中持续校正，防止第一版方向性失败；
    
- 不要求预先穷尽全部需求；
    
- 不让第二版承担全章重写；
    
- 增加少量对齐token，换取大幅减少废稿和返工token。
    

因此，关于token的最终结论是：

> **第一阶段应当质量优先，但不是token失控。先完整验证所有真正影响质量的机制，同时立即删除明确无效的重复信息；等第一版合格率稳定后，再对必要上下文和对齐频率进行系统压缩。**

不能先为了省token削弱意图对齐，因为这会重新回到“整章生成错误、整章重写”的旧问题。也不能完全不记录token，否则后续无法判断哪些机制成本过高。最合理的是从第一次测试开始，同时记录质量和token，但只以质量作为当前阶段的淘汰标准。
