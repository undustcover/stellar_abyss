

> **Skill可以规定：检测到关键分歧时，立即停止本轮输出，向作者提出问题；作者回答后，再从中断点继续生成。**

但它不能在一次已经开始的长文本输出中，像程序断点一样写到第八百字突然冻结、等待用户回答，然后在同一轮响应里恢复。

因此，真正可实现的动态对齐不是“单次调用内部暂停”，而是：

```text
生成前检查
→ 判断是否存在关键分歧
→ 有分歧：结束当前轮次并提问
→ 用户回答
→ 更新意图状态
→ 下一轮继续生成
```

也就是说，Skill可以实现的是**会话轮次级中断**，不是单次推理内部的同步中断。

目前《意图控制 v1.0》已经提出了场景检查点、短示例探测和高风险分歧识别，但还需要把“什么时候必须停止、停止后输出什么、作者回答后如何恢复”正式写成执行协议。

# 一、动态对齐的核心不是“多问”，而是“条件中断”

动态对齐不能写成：

```text
每进入一个场景，都向作者提问。
```

这样会导致：

- 作者频繁被打断；
    
- 大量问题其实没有必要；
    
- 对话成本和token成本过高；
    
- 章节生成失去连续性；
    
- Skill沦为机械问卷。
    

正确设计应该是：

```text
每进入一个场景，都执行分歧检查；
只有出现影响正文方向的关键分歧时，才中断并提问。
```

因此需要区分两个概念。

## 场景检查

每个场景都必须执行，但可以由模型内部完成。

## 用户中断

只有发现模型无法安全自行判断的高风险分歧时才触发。

动态对齐本质上是一个：

> **以风险判断为触发条件的交互式生成机制。**

# 二、Skill能够做到什么，不能做到什么

## Skill能够做到

Skill可以明确规定：

1. 每个场景生成前先检查意图；
    
2. 发现关键歧义时不得自行选择；
    
3. 不得继续生成后续正文；
    
4. 必须输出分歧点、模型判断和极短示例；
    
5. 等待作者回答；
    
6. 作者回答后更新当前章节意图；
    
7. 从中断点继续，而不是重写前文。
    

例如：

```text
检测到当前场景存在关键分歧：
马雯雯是将现场理解为普通高端会所，
还是怀疑这里存在不能见光的业务。

两种理解会直接影响她后续对白和观察重点，
不能由模型自行决定。

模型建议采用第二种，但保持“怀疑而非确认”。

示例：
……

请选择：
A. 普通高端会所
B. 怀疑存在灰色业务
C. 其他解释
```

输出到这里就结束。

作者回答后，再继续正文。

## Skill不能稳定做到

Skill不能保证：

```text
在一次连续生成整章正文的单个响应里，
写到某个位置后真正暂停，
等待用户输入，
然后在同一响应中恢复。
```

因为模型一次响应一旦开始，就必须在当前轮次内输出完毕或停止。

所以“整章生成”在技术上不能理解为：

```text
一次API调用输出整章。
```

而应理解为：

```text
一个完整的章节生成任务，
可以跨越多个会话轮次，
最终交付整章V1。
```

# 三、真正的执行结构：场景检查点

整章生成过程应被划分为若干逻辑检查点：

```text
章节计划确认
    ↓
Scene 1 Checkpoint
    ├─ 无关键分歧：生成场景一
    └─ 有关键分歧：提问并停止
    ↓
Scene 2 Checkpoint
    ├─ 无关键分歧：生成场景二
    └─ 有关键分歧：提问并停止
    ↓
Scene 3 Checkpoint
    ...
    ↓
整章组装与检查
    ↓
输出完整V1
```

这里的关键是：

> **模型不是先把整章写出来，再检查有没有分歧，而是在准备进入每个场景时先执行检查。**

否则等写完才发现分歧，动态对齐就失去意义了。

# 四、场景分歧必须有明确的判定标准

如果只在Skill里写：

```text
如果出现分歧，就向作者提问。
```

这条规则几乎无法执行。

因为模型不知道什么叫“分歧”，最终会出现两种极端：

- 什么都不问，继续自行补全；
    
- 什么都问，导致流程严重臃肿。
    

所以必须定义“关键分歧”。

建议将场景分歧分为四级。

## L0：无分歧

现有资料足够明确，且只有一种合理写法。

处理：

```text
直接继续生成。
```

例如：

- 已确认人物从大厅进入包间；
    
- 只是补充动作和衔接；
    
- 对话目标已经明确；
    
- 不涉及重要信息释放。
    

## L1：表现型分歧

不同选择只影响措辞或轻微表现，不改变场景机制。

例如：

- 人物是冷笑还是皱眉；
    
- 某句对白用反问还是陈述；
    
- 过渡段略写多少。
    

处理：

```text
模型自行选择，不询问作者。
```

这类问题必须授权模型判断，否则作者会被大量低价值问题淹没。

## L2：重要表现分歧

选择会影响人物状态、喜剧强度、场景重点，但不改变主线。

例如：

- 马雯雯是完全没发现异常，还是发现异常但理解错误；
    
- 一个笑点是通过对白落地，还是通过动作落地；
    
- 本场重心偏喜剧，还是偏暗流。
    

处理：

```text
模型优先根据已有规则判断。
只有缺乏足够依据或两个方向都合理时，才中断。
```

## L3：机制性分歧

选择会改变以下任一内容：

- 人物认知；
    
- 人物目标；
    
- 场景主动权；
    
- 信息差；
    
- 误会机制；
    
- 伏笔强度；
    
- 信息释放时机；
    
- 主次结构；
    
- 情节因果；
    
- 人物关系变化；
    
- 章节结尾状态。
    

处理：

```text
必须中断，不得自行决定。
```

这才是真正需要动态对齐的部分。

# 五、建议定义“强制中断条件”

可以直接在Skill里列出以下条件。

出现任一条件时，模型必须停止正文生成并向作者提问。

## 1. 存在两个以上合理解释

例如同一段细纲既可以理解为：

- 马雯雯已经怀疑陈艾艾；
    
- 马雯雯只是觉得她说话奇怪。
    

两种都符合现有文本，却会导致不同对白。

必须中断。

## 2. 当前选择会改变人物认知

尤其是误会喜剧中：

```text
人物知道什么
人物不知道什么
人物误以为什么
```

只要不明确，就不能继续。

## 3. 当前选择会改变读者信息量

例如：

- 是否让读者明确发现诈骗；
    
- 是否只留下异常细节；
    
- 是否提前解释身份证用途；
    
- 是否明确某角色在撒谎。
    

必须中断。

## 4. 当前选择可能违反已确认规则

例如：

- 可能破坏马雯雯POV；
    
- 可能出现设定解释型对白；
    
- 可能让陈科说出“圈子、资格、阶层”等破坏误导的词；
    
- 可能让读者提前知道作者不希望公开的信息。
    

必须中断。

## 5. 场景细纲只能说明“发生什么”，不能说明“怎么表现”

而不同表现方式会明显影响场景效果。

必须中断或至少用短示例验证。

## 6. 前文实际生成结果改变了原计划

例如：

- 前面已提前释放某条信息；
    
- 前面的笑点过重，后面需要降温；
    
- 前一个场景比预期更长；
    
- 人物已经表现出某种态度，后面不能重复建立。
    

必须重新对齐后续场景。

# 六、什么情况禁止提问

同样要在Skill里写出“禁止中断条件”，否则模型很容易过度谨慎。

以下情况不能向作者提问：

- 只是词语选择；
    
- 只是一般动作；
    
- 只是过渡衔接；
    
- 已有规则能够明确推导；
    
- 作者此前已经回答过；
    
- 可以通过人物卡、章节意图或前文唯一确定；
    
- 两种写法差异很小，不影响后续；
    
- 只是模型担心自己写不好，而不是存在真实意图分歧。
    

特别要明确：

> **写作难度高，不等于意图存在分歧。**

模型不能因为“不确定怎么写得好”就把创作责任重新抛给作者。

例如：

```text
这里怎么写得更幽默？
```

不是一个合格的问题。

模型应该自己完成表现设计。

只有当幽默机制本身存在两个不同方向时，才需要询问：

```text
这里的强笑点究竟应来自马雯雯的误判，
还是来自陈科过度自信后的反转？
```

# 七、提问前必须先做判断，不能只抛开放式问题

错误提问：

```text
这个场景你想怎么写？
```

这等于要求作者重新写一遍场景设计。

正确提问应该包含四部分。

## 1. 已确认内容

说明当前没有争议的部分。

## 2. 唯一分歧点

只提出真正阻止生成的问题。

## 3. 模型判断

模型必须先给出建议，不能把所有责任推给作者。

## 4. 最小示例

通过三五句正文证明自己理解的差异。

标准格式可以是：

```markdown
## 场景中断：需要确认一个关键分歧

### 已确认

- 陈艾艾认为马木已经说明情况；
- 马雯雯不知道真实情况；
- 双方对话依靠前提省略继续；
- 不能直接暴露诈骗。

### 当前分歧

马雯雯看到现场异常后，应当处于：

A. 没有意识到异常，只觉得这里管理严格；

B. 意识到不对，但误判为存在灰色业务。

### 我的判断

建议采用B。

原因：这样可以保持马雯雯的观察能力，同时让她得出错误结论，误会更合理。

### 最小示例

……

请确认A、B，或指出需要调整的差异。
```

提问完立即停止。

不得在问题后继续写正文。

# 八、作者回答后如何恢复

动态对齐最容易失败的地方，不是提问，而是恢复生成。

作者回答后，模型必须完成四个动作。

## 1. 将回答转成明确规则

例如作者说：

> 选B，但她不是确定，只是有点怀疑。

模型内部应更新为：

```text
马雯雯察觉异常；
她怀疑现场有灰色业务；
怀疑强度较低；
她尚未形成确定判断；
后续对白不能直接表现为笃定。
```

不能只记住“选B”。

## 2. 确认影响范围

判断这条回答影响：

- 仅当前几句；
    
- 当前整个场景；
    
- 本章后续场景；
    
- 长期人物认知规则。
    

## 3. 更新章节运行态

把新规则放入当前章节的临时状态，而不是每次重复读取整轮对话。

## 4. 从断点继续

恢复时不能：

- 重复输出此前正文；
    
- 重新解释一遍整个场景；
    
- 从章节开头重写；
    
- 忘记已生成内容；
    
- 擅自修改已经确认的前文。
    

正确恢复形式：

```text
已确认：马雯雯察觉异常，但只形成轻微怀疑，误判方向是灰色业务。

继续第三幕正文：
……
```

# 九、Skill需要保存一个“章节运行态”

单靠自然语言聊天记录，短期测试可以运行，但随着章节变长，模型可能忘记：

- 当前写到哪里；
    
- 哪些规则已确认；
    
- 哪些正文已经生成；
    
- 哪些分歧已解决；
    
- 后续计划是否被调整。
    

所以Skill中最好规定一个轻量运行态。

例如：

```yaml
chapter_state:
  chapter_id: C03
  status: V1_IN_PROGRESS
  current_scene: scene_03
  current_position: before_dialogue_round_2

confirmed_intent:
  chapter_goal:
    - 维持三重误会
    - 不揭露诈骗
    - 完成黄金三章收束

  scene_goal:
    - 马雯雯进入现场
    - 通过细节察觉异常
    - 将异常误判为灰色业务

  character_knowledge:
    马雯雯:
      knows:
        - 马木来过这里
        - 身份证在现场有用途
      does_not_know:
        - 陈科正在进入诈骗局
      misbelief:
        - 现场可能经营不能明说的业务

  forbidden:
    - 直接出现诈骗判断
    - 陈艾艾完整解释背景
    - 旁白说明双方都误会

resolved_divergences:
  - id: D03-01
    decision: 马雯雯察觉异常，但只是轻度怀疑

pending:
  - scene_03正文
```

不一定每次都向作者展示这份状态，但系统应保留。

# 十、纯Skill实现与程序化实现的区别

## 方案一：纯Skill实现

通过提示词和会话规则完成。

流程：

```text
模型检查
→ 发现分歧
→ 输出问题并结束
→ 作者回答
→ 模型继续
```

优点：

- 现在就可以测试；
    
- 不需要开发GUI；
    
- 适合验证方法论；
    
- 修改规则方便。
    

缺点：

- 中断判断依赖模型自觉；
    
- 可能忘记状态；
    
- 有时会继续多写几段；
    
- 长章节中上下文管理较弱；
    
- 无法真正程序化保证状态迁移。
    

现阶段应先采用这个方案。

## 方案二：Skill加结构化状态

要求模型每轮先输出内部决策：

```json
{
  "action": "ASK_USER",
  "risk_level": "L3",
  "divergence": "马雯雯对现场异常的理解",
  "question": "...",
  "resume_point": "scene_03_before_entry_dialogue"
}
```

外层系统读取：

- `ASK_USER`：展示问题，不调用正文生成；
    
- `CONTINUE`：继续生成；
    
- `ASSEMBLE`：组装整章；
    
- `REVISE`：进入修改。
    

这已经需要一个轻量Agent或脚本作为编排器。

优点：

- 真正可靠；
    
- 可以强制停止；
    
- 状态清楚；
    
- 便于记录token与版本。
    

## 方案三：完整GUI工作流

GUI负责：

- 展示当前章节状态；
    
- 分屏显示正文和问题；
    
- 作者选择分歧方案；
    
- 保存各场景正文；
    
- 比较版本；
    
- 自动拼接整章。
    

这是后期产品化方向，现在没有必要先做。

# 十一、推荐先采用“两段式Skill”

为了提高纯Skill模式的可靠性，可以把一次场景生成拆成两个逻辑步骤。

## 第一步：场景预检

Skill只执行：

```text
读取当前状态
→ 检查分歧
→ 输出CONTINUE或ASK_USER
```

如果返回：

```text
CONTINUE
```

才进入第二步。

如果返回：

```text
ASK_USER
```

则停止。

## 第二步：正文生成

只根据已经确认的场景意图写正文。

这样比让模型在同一次推理中：

```text
一边检查
→ 一边生成
→ 生成途中再决定是否提问
```

可靠得多。

实际会话中可以表现为：

```text
【场景预检结果】
不存在需要作者确认的机制性分歧，继续生成。

【正文】
……
```

但成熟后，“不存在分歧”的预检结果不必每次展示，可以内部完成。

# 十二、动态对齐的最小可行版本

第一版不要设计得过于复杂。可以先规定五条核心规则：

```text
1. 每进入一个新场景前，先检查人物认知、信息释放、场景目标和表现机制是否存在关键歧义。

2. 若不同选择会改变人物认知、信息差、因果、喜剧机制、伏笔或章节节奏，则视为关键分歧。

3. 出现关键分歧时，不得自行选择，不得继续生成正文；必须输出：
   - 已确认内容；
   - 唯一关键分歧；
   - 推荐方向；
   - 极短示例。

4. 作者回答后，将答案转化为明确的场景规则，更新章节运行态，并从中断点继续。

5. 不得询问措辞、普通动作、轻微表现差异，以及能够从既有资料唯一推导的问题。
```

仅靠这五条，就可以开始进行真实测试。

# 十三、最关键的设计结论

动态对齐不是依靠一句：

```text
有问题就问作者。
```

它必须由四个模块组成：

```text
场景检查点
+
分歧分级
+
条件中断协议
+
恢复生成状态
```

缺少其中任何一个都会失效：

- 没有检查点：模型写完才发现问题；
    
- 没有分级：要么不问，要么什么都问；
    
- 没有中断协议：模型提问后仍继续写；
    
- 没有恢复状态：作者回答后又从头生成。
    

因此，Skill层面完全可以先实现动态对齐，但其准确表述应当是：

> **通过场景检查点和条件中断，让整章生成任务跨多个会话轮次连续完成。**

而不是：

> 在一次整章输出中随时暂停和恢复。

第一阶段先用纯Skill验证“分歧识别是否准确、提问是否真正有价值、是否减少整章返工”。如果这三个指标成立，再把 `ASK_USER / CONTINUE / RESUME` 做成程序化状态机。
