# 小说写作系统架构需求文档 V1.0

## 一、系统目标

本系统用于搭建一套面向长篇小说创作的结构化写作系统。

系统以 Obsidian 作为主要资料承载平台，通过双链、标签、属性、看板、文件夹结构等方式，对小说创作中的叙事内容、设定内容、事件内容和 Skill 规则进行模块化管理。

系统最终目标是：在 Skill 系统的控制下，调用设定系统与事件数据库中的资料，支撑叙事系统完成从总纲、卷纲、单元纲、章纲、细纲到正文的生产。

---

## 二、系统总体架构

小说写作系统分为四大核心部分：

```text
01_叙事系统
02_设定系统
03_事件数据库
04_Skill系统
```

四大系统的核心定位如下：

|系统|核心定位|主要作用|
|---|---|---|
|叙事系统|正文生产线|面向读者，负责大纲、章纲、细纲、正文等内容生产|
|设定系统|世界规则库|记录人物、世界观、组织、道具、能力、种族、地区等硬规定|
|事件数据库|世界时间轴事实库|记录小说世界中真实发生的大事件、历史阶段、人物经历和组织行动|
|Skill系统|LLM创作控制系统|定义 LLM 如何调用资料、组合资料、校验资料和生成高质量文本|

四大系统的逻辑关系为：

```text
设定系统 + 事件数据库
        ↓
Skill系统调用、加工、校验
        ↓
叙事系统生成大纲、章纲、细纲、正文
```

---

## 三、核心设计原则

### 1. 模块化原则

四大系统相对独立，各自承担不同职责。

设定系统负责规则，事件数据库负责事实，叙事系统负责讲法，Skill系统负责调用与生成。

任一模块可以单独修改、扩展和维护，但不得混淆职责。

---

### 2. 世界时间轴原则

系统底层以世界时间轴为基础。

事件数据库、人物经历、组织行动、历史事件等内容，必须使用小说世界内部时间进行定义。

例如：

```text
2026年5月—2026年6月
十几年前
灾变后第3年
星历XXX年
六合集市开放前7天
```

事件数据库不得使用 P0、C1、B1 等叙事编号作为主要时间定义。

---

### 3. 叙事结构编号原则

叙事系统可以使用 B / P / C 等小说结构编号。

例如：

```text
B0：前置部
P0：命运的馈赠
C01：我那一事无成的堂哥
C02：女儿先慌了
```

B：定义小说第几部
V：定义小说第几卷
P：定义小说第几篇
C：定义小说第几章

---

### 4. 双时间轴原则

系统中必须区分两种时间轴：

```text
世界时间轴：事情在小说世界中真实发生的顺序
叙事结构轴：读者在小说中看到这些事情的顺序
```

世界时间轴服务于逻辑一致性。

叙事结构轴服务于读者体验。

二者可以不一致。倒叙、插叙、伏笔、误导、解密、反转、回收等写作手法，均依赖二者的分离。

---

### 5. 大事件颗粒度原则

事件数据库只记录故事级大事件，不拆分章节，不设计场景，不安排爽点，不承担章纲和细纲职责。

例如，事件数据库可以记录：

```text
事件：《陈科被高端分享局诈骗事件》
时间：2026年5月—2026年6月
```

但不应细分为：

```text
陈科接电话
陈科去会所
马木赶到会场
陈科填写登记表
任战设置门槛
陈科反咬马木
```

上述内容属于叙事系统中的大纲、章纲或细纲设计，不属于事件数据库的基础记录范围。

---

### 6. Skill服务LLM原则

Skill系统不是资料库，而是面向 LLM 的创作控制系统。

Skill系统负责定义：

```text
LLM应该调用哪些资料
LLM应该如何理解资料
LLM应该如何组合资料
LLM应该如何校验资料
LLM应该如何生成大纲、章纲、细纲和正文
LLM应该如何保证正文质量、风格和去AI化
```

---

## 四、叙事系统

### 1. 系统定位

叙事系统是正文生产线。

它面向读者体验，负责在 Skill 系统控制下，将设定系统和事件数据库提供的资料，加工为小说的总纲、卷纲、单元纲、章纲、细纲、正文和修订稿。

叙事系统使用 B / P / C 等小说结构编号。

---

### 2. 核心职责

叙事系统负责：

```text
总纲设计
分部设计
分卷设计
单元设计
章节设计
场景设计
正文生成
正文修订
信息释放控制
伏笔埋设与回收
悬念设计
反转设计
情绪曲线设计
读者爽点设计
章节节奏控制
POV安排
```

---

### 3. 内容层级

叙事系统建议包含以下层级：

```text
叙事系统
├── 全书总纲
├── 分部大纲
├── 分卷大纲
├── 单元大纲
├── 章节大纲
├── 章节细纲
├── 正文
├── 修订稿
└── 废稿/版本记录
```

---

### 4. 工作重点

叙事系统关注的是：

```text
读者先看到什么，后看到什么
哪些事实提前展示
哪些事实暂时隐藏
哪些内容用于制造误导
哪些内容用于埋伏笔
哪些内容用于后续解密
每章提供什么读者奖励
每章如何形成推进感
每章如何结尾留钩子
```

叙事系统不直接承担世界事实记录职责。

---

## 五、设定系统

### 1. 系统定位

设定系统是世界规则库。

它负责记录小说中的人物、世界观、组织、道具、能力、种族、地区、术语等硬规定，为事件数据库和叙事系统提供规则约束。

---

### 2. 核心职责

设定系统负责：

```text
定义人物设定
定义世界观规则
定义能力边界
定义组织结构
定义道具规则
定义种族文明
定义地区场所
定义术语体系
定义信息暴露限制
定义禁用设定
定义设定冲突校验规则
```

---

### 3. 建议分类

设定系统建议包含：

```text
设定系统
├── 世界观
├── 人物
├── 组织/派系
├── 地区/场所
├── 道具/技术
├── 种族/文明
├── 能力系统
├── 术语表
├── 信息暴露规则
└── 禁用设定/冲突记录
```

---

### 4. 工作重点

设定系统关注的是：

```text
人物是否崩坏
能力是否越界
世界规则是否冲突
组织逻辑是否成立
术语是否统一
某个设定是否可以在当前阶段暴露
某个角色是否应该知道某条信息
```

设定系统不负责章节拆分，也不负责正文节奏设计。

---

## 六、事件数据库

### 1. 系统定位

事件数据库是基于世界时间轴的故事事实库。

它记录小说世界中真实发生的大事件、历史阶段、人物经历和组织行动。

事件数据库使用世界内时间定义，不使用 B / P / C 等叙事编号作为核心时间标识。

---

### 2. 核心职责

事件数据库负责：

```text
记录世界大事件
记录历史阶段
记录人物经历
记录组织行动
记录事件时间跨度
记录事件起因
记录事件结果
记录事件影响
记录事件涉及的人物、组织、地点和设定
维护世界时间轴
维护人物状态逻辑
维护组织状态逻辑
维护因果关系
防止故事事实冲突
```

---

### 3. 事件颗粒度

事件数据库采用故事级大事件颗粒度。

事件数据库记录的是：

```text
一段完整剧情事件
一段人物经历
一次组织行动
一次历史转折
一个世界阶段
```

不记录：

```text
章节节点
场景节点
对话安排
爽点设计
反转设计
悬念设计
章节结尾钩子
```

---

### 4. 示例事件颗粒度

正确颗粒度示例：

```text
事件：《马木当年合伙人捐款跑路事件》
时间：十几年前
性质：主角前史事件
影响：导致马木从人生巅峰跌落谷底，形成对“天降机会”的深层不信任。
```

```text
事件：《陈科被高端分享局诈骗事件》
时间：2026年5月—2026年6月
性质：现实诈骗事件 / 主角低谷事件
影响：陈科被诱导入局，马木介入失败，最终被反咬，触发断卡与开除。
```

```text
事件：《张凡与小满分离事件》
时间：2026年6月
性质：现实拯救线事件
影响：张富贵为孩子筹钱卖猫，张凡情绪崩落，形成马木“救不了所有人”的现实挫败。
```

---

### 5. 事件数据库字段建议

事件数据库中的大事件卡可采用以下字段：

```yaml
id: EVT-0001
title: 陈科被高端分享局诈骗事件
event_type: 现实诈骗事件
time_span: 2026年5月—2026年6月
status: 已发生
related_characters:
  - 陈科
  - 陈艾艾
  - 马木
  - 任战
related_organizations:
  - 高端分享局
related_locations:
  - 会所
related_settings:
  - 现实骗局机制
  - 任战能力
summary: >
  陈科被包装成高端圈层机会的分享局吸引，马木介入阻止失败，最终被陈科反咬，导致马木断卡与被开除。
cause: >
  陈科存在体面焦虑和机会焦虑，任战利用其心理弱点进行诱导。
result: >
  陈科没有彻底脱离骗局，反而成为马木被牵连的关键人物。
impact: >
  推动马木现实生活崩塌，并形成其后续对“系统馈赠”的强烈不信任。
logic_constraints:
  - 马木此时不知道灵能存在
  - 陈科不是纯粹恶人，而是焦虑、自我合理化后反咬
  - 任战不能被普通证据轻易击倒
```

## 七、Skill系统

### 1. 系统定位

Skill系统是面向 LLM 的创作控制系统。

它负责定义 LLM 如何调用、筛选、组合、加工和校验资料，以及如何生成高质量文本。

Skill系统不是资料库，不存放具体小说内容，而是存放 LLM 使用资料和生成内容的方法。

---

### 2. 核心职责

Skill系统负责：

```text
定义资料调用规则
定义资料优先级
定义资料冲突处理规则
定义大纲生成流程
定义章纲生成流程
定义细纲生成流程
定义正文生成流程
定义正文修订流程
定义设定校验规则
定义事件校验规则
定义人设校验规则
定义信息释放规则
定义文风规则
定义去AI化规则
定义章节节奏规则
定义禁用写法
```

---

### 3. Skill系统内部分类

Skill系统分为两大类：

```text
Skill系统
├── 资料调用Skill
└── 写作生成Skill
```

---

## 八、资料调用Skill

### 1. 系统定位

资料调用Skill负责告诉 LLM 在执行某项写作任务时，应该读取哪些资料、如何筛选资料、如何判断资料优先级、如何处理资料冲突。

---

### 2. 核心职责

资料调用Skill负责：

```text
调用设定系统
调用事件数据库
调用叙事系统已有内容
调用上一章正文摘要
调用相关人物卡
调用相关事件卡
调用相关世界观设定
调用相关术语表
调用伏笔与信息释放记录
判断资料优先级
处理资料冲突
限定资料使用范围
防止提前暴露信息
```

---

### 3. 调用原则

LLM生成内容前，应根据任务类型调用不同资料。

生成总纲时，优先调用：

```text
核心世界观
主角设定
核心冲突设定
主要大事件
全书主题
类型文定位
```

生成单元大纲时，优先调用：

```text
单元目标
相关大事件
相关人物设定
相关组织设定
单元情绪曲线
单元结局要求
```

生成章纲时，优先调用：

```text
单元大纲
相关大事件
出场人物卡
相关伏笔
信息释放限制
上一章摘要
下一章目标
```

生成正文时，优先调用：

```text
章节细纲
出场人物卡
当前场景信息
上一章正文摘要
文风规则
对话规则
节奏规则
禁用写法
```

---

## 九、写作生成Skill

### 1. 系统定位

写作生成Skill负责定义 LLM 应该如何写。

它关注正文质量、文风一致性、人物表现、节奏控制、去AI化、网络小说可读性和读者体验。

---

### 2. 核心职责

写作生成Skill负责：

```text
定义正文文风
定义人物对话方式
定义章节结构
定义网文节奏
定义爽点设计
定义悬念设计
定义反转设计
定义情绪推进
定义信息差
定义伏笔埋设与回收
定义去AI化规则
定义禁止写法
定义修改标准
```

---

### 3. 写作规则方向

写作生成Skill应包含以下规则：

```text
对话必须口语化、生活化
禁止人物用说明文口吻解释设定
人物关系通过动作、停顿、寒暄和留白表现
少写心理，少写大段环境
正文主要靠动作和对话推进
每章必须有明确读者奖励
每章应形成压制、触发、兑现、余波
主角不能降智
读者可以领先主角，但主角不能显得愚蠢
异常内容可被读者感知，但不能让主角过早理解
人物台词必须符合人物身份、年龄、经验和当下处境
禁止AI腔、总结腔、解释腔、概念堆砌
```

## 十、四大系统边界

### 1. 设定系统与事件数据库的边界

设定系统记录规则。

事件数据库记录事实。

例如：

```text
任战的能力机制 → 设定系统
任战在某一时间段利用该能力布局现实骗局 → 事件数据库
```

---

### 2. 事件数据库与叙事系统的边界

事件数据库记录世界真实发生的大事件。

叙事系统决定这些大事件如何被拆分、重组、隐藏、揭露并写给读者。

例如：

```text
《陈科被高端分享局诈骗事件》 → 事件数据库
该事件被拆成若干章节并安排悬念、反转、节奏 → 叙事系统
```

---

### 3. 叙事系统与Skill系统的边界

叙事系统存放输出结果。

Skill系统存放生成规则。

例如：

```text
P0十五章大纲 → 叙事系统
如何根据事件生成P0十五章大纲 → Skill系统
```

---

### 4. 设定系统与Skill系统的边界

设定系统记录小说内容本身。

Skill系统规定 LLM 如何使用这些内容。

例如：

```text
马木的人物设定 → 设定系统
写马木时如何表现嘴直但克制、热心但不圣母 → Skill系统
```

---

## 十一、Obsidian落地原则

### 1. 文件标准化

所有资料必须采用标准文件模板存储，便于 LLM 调用、检索、筛选和校验。

文件应包含：

```text
唯一ID
标题
类型
状态
所属系统
相关人物
相关事件
相关设定
时间信息
摘要
正文内容
双链关系
标签
```

---

### 2. 双链关联

系统应充分利用 Obsidian 双链能力。

常见关联包括：

```text
人物 ↔ 事件
人物 ↔ 组织
人物 ↔ 地点
事件 ↔ 设定
事件 ↔ 叙事单元
设定 ↔ 术语
章节 ↔ 伏笔
章节 ↔ 事件
章节 ↔ 人物
```

---

### 3. YAML属性

所有关键文档应使用 YAML 属性，方便后续通过 Dataview、脚本或 LLM 检索。

示例：

```yaml
id: CHR-0001
type: character
system: canon
status: active
related_events:
  - EVT-0001
related_chapters:
  - C01
tags:
  - character
  - main
```

```text
    ├── 06_一致性校验Skill/
    └── 07_写作风格Skill/
```

## 十二、系统运行流程

系统的基础运行流程如下：

```text
1. 作者在设定系统中维护人物、世界观、能力、组织、道具等硬规则
2. 作者在事件数据库中维护世界时间轴和故事级大事件
3. Skill系统根据写作任务调用相关资料
4. LLM在Skill规则约束下生成叙事系统内容
5. 叙事系统形成总纲、卷纲、单元纲、章纲、细纲和正文
6. 正文生成后，系统根据结果回写必要的事件、设定、伏笔和修订记录
7. Skill系统继续进行一致性校验和正文质量修订
```

# 结论

小说写作系统应采用四大模块架构：

```text
设定系统 = 世界规则库
事件数据库 = 世界时间轴事实库
叙事系统 = 正文生产线
Skill系统 = LLM创作控制系统
```

系统底层以世界时间轴为基础，保证事件、人物经历和组织行动的逻辑一致。

叙事系统使用 B / P / C 等小说结构编号，负责将世界事实加工为读者可阅读的小说结构。

Skill系统服务于 LLM，既负责资料调用，也负责写作生成，是连接资料系统与正文生产的核心控制层。
